Jira interactions use `{{BINARY_PATH}} exec jira`. Run `{{BINARY_PATH}} exec jira --help` for the full command list, and `--help` on any subcommand for details.

Workflows:
- Inspect an issue: `exec jira ticket view <KEY>`, `exec jira ticket list-children <KEY>`, `exec jira ticket list-transitions <KEY>`.
- Mutate an issue: `exec jira ticket transition`, `exec jira ticket assign`, `exec jira ticket comment`, `exec jira ticket set-priority`.
- Create issues: `exec jira ticket create`. Use `--parent` to create sub-issues.
- Search: `exec jira ticket search --jql "<query>"`. JQL is passed through directly.

Gotchas:
- Server/DC vs Cloud differences (parent linking, description format) are handled internally — you don't need to worry about them.
- For multi-line `--body` or `--description` arguments, use the same heredoc pattern as GitHub tools.

# Workspace (per-run worktree materialization)

Jira runs start with no codebase. Your initial cwd is the run-root — a throwaway directory with no git, no source, no `--repo` context. You decide which configured repo(s) the ticket belongs in and materialize a worktree per repo via `{{BINARY_PATH}} exec workspace`. Run `{{BINARY_PATH}} exec workspace --help` for the full surface.

Discovery:
- `{{BINARY_PATH}} exec workspace list` — JSON inventory with two sections:
  - `available`: configured repos you could `add` but haven't yet, each with a `description` field (one-line upstream metadata, e.g. "Core API service") that's your strongest disambiguation signal when the ticket alone isn't conclusive
  - `materialized`: worktrees you've already added for this run, with their absolute paths and feature branches
  Run this first when the ticket text doesn't make the target repo obvious. The `available` section is your universe of options; `add` rejects anything outside it.

Materialize:
- `cd "$({{BINARY_PATH}} exec workspace add <owner>/<repo>)"` — clones (or reuses the bare cache), creates a fresh feature branch (`feature/<ISSUE_KEY>`) off the repo's configured base, and prints the absolute worktree path. Idempotent: a second `add` for the same repo prints the existing path without re-creating, so re-running on retry is safe.
- After the cd you're inside a real git checkout — gh subcommands resolve via `.git/config` (no `--repo` needed), and you can use the full git/build/test toolchain.
- If the ticket spans multiple repos, repeat the `add` for each one — they materialize as siblings under the run-root.

Failure modes:
- "repo is not configured in Triage Factory" — the owner/repo isn't in your `workspace list available`. Don't keep guessing; either pick from the available list or report `task_unsolvable` if the ticket genuinely belongs in a repo TF doesn't manage.
- "repo has no clone URL on its profile; try re-profiling" — the user's repo profile is incomplete. Report `task_unsolvable` and surface the message; the user fixes it on the Settings page.
- "only supported for Jira runs" — you're on a GitHub PR run with an eagerly-materialized worktree; you shouldn't be calling workspace at all.

Don't `cd` outside the materialized worktrees for code work. Files outside `$TRIAGE_FACTORY_RUN_ROOT` are blocked by the session's file-access scope, and the only writable area outside the worktrees is `$TRIAGE_FACTORY_RUN_ROOT/task_memory/` (see <task_memory> in the envelope).
