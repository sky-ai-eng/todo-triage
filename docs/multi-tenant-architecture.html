Multi-tenant & Containerization Spec

The architectural decisions, schema additions, auth model, and work breakdown for turning Triage Factory from a single-user local binary into a self-hostable multi-tenant deployment — with an eventual SaaS path on app.triagefactory.com. Local mode = multi mode at N=1; same schema, same routing, same primitives. The deployments differ in onboarding and credentials, not in model.

Reframe (2026-05-10): the model has converged on team-as-unit-of-work with the agent (bot) as a separate domain from users. Earlier framings of "private-by-default with v2 team toggle" and "bot is a users row with kind='bot'" have been replaced. Team is the default visibility tier. Local mode is a single-org single-team single-user single-agent install — same shape as multi-mode, just N=1. See the "reframe" cards in §1 for the load-bearing decisions.

Tenant model

Org owns billing & integrations. Team owns work. Members of a team are humans (via memberships for team-scoped role + org_memberships for org-scoped role) and the org's bot (via team_agents). The bot is a separate domain — not a flavor of user. A user's org membership is transitive via team membership; the explicit org_memberships row exists for the org-admin role distinction independent of which team(s) the user belongs to.

Two tables, two roles: memberships(user_id, team_id, role) tracks team membership with a team-scoped role (owner/admin/member/viewer of that team). org_memberships(user_id, org_id, role) tracks org-wide admin distinction (org owner vs admin vs member vs viewer at the org level — independent of which teams the user is on). The two are distinct: a user might be team admin of Platform but only a member at the org level, or org admin without being explicitly added to every team. Org membership IS still transitive (any team membership implies org access), but the org_memberships row is what surfaces the org-admin write-on-shared-rows capability (see §9 RLS pattern).
Agents (the org's bot) live in a separate agents table, not the users table. One agents row per org. Team membership for the agent is via team_agents(team_id, agent_id, enabled, per_team_model, per_team_autonomy_suitability). Default-enabled at team creation; admin can disable per team. The agent acts under its own credentials (multi mode: agents.github_app_installation_id; local mode + small-org fallback: agents.github_pat_user_id pointing at a designated user's PAT). Tasks reference the agent via tasks.claimed_by_agent_id alongside tasks.claimed_by_user_id with CHECK XOR — same uniform claim semantics, distinct identity types.

Roles

Roles are per-membership, not global. Same user can be owner of Acme/Platform and viewer of Globex/Engineering. The role enum is shared between memberships (team-scoped) and org_memberships (org-scoped) — same semantic meanings ("owner" means "can manage and delete this scope"), different scopes.

-- core role enum, shared by memberships + org_memberships
CREATE TYPE membership_role AS ENUM (
  'owner',    -- can delete the scope, manage all memberships at this scope
  'admin',    -- can invite/remove non-owners, edit settings at this scope
  'member',   -- standard access — read/write own resources at this scope
  'viewer'     -- read-only at this scope
);

RLS helpers tf.user_is_org_admin(org_id) (checks org_memberships) and tf.user_in_team(team_id) (checks memberships) gate writes on org-visible and team-visible rows respectively — see §9. Agents don't have a role column; the team_agents row IS the membership and writes by the agent always go through server-side workload code, never through the RLS-active app pool.

Cross-org user identity

auth.users.email in GoTrue is UNIQUE. A consultant works at multiple orgs as one account, multiple memberships. The active org is encoded in the URL path — /api/orgs/<org_id>/... — so each request is unambiguous about which org context applies. The frontend persists last-active org ID in localStorage to construct API URLs and to default the user back to their last context after a reload.

Deployment modes

Same binary, same image, same model. The mode flag flips which subsystems initialize (database driver, auth pipeline, credential source) but the data model and routing logic are identical. Local mode is a single-org, single-team, single-user, single-agent install of the same primitives — same code paths, just N=1.

Activation

Default when TF_MODE is unset or local. Works exactly as today's binary.

Storage

  • SQLite at ~/.triagefactory/triagefactory.db
  • OS keychain for secrets (internal/auth)
  • Worktrees at ~/.triagefactory/worktrees/
  • SDK install at ~/.triagefactory/sdk/ (runtime install, ~230MB)

Auth

Implicit single user. SQLite skips the auth surface — there's only one human and one bot, both bootstrapped at install. The model still has one synthetic org, one team, one user, one agent (same shape as multi mode at N=1) so all routing/visibility/claim logic shares the multi-mode code path. assertLocalOrg in the SQLite store impls pins the synthetic org id and panics on any caller that passes something else.

Integrations

PAT-based as today. GitHub PAT, Jira API token, Linear API key — all in keychain. The local agent borrows the user's PAT via agents.github_pat_user_id pointing at the lone user. No GitHub App in local mode; that's a multi-mode concept tied to an org-level install.

Distribution

./triagefactory binary. Embedded SPA via go:embed. Browser opens to localhost:3000.

Activation

TF_MODE=multi. Required env: TF_DATABASE_URL (Postgres), TF_GOTRUE_URL, TF_GOTRUE_JWKS_URL, TF_SESSION_KEY (32 bytes, AES-GCM key for session-token encryption only), TF_PUBLIC_URL.

Storage

  • Postgres (supabase/postgres image — includes vault, pgsodium) for everything: TF rows + GoTrue's auth.*
  • Supabase Vault for org-level secrets (PATs, API tokens, future GH App keys) — accessed via SECURITY INVOKER wrappers
  • Sessions encrypted app-side with TF_SESSION_KEY (BYTEA + nonce in the sessions table) — separate from Vault because per-request RPC overhead is unacceptable
  • Worktrees at /data/<org_id>/worktrees/run-<id>/ (named Docker volume)
  • SDK pre-installed at /opt/triagefactory/sdk/ at image build time

Auth

GoTrue handles signin (GitHub OAuth, Entra SAML). TF Go backend acts as auth proxy: handles OAuth dance, mints opaque session ID cookie, validates JWT every request via JWKS.

Integrations

Agent has its own identity via a per-org GitHub App install (recommended) or PAT-borrow from a designated user (fallback for small orgs / B2C trials). agents.github_app_installation_id holds the App install id; agents.github_pat_user_id is the fallback FK. Jira/Linear OAuth per org. App private key + PAT material + OAuth refresh tokens all in Supabase Vault, scoped via org/<uuid>/<key> naming. The agent is the actor on every PR comment / push / Jira transition — audit logs attribute work to "Triage Factory Bot", not to any human.

B2C / single-member orgs

Supported via the same schema. A solo user on app.triagefactory.com gets an org with one team and one human member plus the org's agent. Onboarding hides team-creation UI by default and shows it behind a "grow my team" CTA. PAT-borrow is reasonable here; the audit-log caveat doesn't matter when there's only one human.

Distribution

Docker image. docker compose up. Compose stack = TF + Postgres + GoTrue. Optional --profile admin adds pgAdmin. Optional --profile https adds Caddy.

Auth flow

Backend-as-auth-proxy. The browser only sees an opaque session cookie; the JWT lives server-side on the sessions row. JWKS verify on every request.

Sessions table

Tokens encrypted at rest with the same envelope as secrets. Soft revocation via revoked_at (audit trail + active-sessions UI). Session-level expiry (expires_at) is independent of JWT expiry — refresh tokens cycle within the session window.

View sessions schema SQL · encrypted · soft delete 
CREATE TABLE sessions (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id           UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,

  -- tokens encrypted at rest (AES-GCM envelope, master key from TF_SESSION_KEY)
  jwt_enc           BYTEA NOT NULL,                  -- gotrue access JWT, ciphertext
  jwt_nonce         BYTEA NOT NULL,
  refresh_token_enc BYTEA NOT NULL,                  -- gotrue refresh token, ciphertext
  refresh_nonce     BYTEA NOT NULL,

  -- two distinct expirations:
  jwt_expires_at    TIMESTAMPTZ NOT NULL,             --   when the current JWT needs refresh (~1h)
  expires_at        TIMESTAMPTZ NOT NULL,             --   when the session itself expires (~30d, force re-login)

  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_seen_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  revoked_at        TIMESTAMPTZ,                     -- soft delete; row stays for audit/forensics

  user_agent        TEXT,                            -- "active sessions" UI fingerprint
  ip_addr           INET,

  CHECK (expires_at > created_at),
  CHECK (last_seen_at >= created_at),
  CHECK (jwt_expires_at <= expires_at)                  -- JWT can't outlive the session
);

-- partial indexes skip revoked rows entirely
CREATE INDEX sessions_user_idx
  ON sessions(user_id)    WHERE revoked_at IS NULL;
CREATE INDEX sessions_expires_idx
  ON sessions(expires_at) WHERE revoked_at IS NULL;
Logout-everywhere: UPDATE sessions SET revoked_at = now() WHERE user_id = $1 AND revoked_at IS NULL. Subsequent requests 401 (middleware filters WHERE revoked_at IS NULL), frontend's useAuth hook clears state. Rows persist for audit — when, from where, after how many active hours. Cron job reaps revoked rows older than the retention window.

Org context per request

Org-scoped routes are /api/orgs/<org_id>/... — middleware extracts org_id from the path via Go 1.22+ pattern routing (r.PathValue("org_id")), then validates the user has at least one membership in any team within that org. Single indexed query (~<1ms). The path is just a request hint — the memberships row is the authority. Active org ID is persisted in localStorage by the SPA to default the user's URL after reload and to construct API URLs.

Status codes — Not authenticated: 401 Unauthorized. Authenticated but not a member of the org in the URL: 404 Not Found (don't leak existence; same response as "no such org"). Authenticated and a member but lacking the role to perform the action (e.g., viewer trying to delete): 403 Forbidden. Within-org permission failures use 403 because existence is already disclosed by membership.

Routes that are not org-scoped: /api/auth/* (login/callback/logout — no org context yet), /api/me (returns the user profile + their org memberships — cross-org by definition), /api/webhooks/* (v2; the receiver figures out the org from the payload), /health.

§5

Service topology (multi mode)

Three required services in the default compose. Two optional services behind --profile flags. No Studio, no Kong, no Realtime in v1.

Hover a service to highlight its connections. agent-runner pulls queued delegations from runs, mounts tf-data, and spawns each delegation as a runsc rootless subprocess (gVisor) — kernel-level isolation without Docker-in-Docker or KVM. TF web tier never touches worktrees and has no privilege to spawn agents.

Future services (not in v1) deferred

Realtime — TF's existing WebSocket hub already streams run updates; adding Supabase Realtime is duplicate plumbing until there's a real use case. Defer.

Storage — would be useful for run artifacts, generated specs, or KB attachments. No use case in v1. Add when first need arises.

Studio — drags in 4 dependencies (Kong, Meta, Analytics, Vector). pgAdmin covers DB ops at a fraction of the operational cost.

Filesystem layout

Local writes to ~/.triagefactory/. Multi writes to /data/ inside the container, backed by a named volume. Both paths are derived through a single paths package; no callsite hardcodes either.

Local mode (~/.triagefactory/)

├── triagefactory.db      # SQLite
├── worktrees/            # bare clones + per-run dirs
│   ├── github.com/sky-ai-eng/foo.git/

├── takeovers/            # held takeovers
│   └── run-<uuid>/
├── sdk/                  # runtime-installed SDK
│   └── node_modules/

    └── <project_id>/knowledge-base/

/data/                            # mounted into agent-runner; TF web tier has no /data
└── orgs/
    └── <org_id>/                # per-org subtree
        └── worktrees/            # operational state only (git working directories)
            ├── github.com/.../foo.git/  # BARE CACHE — shared across runs in org;
            │                     # background-refreshed via `git remote update`;
            │                     # NEVER bind-mounted into the sandbox
            └── run-<uuid>/      # PER-RUN CLONE — full self-contained .git dir;
                                  # created via `git clone <bare> run-<uuid>` before
                                  # sandbox start; bind-mounted as /work into gVisor;
                                  # stays in place through the full lifecycle
                                  # (including any v2 takeover — same dir, same sandbox)

Source-of-truth content is NOT on disk in multi mode. Project knowledge bases, agent memories, and other persistent context live as rows in Postgres (org-scoped, RLS-isolated). agent-runner materializes them into the worktree's _scratch/ dirs at delegation start, and ingests changes back at completion. See the materialize/ingest pattern callout below.

Implementation: introduce internal/paths package. Functions like WorktreeDir(orgID, runID) for operational state. KB / memory access goes through store.KBStore / store.MemoryStore instead — Postgres in multi mode, filesystem in local mode (today's ~/.triagefactory/projects/<id>/knowledge-base/ and friends).
Clone-from-bare, not worktree-add. Today's single-user mode uses git worktree add against the bare clone, which creates a per-run dir whose .git is a file pointing back to the bare's metadata. That doesn't survive the gVisor bind-mount: only the run dir is mounted into the sandbox, and the .git file's path-to-bare doesn't exist inside the sandbox.

Multi mode replaces it with git clone <local-bare> run-<uuid> before sandbox start. The per-run dir gets a complete, self-contained .git directory. Bind-mount the run dir alone; agent's git operations work natively. The bare stays at /data/orgs/<org>/worktrees/foo.git/ as a cache (background-refreshed via git remote update by agent-runner, never touched by per-run sandboxes).

Cost: 2× the repo size on disk per org (bare + active per-run clones). Local-to-local clone is ~1–2s for typical PR-sized repos. For 5GB monorepos: revisit with --filter=blob:none partial clone when a customer needs it.
Materialize / ingest pattern. Memories and project knowledge in multi mode follow the same shape:
  1. At delegation start: agent-runner queries Postgres for the org's relevant rows, writes them as files into the worktree's _scratch/entity-memory/ and _scratch/project-knowledge/ dirs. The gVisor-sandboxed agent reads them as ordinary files via the SDK's tools.
  2. At completion: agent-runner reads back the same dirs, diffs against what it wrote, upserts changed files as rows. Curator runs are how knowledge bases get updated; ordinary delegations write only memories.
  3. Why this shape: the agent's tooling (Read/Write/Edit) operates on files. We don't want the agent to know about Postgres. The runner bridges DB → filesystem at the gVisor boundary.
Identical to today's local-mode flow, just with Postgres as the source instead of ~/.triagefactory/projects/ and memories rows in the local SQLite.
Takeovers — addressed by deferral + same-sandbox handover. A delegation run is fully gVisor-sandboxed; the agent has no path out. A takeover would hand control to a human, and any shell-shaped access fundamentally lives outside gVisor by default. Without care, a takeover-authorized user could ls /data/orgs/ and read every other org's worktrees.
  • v1 multi-mode ships with takeover disabled. Runs that yield surface as awaiting_input with text-reply resume / cancel — no filesystem access, no shell.
  • v2 multi-mode adds the gVisor-wrapped web shell (D18): the agent's Node process is terminated, the sandbox is kept alive, runsc exec drops a bash into the same sandbox bound to the same /work. The worktree never moves — it stays at /data/orgs/<org>/worktrees/run-<uuid>/ throughout. xterm.js + websocket pumps the pty. User inherits the agent's exact isolation: same mount namespace, same network ACL, same kernel boundary. No new isolation layer to audit, no parallel filesystem subtree.
Takeover-specific bookkeeping (replica pinning, sandbox ID, last git HEAD) lives as columns on the runs row — small enough not to warrant a filesystem subtree.

Isolation boundary (multi mode only)

In multi mode /data/ is mounted into agent-runner only, not TF's web tier. When agent-runner claims a queued delegation, it spawns a runsc do --rootless child process. The agent's syscalls go through gVisor's userspace kernel — the host kernel never executes them directly. The sandboxed view:

# What the gVisor-sandboxed agent process sees:
/work/                          # bind-mount of /data/orgs/<org>/worktrees/run-<id>
├── .git/
├── README.md
├── _scratch/
│   ├── entity-memory/           # materialized priors
│   └── project-knowledge/       # materialized KB (org-scoped, copied in by runner)
└── …

/tmp/                          # tmpfs, ephemeral, sandbox-private
/opt/sdk/                      # read-only bind-mount of agent-runner's bundled SDK
                                # NO other /data/orgs paths (mount-isolated to this run only)
                                # NO /proc/<agent-runner-pid>/environ (sandbox sees only its own /proc)
                                # NO env vars from agent-runner (clean env on spawn)
                                # NO network except: api.anthropic.com + api.github.com (iptables ACL)
What this blocks: find /, cat /proc/*/environ, nsenter, kernel-CVE-based escapes — none of these reach the host kernel because gVisor intercepts every syscall in userspace. The TF_SESSION_KEY env var, other orgs' worktrees, the Vault-encrypted org secrets — all live outside the sandbox's namespace. Prompt-injection attacks aimed at exfiltration cap out at "what's already in this run's working directory."
Pre-flight check: runsc rootless requires unprivileged user namespaces (kernel.unprivileged_userns_clone=1, no seccomp blocking CLONE_NEWUSER). Most modern Linux container runtimes allow this — verify on Railway with a 10-line smoke test before committing. If blocked, fall back path is documented in D10.

Schema changes

Multi-tenant Postgres carries the full schema: tenancy tables, agent identity, sessions, integrations, secrets, limits, plus every TF resource table with org_id and (where applicable) team_id + creator_user_id baked in from CREATE TABLE. Local SQLite carries a subset — same table shapes for resources but no org_id column (the synthetic local org id is a runtime constant). Both backends share the same domain types and store interfaces.

New tables (Postgres only)

14 new tables across identity, agent, sessions, integrations, secrets, limits, and per-user view state.

Two domains, kept separate: users + memberships + org_memberships represent human identities (FK'd to auth.users; one row per authenticated person). agents + team_agents represent workload identities (one row per org; the bot has no auth.users entry, no session, no JWT). Tasks reference both with two FK columns (claimed_by_user_id + claimed_by_agent_id with CHECK XOR) — audit/claim flows stay uniform without polymorphic discriminators. The split keeps Supabase's auth.users invariant clean (every public.users row IS an authenticated identity) and lets agent config grow without bloating the users table.

-- Wrapper functions (callable from Go via pgx, no PostgREST needed).
-- Pattern lifted from skynet/sky/supabase/migrations/00000000000000_skynet_base.sql

CREATE OR REPLACE FUNCTION public.vault_put_org_secret(
  p_org_id     UUID,
  p_key        TEXT,                                    -- e.g. 'github_pat', 'jira_api_token'
  p_secret     TEXT,
  p_description TEXT DEFAULT NULL
) RETURNS UUID
LANGUAGE plpgsql SECURITY INVOKER
SET search_path TO public, vault
AS $$
DECLARE
  v_full_name TEXT := 'org/' || p_org_id::text || '/' || p_key;
  v_existing  UUID;
BEGIN
  SELECT id INTO v_existing FROM vault.decrypted_secrets WHERE name = v_full_name;
  IF v_existing IS NOT NULL THEN
    PERFORM vault.update_secret(v_existing, p_secret, v_full_name, p_description);
    RETURN v_existing;
  END IF;
  RETURN vault.create_secret(p_secret, v_full_name, p_description);
END;
$$;

CREATE OR REPLACE FUNCTION public.vault_get_org_secret(p_org_id UUID, p_key TEXT)
RETURNS TEXT
LANGUAGE sql SECURITY INVOKER STABLE
SET search_path TO public, vault
AS $$
  SELECT decrypted_secret FROM vault.decrypted_secrets
  WHERE name = 'org/' || p_org_id::text || '/' || p_key
  LIMIT 1;
$$;

CREATE OR REPLACE FUNCTION public.vault_delete_org_secret(p_org_id UUID, p_key TEXT)
RETURNS VOID
LANGUAGE plpgsql SECURITY INVOKER
SET search_path TO public, vault
AS $$
DECLARE
  v_id UUID;
BEGIN
  SELECT id INTO v_id FROM vault.decrypted_secrets
    WHERE name = 'org/' || p_org_id::text || '/' || p_key;
  IF v_id IS NOT NULL THEN
    DELETE FROM vault.secrets WHERE id = v_id;
  END IF;
END;
$$;

Existing TF tables in multi-mode Postgres

Multi-mode Postgres carries the full TF surface (entities, events, tasks, runs, prompts, projects, task_rules, prompt_triggers, memories, etc.) with org_id UUID NOT NULL REFERENCES orgs(id), team_id (NOT NULL on team-scoped resources), and (where applicable) creator_user_id UUID REFERENCES users(id) baked in from CREATE TABLE. Default visibility = 'team' on every row-scoping table; 'private' is an escape hatch and 'org' requires admin. Local SQLite carries the same domain types and store interfaces with the synthetic single-org / single-team runtime-constant approach (no org_id column needed when there's only ever one value).

What this means in practice team-default · two-FK claim · agent in the model
  • No ALTER TABLE dance. The per-resource store interfaces (D2) route to either sqlite/* impls (synthetic-N=1 schema, no org_id column) or postgres/* impls (full multi-mode schema, with org_id + team_id everywhere from inception).
  • No "default" org placeholder in multi. First user that signs up to a fresh multi-mode deployment creates the first real org. Subsequent rows get a real org_id. Local mode pins a runtime-constant synthetic org id (runmode.LocalDefaultOrg) which the SQLite store impls assert via assertLocalOrg at every method entry.
  • Bootstrap on org creation: one agents row + one team_agents row per team. The org admin is also automatically inserted as the first org_memberships row with role 'owner'. Default team (slug='default') is created with the admin as member + the agent as team_agent (enabled).
  • Local users don't migrate to multi. A user running TF locally on their laptop stays on SQLite. If they later set up a hosted multi-tenant deployment, they spin up a fresh Postgres — their old SQLite data is theirs to keep, not something we transplant. Both deployments use the same primitives; only the data is separate.
  • Visibility column on every team-scopable table with NOT NULL DEFAULT 'team' + CHECK in (private, team, org). Team is the canonical scope. RLS branches on visibility — see §9. task_rules and prompts can additionally have system-source rows (shipped by us, creator_user_id NULL, visibility='org') that the migration 202605110001_system_rows_nullable_creator.sql enables. Org admins (per org_memberships) can edit those; team members can't.
  • Tasks have a two-column claim FK: claimed_by_user_id UUID NULL REFERENCES users(id) + claimed_by_agent_id UUID NULL REFERENCES agents(id) with CHECK (claimed_by_user_id IS NULL OR claimed_by_agent_id IS NULL). Rules create unclaimed tasks; triggers create tasks with claimed_by_agent_id stamped (= the team's assigned agent). A swipe-to-claim by a human writes claimed_by_user_id; release clears both.
  • Runs carry both an actor and a delegator: actor_agent_id UUID NOT NULL REFERENCES agents(id) (the bot did the work) + creator_user_id UUID NULL REFERENCES users(id) (the human who delegated — NULL when a trigger fired and the run was bot-initiated).
  • No skills table. Skills are not a multi-mode concept — they're imported as prompts. Local mode keeps file-based skills at ~/.claude/agents/*.md; multi mode users paste the markdown into a prompt with whatever visibility scope they want.
  • System-emitted tables stay user-anonymous. events and events_catalog only need org_id — events come from pollers/system, not specific users.

Why not generics over the SQL? Tempting, but SQLite/Postgres dialect divergence (JSON ops, partial-index expressions, ON CONFLICT shapes, type names) leaks into raw queries. Cleaner to write each impl's queries native to its dialect than to fight a lowest-common-denominator query builder.
Cross-store transactions. Anything that mutates rows in two stores (e.g., creating a run + emitting an event atomically) uses a transaction wrapper that begins a *sql.Tx and hands tx-bound versions of the relevant stores to a closure. Prevents partial commits and keeps the per-store interfaces clean (no transaction primitives leaking into them). Spec'd in D2.

RLS pattern

Service-role connection + SET LOCAL request.jwt.claims per transaction. Pgbouncer-friendly. Real enforcement: even if a query forgets to filter by org_id, RLS rejects rows the user can't see.

Per-transaction setup

View Go transaction wrapper Go 
// internal/store/postgres/tx.go — wraps a tx + tx-bound versions of every store
type TxStores struct {
  Tasks       store.TaskStore
  Runs        store.RunStore
  Events      store.EventStore
  Memberships store.MembershipStore
  // … etc, mirroring store.Stores but each store wired to *sql.Tx instead of *sql.DB
}

func WithTx(ctx context.Context, db *sql.DB, fn func(s TxStores) error) error {
  tx, err := db.BeginTx(ctx, nil)
  if err != nil { return err }
  defer tx.Rollback()

  claims := ctxClaims(ctx)         // pulls user_id, current_org_id from request ctx
  payload, _ := json.Marshal(claims)

  // SET LOCAL: scoped to this transaction, doesn't leak across the connection pool
  if _, err := tx.ExecContext(ctx,
    `SELECT set_config('request.jwt.claims', $1, true)`, string(payload)); err != nil {
    return err
  }

  if err := fn(newTxStores(tx)); err != nil { return err }
  return tx.Commit()
}

// Caller usage:
err := pgstore.WithTx(ctx, db, func(s pgstore.TxStores) error {
  run, err := s.Runs.Insert(ctx, r); if err != nil { return err }
  return s.Events.Append(ctx, runStartedEvent(run))
})

-- helper: extract current user_id from session JWT claims
CREATE FUNCTION tf.current_user_id() RETURNS UUID AS $$
  SELECT NULLIF(current_setting('request.jwt.claims', true)::jsonb ->> 'sub', '')::uuid;
$$ LANGUAGE SQL STABLE;

-- helper: extract active org_id from claims (set by middleware from URL path /api/orgs/<org_id>/...)
CREATE FUNCTION tf.current_org_id() RETURNS UUID AS $$
  SELECT NULLIF(current_setting('request.jwt.claims', true)::jsonb ->> 'org_id', '')::uuid;
$$ LANGUAGE SQL STABLE;

-- helper: verify org membership transitively via teams (any team in org)
CREATE FUNCTION tf.user_has_org_access(target_org UUID) RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM memberships m
    JOIN teams t ON t.id = m.team_id
    WHERE m.user_id = tf.current_user_id() AND t.org_id = target_org
  ) OR EXISTS (
    SELECT 1 FROM org_memberships om
    WHERE om.user_id = tf.current_user_id() AND om.org_id = target_org
  );
$$ LANGUAGE SQL STABLE;

-- helper: verify direct team membership (for team-visible writes)
CREATE FUNCTION tf.user_in_team(target_team UUID) RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM memberships
    WHERE user_id = tf.current_user_id() AND team_id = target_team
  );
$$ LANGUAGE SQL STABLE;

-- helper: org-admin role check (org-wide, independent of team membership)
CREATE FUNCTION tf.user_is_org_admin(target_org UUID) RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM org_memberships
    WHERE user_id = tf.current_user_id()
      AND org_id = target_org
      AND role IN ('owner', 'admin')
  );
$$ LANGUAGE SQL STABLE;

-- example: tasks. Team-default visibility, three policies (select/insert/update).
ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;

-- READ: team-visible rows visible to team members; org-visible to any org member;
-- private only to the creator. Same shape applies to task_rules / prompts / projects.
CREATE POLICY tasks_select ON tasks
  FOR SELECT USING (
    org_id = tf.current_org_id() AND tf.user_has_org_access(org_id)
    AND (
      (visibility = 'private' AND creator_user_id = tf.current_user_id())
      OR (visibility = 'team'    AND team_id IS NOT NULL AND tf.user_in_team(team_id))
      OR (visibility = 'org')
    )
  );

-- INSERT: creator must equal current user; team_id required for team visibility.
-- A NULL creator_user_id is only allowed for system-source rows, which are
-- inserted by the deploy-time/seed actor on the BYPASSRLS admin pool — never
-- via the request-path app pool — so this WITH CHECK refuses every tf_app
-- attempt to forge a system row regardless of what columns it sets.
CREATE POLICY tasks_insert ON tasks
  FOR INSERT WITH CHECK (
    org_id = tf.current_org_id() AND tf.user_has_org_access(org_id)
    AND creator_user_id = tf.current_user_id()
    AND (visibility <> 'team' OR tf.user_in_team(team_id))
  );

-- UPDATE: own private row · any team-visible row in a team you're in ·
-- org-visible row IFF you're an org admin. The org admin gate is what
-- prevents "any org member can disable a shipped default for everyone".
CREATE POLICY tasks_update ON tasks
  FOR UPDATE USING (
    org_id = tf.current_org_id()
    AND (
      (visibility = 'private' AND creator_user_id = tf.current_user_id() AND tf.user_has_org_access(org_id))
      OR (visibility = 'team'    AND team_id IS NOT NULL AND tf.user_in_team(team_id))
      OR (visibility = 'org'     AND tf.user_is_org_admin(org_id))
    )
  ) WITH CHECK (
    -- mirror of USING; prevents row-visibility-by-rewrite
    org_id = tf.current_org_id()
    AND (
      (visibility = 'private' AND creator_user_id = tf.current_user_id() AND tf.user_has_org_access(org_id))
      OR (visibility = 'team'    AND team_id IS NOT NULL AND tf.user_in_team(team_id))
      OR (visibility = 'org'     AND tf.user_is_org_admin(org_id))
    )
  );
Service-role caveat: Postgres role used by TF must NOT be a superuser (those bypass RLS) and must NOT have BYPASSRLS. Create a dedicated tf_app role with grants but no bypass. The migration role (used by goose) can be more privileged since it never serves user requests.

D18 SKY-268 Web takeover terminal — same-sandbox handover (v2 fast-follow) M priority? todo deps: D7, D8, D10 · v2 (post-v1 ship)
Why

v1 multi-mode ships without takeover — runs that yield surface as awaiting_input with text-reply resume only. That covers the ~80% case (agent stuck on a clarifying question, user types an answer, run continues). The 20% case is a real takeover: user wants to manually edit code, run git commands, push fixes. Today's single-user takeover hands them shell access to the worktree dir, which in multi mode is a cross-tenant escape. This domain adds takeover back without breaking isolation.

Design — same-sandbox handover

Instead of starting a new shell environment for the user, we hand over the existing gVisor sandbox the agent was running in. The user inherits the agent's exact mount namespace, network ACL, and kernel boundary. No second isolation layer to audit.

  1. Agent yields. agent-runner terminates the Node process but keeps the gVisor sandbox alive (sandbox is the unit, the Node process inside it is replaceable).
  2. agent-runner writes { replica_id, sandbox_id, last_git_head, anthropic_session_id } as columns on the runs row. No filesystem state — the worktree stays at /data/orgs/<org>/worktrees/run-<uuid>/.
  3. User clicks "Take over" in the UI. TF web tier looks up the holding replica, opens a websocket to it (sticky-routed via the run's replica_id).
  4. That replica runs runsc exec <sandbox_id> bash — bash starts as a sibling of the suspended Node process, sharing all namespaces.
  5. Replica pumps the pty over the websocket; browser renders via xterm.js. SIGWINCH propagates on resize. Heartbeats prevent idle disconnect.
  6. User does manual work. Releases via "Done" (drop the bash, mark run completed) or "Resume agent" (kill bash, kill+respawn Node with SDK session-resume + a context message describing user-side changes).
Known flaws & mitigations
  • Long-lived sandboxes. gVisor sandboxes are designed to be ephemeral; keeping one alive for hours of human work is unusual. Mitigation: 24h TTL on the takeover, after which the sandbox is torn down and the worktree is preserved on disk for a future "fresh takeover" (which would be a new sandbox bound to the same /work).
  • Replica failover. The sandbox lives on a specific agent-runner replica. If that replica restarts mid-takeover, the sandbox is lost and the user disconnects. Mitigation: write replica_id to runs; on disconnect, surface "your takeover replica was replaced — reopen to start a fresh sandbox bound to the preserved worktree." Loses agent-process state but not file state.
  • SIGSTOP vs terminate decision (chosen: terminate). Suspending Node with SIGSTOP would let us literally resume the same agent process, but the Anthropic API session ages and times out before most takeovers release. We terminate the Node process on takeover-start and rely on the SDK's session-resume primitive with the original session_id on takeover-end. Loses in-flight tool-call state, gains a much cleaner state boundary. The sandbox itself stays alive — we're not destroying anything we'd need to rebuild.
  • Filesystem divergence on resume. User edits files; agent on resume has stale mental model. Mitigation: when resuming, send the agent a system message with git status --porcelain + git diff HEAD output capped at 8KB ("the user took over and made these changes — pick up from here"). Same shape as how we materialize prior memories.
  • Git state divergence. User commits or pushes during takeover. Agent on resume could re-do work, or push conflicting commits. Mitigation: same as above — the resume context includes git log $LAST_KNOWN_HEAD..HEAD --oneline. Agent reads, adapts.
  • Concurrent connections. Two browser tabs open to the same takeover share the same pty (visible to both, both type into it). Probably fine for v1 — actually nice for screen-share-style collaboration. Document, don't gate.
  • Auth on the websocket. Cookie session validates the user; org-membership middleware validates they have takeover rights for this run. Standard. The websocket inherits the same auth pipeline as /api/*.
  • Idle abandonment. User opens takeover, walks away. Mitigation: 30min of pty inactivity → server-side warning ping; 1h → tear down bash but keep worktree; 24h → full TTL kicks in.
What changes
  • Frontend: xterm.js (~30KB gzipped) + a TakeoverTerminal page component with websocket connection + resize handling
  • Backend: new endpoint GET /api/runs/<id>/takeover/ws (websocket upgrade) on TF web tier; proxies to the replica holding the sandbox
  • agent-runner: POST /takeover internal endpoint that runs runsc exec, attaches pty, returns a streaming handle
  • SDK session-resume integration in internal/agentproc (we already have session ID tracking; just wire the resume path)
  • Resume context builder: produces the "user took over and changed these files" system message from git diff
  • TTL/abandonment cron in agent-runner
  • Takeover button + held-takeover banner in the SPA (replaces the v1 text-reply-only UI)
Acceptance
  • End-to-end: user starts a delegation, agent yields, user opens takeover, edits a file, commits, clicks "Resume agent" — agent picks up with awareness of the file change and continues to completion
