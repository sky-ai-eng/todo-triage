D4 — Mode flag + path scoping

Introduce internal/paths as the single source of truth for state-root resolution. Today every callsite reaches for os.UserHomeDir() and joins .triagefactory directly; multi mode needs /data/orgs/<org_id>/ instead. Mode is set once at startup via TF_MODE=local|multi; local mode strips the org segment so existing installs see no on-disk change.

§1

Why now

Doing D4 before D3 (Postgres schema) and D2 (store impls) is a sequencing call: D4 changes function signatures across every path-touching file in worktree/, delegate/, repoprofile/, curator/ by adding an orgID argument. Cheaper to do that mechanical sweep against today's codebase than against today's-plus-D2's-store-impls-plus-D3's-test-harness.

D2 (per-resource stores) and D3 (Postgres schema) are both genuine downstream consumers of internal/paths. D2's store implementations need orgID-aware path resolution; D3's tests don't directly touch paths but the integration tests they enable will. Landing D4 first means both subsequent tickets adopt the new signatures from day one rather than retrofitting.

On the Linear "D4 deps D2" listing: soft dep, not strict. D4 can land standalone — define the package, wire the mode flag, sweep the existing callers — and D2 just consumes the new functions when it lands. The architecture doc's full picture is D2 → D4, but the actual code dependency is one-way (D2 imports paths, paths doesn't import stores).

§3

Target state

One package. Eight functions. Mode flag set once at startup.

// internal/paths/paths.go (sketch)
package paths

type Mode string

const (
  ModeLocal Mode = "local"
  ModeMulti Mode = "multi"
)

// LocalDefaultOrg is the synthetic org id used in local mode. Storage
// layer treats it as "no real org context"; path layer strips it from
// resolved paths so existing local installs see no disk-layout change.
const LocalDefaultOrg = "default"

func Init(m Mode) error    // called once from main.go
func Current() Mode

// Per-resource resolvers. orgID is required everywhere; in local mode
// the package strips it (when == LocalDefaultOrg) so paths look the
// same as today.
func DBPath() (string, error)                              // state-root-relative; not org-scoped
func SDKDir() (string, error)                              // state-root-relative; not org-scoped
func LegacyConfigPath() (string, error)                    // state-root-relative; not org-scoped

func BareCacheDir(orgID, owner, repo string) (string, error)

// Test helper.

Why orgID is mandatory in local mode too: if the function lets local-mode callers pass empty / omit, two code paths drift. Every store-layer call (D2) takes an orgID anyway — the SQLite impl just asserts it's "default". Pushing the same convention into paths means D2's call sites are uniform, and there's nowhere local-mode can forget to pass org context as it gets ported toward multi.
§4

paths package API

State-root resolution

// stateRoot returns the root under which DBPath/SDKDir/etc. resolve.
// Local: ~/.triagefactory/. Multi: /data/.
func stateRoot() (string, error) {
  switch currentMode {
  case ModeLocal:
    home, err := os.UserHomeDir()
    if err != nil { return "", err }
    return filepath.Join(home, ".triagefactory"), nil
  case ModeMulti:
    return "/data", nil
  }
  return "", fmt.Errorf("paths: mode not initialized")
}

// orgScopedSubdir returns "" in local mode when orgID == LocalDefaultOrg,
// otherwise filepath.Join("orgs", orgID). Multi mode always emits the
// orgs/<id> segment (even though the only valid orgID is a real UUID
// from the orgs table — no LocalDefaultOrg in multi).
func orgScopedSubdir(orgID string) (string, error) {
  if orgID == "" {
    return "", fmt.Errorf("paths: orgID required (use LocalDefaultOrg in local mode)")
  }
  switch currentMode {
  case ModeLocal:
    if orgID != LocalDefaultOrg {
      return "", fmt.Errorf("paths: local mode received orgID=%q (must be %q)", orgID, LocalDefaultOrg)
    }
    return "", nil                          // strip in local mode
  case ModeMulti:
    if orgID == LocalDefaultOrg {
      return "", fmt.Errorf("paths: multi mode received LocalDefaultOrg sentinel — caller forgot to pass real org")
    }
    return filepath.Join("orgs", orgID), nil
  }
  return "", fmt.Errorf("paths: mode not initialized")
}

Each resolver composes stateRoot + orgScopedSubdir + the per-resource segment. WorktreeRoot:

func WorktreeRoot(orgID string) (string, error) {
  root, err := stateRoot()
  if err != nil { return "", err }
  org, err := orgScopedSubdir(orgID)
  if err != nil { return "", err }
  return filepath.Join(root, org, "worktrees"), nil
}

Mode-mismatch errors

Resolvers that don't apply in a given mode (DBPath in multi, ProjectKBRoot in multi, TakeoverDir in multi) return a typed error rather than fabricating a path. Caller decides — usually skip the operation or branch on the mode.

var ErrModeMismatch = errors.New("paths: function not applicable in current mode")
Why fail loudly instead of silently returning a "best effort" path: the architecture-feedback loop on past tickets has been "render no action instead of a plausible-but-wrong default." A multi-mode caller that asks for ProjectKBRoot is a bug — KB lives in Postgres in multi. Returning a path it'll then try to os.Stat would either silently no-op (mask the bug) or pollute /data with empty dirs.
§5

Mode flag wiring

How TF_MODE is read

One env-var read in main.go before any subsystem opens a path:

// main.go (server entry, before db.Open())
mode := paths.ModeFromEnv(os.Getenv("TF_MODE"))   // "" → ModeLocal, "local" → ModeLocal, "multi" → ModeMulti, anything else → fatal
if err := paths.Init(mode); err != nil {
  log.Fatalf("paths: %v", err)
}

CLI subcommands (exec, migrate, resume, install, uninstall) call the same paths.Init in their entry. Any unset / unrecognized value is fatal — better than booting with an ambiguous mode.

Where the mode lives at runtime

Package-level var currentMode Mode, set by Init and never reassigned. Reads are unsynchronized because writes only happen at startup (before any goroutine spawns) and Go's memory model guarantees the init-time write is visible. If a future ticket needs runtime mode flips (it shouldn't), the field becomes atomic.Value at that point.

Tests

paths.SetForTest(t, ModeMulti, "/tmp/test-root") swaps mode + state root for the duration of t, with a t.Cleanup that restores the previous values. Lets a test exercise multi-mode path resolution against a real tmpdir without polluting the suite's mode.

// internal/paths/test_helper.go
func SetForTest(t testing.TB, m Mode, root string) {
  t.Helper()
  prevMode, prevRoot := currentMode, testStateRoot
  currentMode, testStateRoot = m, root
  t.Cleanup(func() { currentMode, testStateRoot = prevMode, prevRoot })
}

testStateRoot is consulted by stateRoot() when set; otherwise the function falls through to the env-driven path. This keeps tests hermetic.

Adding orgID up the call stack

Most worktree/curator/projectclassify callers don't have an orgID today. Until D9 wires real orgs into request context, every call site passes paths.LocalDefaultOrg. That's a one-line addition per call. The lint rule (added in D2 for stores) extends here: a literal "default" string outside paths is an antipattern — callers should reference paths.LocalDefaultOrg.

§7

What we're NOT touching

  • ~/.claude/projects/ session JSONLs — that's Claude Code's own state, not ours. Lives outside internal/paths; same code path stays in internal/worktree/worktree.go. In multi mode, agent-runner runs each delegation as a separate process with its own HOME pointing at a per-run dir, so the .claude/projects path is naturally per-run. (Verified against the architecture's gVisor sandbox model — D8.)
  • ~/.claude/skills + Claude Code config inheritance (internal/skills/agents.go, importer.go) — same: not our state, not in scope. D8 owns the agent-runner-vs-user-home story for multi mode.
  • Postgres DSN / connection-string handling — that's whatever ticket plumbs db.Open()'s multi-mode branch (OpenPostgres(dsn) from D3 / D6). paths doesn't speak DSNs; DBPath errors out in multi.
  • Org context in request handlers — D9 (Org-scoping pass) extracts org_id from URL paths via Go 1.22 pattern routing. D4 just gives the resolver the function shape; D9 fills in real values.
  • Filesystem migration of existing local installs — local mode's resolved paths are byte-identical to today's (orgID stripped, no orgs/default/ segment). No on-disk data movement, no upgrade hook.
§8

Implementation steps

  1. Create internal/paths/ package Three files: paths.go (Mode type, Init, Current, all resolvers), test_helper.go (SetForTest), doc.go (package-level docstring). All public functions return (string, error); the error path is exclusively for mode-mismatch + missing UserHomeDir.
  2. Wire paths.Init into every entry point main.go + each cmd/*/Handle() — call paths.Init(modeFromEnv) as the first thing they do, before db.Open(). Fatal on init failure. Five entry points total: server (main.go), cmd/exec, cmd/migrate, cmd/resume, cmd/install, cmd/uninstall.
  3. Migrate the ~8 entry-point sites One commit per file (or grouped by logical area: db+config+agentproc, worktree, curator+projectclassify+projectbundle, cmd/*). Each is a one-line substitution at the resolution site plus an orgID parameter through any caller layer that didn't have it. Use paths.LocalDefaultOrg at the new call sites until D9.
  4. Unit tests for resolution internal/paths/paths_test.go: table-driven test against every (mode, function, args) combination. Multi-mode test seeds a SetForTest(t, ModeMulti, "/tmp/x") and asserts WorktreeRoot("orgA") ≠ WorktreeRoot("orgB") (the architecture's "two orgs non-overlapping" acceptance criterion). Local-mode test asserts the resolved paths match today's hand-rolled output byte-for-byte against a fixture.
  5. Integration smoke Boot the binary in local mode against a clean $HOME, confirm ~/.triagefactory/triagefactory.db is created at the expected path. Boot with TF_MODE=multi against a tmp /data mount (or override the state root via env for local dev), confirm the worktree root resolves under /data/orgs/default/... when fed LocalDefaultOrg — and rejects when fed an empty orgID.
  6. Update cmd/uninstall for multi-mode refusal Add the early-exit check: if paths.Current() == ModeMulti, print the "uninstall is local-only; tear down the deployment via your orchestrator" message and exit non-zero. Doesn't try to wipe /data.
  7. Documentation CLAUDE.md — add a "Mode flag & paths" bullet pointing at internal/paths as the single source of truth. docs/usage.md — document TF_MODE=local|multi for operators (multi mode isn't user-deployable yet, but the env var is read).
§9

Risks & open questions

Open questions for the owner

QuestionDefault if no answerWhy it matters
Strip orgs/default/ in local mode (architecture says yes), or always include it? Strip — preserves existing local installs' on-disk layout exactly. Architecture doc explicitly says so. Always-include would break every local install on first boot under D4 (the DB file moves from ~/.triagefactory/triagefactory.db → ~/.triagefactory/orgs/default/triagefactory.db); we'd need a one-shot migration. Stripping avoids that entirely.
Is the orgID argument required at every resolver, or can resolvers that don't need org-scoping (DBPath, SDKDir) skip it? Skip when not org-scoped. Resolvers that take orgID are the ones whose path differs by org. DBPath doesn't, because the local SQLite is per-binary and the multi Postgres has no path at all. Forcing an orgID arg on functions that ignore it is noise. Asymmetry is OK — the call site reads "ah, this is per-binary state."
How does the test suite resolve mode when most tests run before main.go sets it? Init-time default in tests: paths package's init() sets currentMode = ModeLocal as the safe default, so a test that doesn't explicitly call SetForTest gets local-mode behavior. Tests that need multi-mode call SetForTest(t, ModeMulti, tmpDir) explicitly. Forgetting to init in test-only code paths used to bite us with config; default-on prevents that.
What happens to /tmp/triagefactory-runs/<run-id>/ (the per-run scratch root) — is that org-scoped? No — local mode keeps it under /tmp; multi mode moves it under /data/orgs/<org>/worktrees/run-<id>/ (gVisor binds the run dir into the sandbox; /tmp isn't writable from inside). This was actually an explicit architecture decision (§6 filesystem layout): multi-mode worktrees live under /data for sandbox-mountability, not /tmp. Calling it out so the resolver matches.
Does cmd/uninstall in multi mode just refuse, or does it have a "wipe all orgs" multi-mode action? Refuse. Multi-mode tear-down is an orchestration concern (docker compose down -v, or platform-specific). A binary subcommand that walks /data and deletes everything is the wrong abstraction. Same reasoning as install's "you're a server, not a desktop app."

Acceptance

  • PASS Existing local installs see no on-disk change — ~/.triagefactory/triagefactory.db still resolves there, worktrees still under ~/.triagefactory/worktrees/, takeover dirs unchanged.
  • PASS Boot the binary with TF_MODE=multi + a tmp state root, call paths.WorktreeRoot("orgA") and paths.WorktreeRoot("orgB") — paths are non-overlapping (/data/orgs/orgA/worktrees vs /data/orgs/orgB/worktrees).
  • PASS Multi-mode paths.WorktreeRoot(LocalDefaultOrg) errors with "caller forgot to pass real org."
  • PASS Multi-mode paths.DBPath() errors with ErrModeMismatch.
  • PASS cmd/uninstall refuses in multi mode with a clear "use your orchestrator" message.
  • TEST Unit tests cover every (mode, resolver, args) combination; fixture test pins local-mode output byte-for-byte against today's path strings.
  • CODE No raw os.UserHomeDir() + ".triagefactory" compositions outside internal/paths/. Lint rule (forbidigo) enforces it.
  • DOC CLAUDE.md mentions the new package; docs/usage.md documents TF_MODE.

Out of scope (do NOT do here)

  • Real orgID propagation through request handlers — that's D9 (SKY-253)
  • Postgres DSN / connection-string handling — that's D3 (SKY-247) wiring
  • gVisor / agent-runner mode-aware sandboxing — that's D8 (SKY-252)
  • Any filesystem migration of existing local installs (we don't move anything)
  • Settings table or per-org config plumbing
