D1 — Adopt goose for migration tooling

Swap the hand-rolled migration runner in internal/db/migrations.go for pressly/goose/v3. Multi-tenant adds 8+ tables and modifies most existing ones; the next 12 months will see more schema motion than the last six. Goose is the standard tool for that — Go-native, embeds migrations, supports SQLite + Postgres with the same file format.

§1

Why now

The hand-rolled runner has carried us this far. It won't carry the multi-tenant rewrite.

What's about to land

  • D3 — Multi-tenant Postgres schema (SKY-247). 8+ new tables (orgs, teams, users, memberships, org_invites, sessions, integrations, org_limits, project_knowledge) plus org_id on every existing table — but in a separate migration set, not a backfill (see SKY-247).
  • Two backends from now on. SQLite continues to serve local mode; Postgres is the multi-mode backend. Two dialects with different syntax (BLOB vs BYTEA, AUTOINCREMENT vs SERIAL, no JSONB in SQLite, etc.). The hand-rolled runner has no concept of "which dialect am I running against."
  • Vault extension setup, RLS helper functions, RLS policies. Postgres-only DDL that must never run against SQLite.

What's painful about the current runner

  • No dialect routing. One migrationsFS embed; if a file has Postgres-only DDL, SQLite chokes.
  • Filename collisions already. 20260505_002_pending_review_diff_hunks.sql and 20260505_002_settings_table.sql share the same YYYYMMDD_NNN prefix — the runner sorts them lexicographically by full filename, which works by accident but isn't the documented invariant. Goose's monotonic sequence numbers prevent this.
  • No status command. "What migrations are applied?" requires a manual SELECT * FROM schema_migrations.
  • No down migrations. Currently fine (forward-only is policy — see CLAUDE.md), but goose supports both with -- +goose Up / -- +goose Down markers and we get the option for free.
  • Re-implementing what goose already does. The 159 lines of migrations.go exist to do what goose.Up does in one call. Removing that code is its own win.
Forward-only stays. Adopting goose doesn't change the project policy from CLAUDE.md: forward-only idempotent migrations, no wipe-and-reseed (real users in the wild). Goose just gives us a better tool to enforce that policy.

§3

Target state

pressly/goose/v3 as the runner. Same SQL files, same forward-only policy, two dialects supported via subdirectory split.

Library

github.com/pressly/goose/v3 — used as a library, not the binary. We embed migrations/ directories with go:embed, set the dialect (goose.SetDialect("sqlite3") or "postgres"), and call goose.Up(db, "migrations") at startup.

Tracking table

Goose uses goose_db_version(id, version_id, is_applied, tstamp). We migrate from schema_migrations in §4.

File format

Goose-flavored SQL with directional markers. Existing files become:

-- +goose Up
-- +goose StatementBegin
CREATE TABLE ...
-- +goose StatementEnd

-- +goose Down
-- +goose StatementBegin
DROP TABLE ...           // or just `SELECT 1;` for forward-only
-- +goose StatementEnd

StatementBegin/End is required when a single migration contains multiple statements separated by semicolons (which most of ours do). Goose uses semicolons as statement delimiters by default; the markers tell it "treat this whole block as one statement."

Forward-only as policy, not as syntax. Goose requires a -- +goose Down block to exist. We make it a single SELECT 1; no-op (or a comment explaining why down isn't supported). Don't write real down migrations — they'd violate the forward-only policy and create a footgun where someone does goose down on prod.
§4

Mapping & compat

Existing installs come in through the pre-goose schema_migrations table. The cutover collapses 18 legacy migrations into a single consolidated baseline and stamps existing installs at that baseline so the consolidated DDL never re-runs against a populated DB.

Four install states

The shim classifies each boot into one of four shapes. Every release that real users have ever installed shipped after the last hand-rolled migration, so the live install distribution is dominated by states 1 and 2; states 3 and 4 are guards against silent corruption.

StateDetectionAction
installAlreadyGooseManaged goose_db_version has rows No-op; goose.Up is the source of truth.
installLegacyRunnerPopulated schema_migrations has rows Stamp baseline + bootstrap rows into goose_db_version; goose.Up no-ops on the next call.
installFresh no app tables, no version metadata No-op; goose.Up runs the consolidated baseline.
installPreRunner app tables present (sentinel: entities) but neither goose_db_version nor populated schema_migrations Refuse with ErrPreRunnerInstall. The operator must run v1.10.1 first (which stamps schema_migrations), or wipe the local DB.

The pre-runner branch closes a hole the simpler "is the legacy table populated?" probe missed: a DB with app tables but no version metadata at all (predates the hand-rolled runner) would otherwise be classified as fresh, run the consolidated baseline's CREATE TABLE IF NOT EXISTS against tables that already exist (no-op), get stamped at baseline by goose, and end up silently behind whatever ALTERs subsequent migrations applied. Failing fast routes the operator through v1.10.1 (the last release with the hand-rolled runner) where stampBaselineIfNeeded brings the DB into the legacy-runner-populated state this shim handles cleanly.

Version number encoding

Goose's tracking column is version_id BIGINT. We pick a 12-digit YYYYMMDDNNNN encoding so future migrations sort numerically and lexicographically the same way. The baseline is 202605090001 (today, sequence 1). Future migrations land alongside as NNNNNNNNNNNN_description.sql.

Stamp shim — startup

Before goose.Up:

// internal/db/migrations.go (sketch)
func importLegacyVersionsIfNeeded(db *sql.DB, dialect string) error {
  if dialect != "sqlite3" { return nil }
  state, err := detectInstallState(db)
  if err != nil { return err }
  switch state {
  case installAlreadyGooseManaged, installFresh:
    return nil
  case installPreRunner:
    return ErrPreRunnerInstall
  case installLegacyRunnerPopulated:
    // fall through to the stamp logic below
  }
  db.Exec(gooseVersionTableDDL)
  // INSERT ... WHERE NOT EXISTS — atomic against concurrent boots,
  // idempotent across re-launches.
  db.Exec(stampSQL, int64(0), int64(0))
  db.Exec(stampSQL, baselineVersionID, baselineVersionID)
  return nil
}
Audit trail / rollback: the legacy schema_migrations table is left in place forever. It's tiny, it's the source of truth for "what did the old runner think had run," and dropping it later — when no one wants to read it — is a separate ticket if it ever matters. It's also the safety net if the stamp shim is ever found to be buggy.
§5

File layout

SQLite and Postgres get separate migration directories. Same package, different go:embed roots, different goose dialect.

internal/db/
├── db.go                          // Open()/OpenAt() — unchanged
├── migrations.go                  // goose wrapper + stamp shim
├── migrations-sqlite/             // replaces migrations/
│   └── 202605090001_baseline.sql  // consolidated post-state of the prior 18 migrations
└── migrations-postgres/           // scaffolded; SKY-247 (D6) fills it
    └── README.md

Why split, not single dir with conditional execution: a SQL file with BYTEA won't parse in SQLite even if guarded by WHERE dialect = 'postgres' — SQLite's parser fails before the runtime check. Two trees keep each dialect's files independently parseable and tooling-friendly (you can sqlite3 < file.sql any sqlite migration to sanity-check).

The empty migrations-postgres/ directory ships in this PR with a README pointing at SKY-247, so D6 has a clear destination and the dialect-routing code is reviewable in isolation.

The legacy internal/db/migrations/ directory is deleted. Git history preserves the granular per-migration files; the live tree's source of truth is the consolidated baseline.

Embedding

//go:embed migrations-sqlite/*.sql
var migrationsSQLite embed.FS

//go:embed migrations-postgres/*.sql
var migrationsPostgres embed.FS

func migrationsFor(dialect string) (fs.FS, string, error) {
  switch dialect {
  case "sqlite3": return migrationsSQLite, "migrations-sqlite", nil
  case "postgres": return migrationsPostgres, "migrations-postgres", nil
  }
  return nil, "", fmt.Errorf("unsupported dialect %q", dialect)
}
§6

Runner integration

db.Migrate() stays the same shape. Internals swap out.

// internal/db/migrations.go after
package db

import (
  "database/sql"
  "embed"
  "fmt"
  "io/fs"

  "github.com/pressly/goose/v3"
)

//go:embed migrations-sqlite/*.sql
var migrationsSQLiteFS embed.FS

//go:embed migrations-postgres
var migrationsPostgresFS embed.FS  // directory embed — README only for now

// Migrate brings the schema to head via goose. Dialect is sqlite3 today;
// the postgres path is scaffolded but not yet exercised (SKY-247 / D6).
func runMigrations(db *sql.DB) error {
  dialect := detectDialect(db)
  if err := importLegacyVersionsIfNeeded(db, dialect); err != nil {
    return fmt.Errorf("legacy import: %w", err)
  }
  treeFS, treeDir, err := migrationsFor(dialect)
  if err != nil { return err }
  goose.SetBaseFS(treeFS)
  if err := goose.SetDialect(dialect); err != nil { return err }
  return goose.Up(db, treeDir)
}

Dialect detection

This PR hardcodes detectDialect to return "sqlite3". The pgx-based driver-name probe lands in SKY-247 (D6) alongside the first real Postgres consumer; centralizing the decision behind a function now keeps the call sites stable when that switch happens. Treat detectDialect as the seam, not a finished implementation.

Baseline body shape

Plain semicolon-delimited DDL with goose's -- +goose Up opener and a -- +goose Down block holding SELECT 'down not supported';. No StatementBegin/End blocks needed — the baseline is just CREATE TABLE and CREATE INDEX statements; goose's default parser handles them. (StatementBegin/End is for migrations whose statements contain semicolons inside, like triggers.)

One inline gotcha: goose's annotation parser keys on the literal prefix -- +goose. Comments inside the migration that mention -- +goose Down as a quoted reference will error out — keep doc text away from that prefix.
§7

CLI subcommands

Two new top-level subcommands for ops visibility. Live next to the existing exec + status in main.go's dispatch.

triagefactory migrate status

Renders applied / pending state for each migration in the routed dialect tree. Today the tree contains a single baseline; the format scales as new migrations land.

$ ./triagefactory migrate status
    Status                      Migration
    ====================================
    Applied                     migrations-sqlite/202605090001_baseline.sql

    db version: 202605090001

triagefactory migrate up

Same effect as starting the server (which auto-migrates on boot), but exits after the migration completes. Useful when an operator wants to apply schema changes from a maintenance window separately from the server restart.

Why no migrate down / redo / create

Forward-only policy. Down/redo would footgun real installs. create isn't needed — new migrations are added by writing a file directly; the IDE is sufficient (no need for goose's date-stamped scaffold).

§8

Postgres readiness

D1 doesn't add Postgres support to TF (D6 does). But the goose wrapper has to handle Postgres correctly so D6 can drop in without rewiring the routing layer.

What this PR does

  • Adds the migrations-postgres/ directory with a non-hidden README so the //go:embed migrations-postgres directive doesn't fail at build time
  • migrationsFor("postgres") resolves to the empty tree without erroring; goose against an empty directory is a clean no-op
  • detectDialect is centralized as a function but currently returns the constant "sqlite3"

What this PR does not do

  • No pgx driver detection or Postgres CI — both land alongside the first real Postgres consumer in D6
  • No actual Postgres schema yet (D6)
  • No TF_MODE=multi branch in db.Open() — that's D4 / SKY-248
  • No connection string parsing for Postgres URLs (D6)

The CI hook is the load-bearing bit: once D3 starts adding files, CI immediately verifies they at least parse + apply against a real Postgres without anyone having to wire it up.

  2. Author the consolidated baseline Walk the 18 hand-rolled migrations and produce internal/db/migrations-sqlite/202605090001_baseline.sql capturing the cumulative post-state. Use plain semicolon-delimited DDL with goose's -- +goose Up / -- +goose Down markers; the Down block is a SELECT 'down not supported'; no-op. Tables ordered by FK dependency so a fresh install creates them cleanly.
  3. Replace the runner Rewrite migrations.go per §6. Drop the lex-sort-and-apply loop and stampBaselineIfNeeded; the new shim is goose-shaped. Keep the Migrate(db) exported signature so main.go needs no edits.

  7. Wire CLI subcommands New cmd/migrate/ package with up + status. Wire "migrate" into main.go's argv dispatch alongside resume / install. up calls db.Migrate; status renders applied/pending via goose.CollectMigrations + goose.GetDBVersion. No down exposed — see §1's note.

§10

Risks & rollback

RiskMitigationRollback
Stamp shim mis-fires → baseline runs against a populated DB → CREATE TABLE IF NOT EXISTS no-op but UNIQUE indexes etc. silently succeed → next migration drift Every CREATE in the baseline uses IF NOT EXISTS. The shim's "shouldn't run" path is a single boolean (legacy table exists with rows). Tests cover both branches and the borderline empty-table case. Revert the PR; the old hand-rolled runner reads schema_migrations (untouched) and resumes.
Pre-runner install (app tables, no version metadata) gets silently stamped at baseline → schema behind the baseline shape because the consolidated DDL is CREATE TABLE IF NOT EXISTS and won't add missing columns Detection now classifies this state (installPreRunner) and returns ErrPreRunnerInstall before any write. Test TestMigrate_PreRunnerInstallErrors pins the contract. Operator runs v1.10.1 first (which stamps schema_migrations via stampBaselineIfNeeded), then upgrades to this version. Or wipes ~/.triagefactory/triagefactory.db for a fresh install.
Goose parser chokes on the consolidated baseline (e.g. an embedded -- +goose Down substring inside a header comment) Goose's annotation parser keys on lines starting with -- +goose; avoid that prefix in comment text. Tests run the full goose.Up path so any parse error fails CI before it reaches users. Patch the offending comment text and re-release.
Goose's tx semantics diverge from the old runner's (per-migration transactions) Default goose behavior: each migration runs in its own tx. Verified against goose's source. —
Future Postgres migrations land but the dialect router is wired wrong Router lives in one place (migrationsFor); D6's first Postgres migration exercises it end-to-end before any production user reaches it. —
The escape hatch: the legacy schema_migrations table is left in place forever as an audit trail / rollback safety net. Reverting this PR puts the old runner back in charge against an untouched legacy table; goose_db_version can be dropped.
