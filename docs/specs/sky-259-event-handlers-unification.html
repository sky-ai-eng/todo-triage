event_handlers unification

Collapse task_rules + prompt_triggers into one event_handlers table with a kind ∈ ('rule','trigger') discriminator. The semantics already converged after the team/agent reframe — rule and trigger are the same primitive with an "auto-claim?" knob. Two tables / two stores / two RLS policies / two router queries become one of each.

SCOPE · 1 PG + 1 SQLite migration + new EventHandlerStore + router rewrite + FE swap to /api/event-handlers NO COMPAT LAYER · old tables + old endpoints dropped in this PR UNBLOCKS · SKY-261 (D-Claims) can add claimed_by_agent_id to one table instead of two

§0

Phasing

One PR, three phases that all ship together. Splitting the migration from the store rewrite would leave an in-between commit with old code reading dropped tables; splitting the API rewrite from the FE swap would break the in-tree frontend on the same commit. The whole unification is one atomic change.

A
Schema
event_handlers table both backends. Backfill from task_rules + prompt_triggers, then DROP both in the same migration. Forward-only — there's no rollback to half-undo to.
B
Store + router
New EventHandlerStore. Router collapses two-phase matching into one query. Spawner reads prompt_id off the row when kind='trigger'. Old TaskRuleStore + TriggerStore deleted.
C
API + FE swap
New /api/event-handlers with ?kind=rule / ?kind=trigger filter. Two existing FE pages keep their split UX, both call the unified endpoint. Old /api/task-rules + /api/triggers routes deleted.
§1

Why now

The post-SKY-269 schema is settled enough that this is purely an internal refactor. Doing it before D-Claims (SKY-261) means adding the claim-FK columns to one table instead of two.

WinConcrete shape
Router collapseOne SELECT ... FROM event_handlers WHERE event_type = ? AND enabled = TRUE instead of GetEnabledForEvent on task_rules + GetActiveForEvent on prompt_triggers. 2 queries → 1 query per event.
RLS surface halvedFour policies (insert/update/select/delete) instead of eight. The system-row + admin-only-on-org-visibility patterns we ironed out across SKY-246/260 only need to be applied once.
D-Claims simplificationauto_claim_by_agent_id column lives on one table. Router's "stamp claim on trigger fire" branch becomes "stamp claim when auto_claim_by_agent_id IS NOT NULL" — orthogonal to the unification.
Rule ↔ trigger promotion"Promote this rule to a trigger" becomes one atomic UPDATE through a dedicated POST /api/event-handlers/{id}/promote endpoint. Today it's a row delete on one table + row insert on another, with no atomicity guarantee.
§2

Schema delta

Two tables, mostly overlapping, into one with per-kind nullables + CHECK constraints. The column matrix below is the unified shape — green-highlighted columns are rule-only, pink-highlighted are trigger-only, cyan-highlighted is the new discriminator.

task_rules — today
  • id, org_id, team_id, visibility, creator_user_id, source
  • event_type, scope_predicate_json, enabled, name
  • created_at, updated_at
  • default_priority REAL · 0.5
  • sort_order INT · 0 (UI ordering)
prompt_triggers — today
  • id, org_id, team_id, visibility, creator_user_id, source
  • event_type, scope_predicate_json, enabled
  • created_at, updated_at
  • prompt_id TEXT · FK prompts.id
  • trigger_type TEXT · 'event'
  • breaker_threshold INT · 4
  • min_autonomy_suitability REAL · 0.0
↓ unify ↓
event_handlers — after this PR
  • kind TEXT · 'rule' | 'trigger' · CHECK
  • id, org_id, team_id, visibility, creator_user_id, source
  • event_type, scope_predicate_json, enabled, name
  • created_at, updated_at
  • default_priority REAL · nullable · rule-only
  • sort_order INT · nullable · rule-only
  • prompt_id TEXT · nullable · trigger-only · FK
  • breaker_threshold INT · nullable · trigger-only
  • min_autonomy_suitability REAL · nullable · trigger-only

CHECK constraints (the per-kind shape gate)

ALTER TABLE event_handlers ADD CONSTRAINT event_handlers_rule_shape
  CHECK (
    kind <> 'rule' OR (
      prompt_id IS NULL
      AND breaker_threshold IS NULL
      AND min_autonomy_suitability IS NULL
      AND default_priority IS NOT NULL
      AND name IS NOT NULL
    )
  );

ALTER TABLE event_handlers ADD CONSTRAINT event_handlers_trigger_shape
  CHECK (
    kind <> 'trigger' OR (
      prompt_id IS NOT NULL
      AND breaker_threshold IS NOT NULL
      AND min_autonomy_suitability IS NOT NULL
      AND default_priority IS NULL
      AND sort_order IS NULL
    )
  );
One row per logical handler — promotion ("turn this rule into a trigger") becomes UPDATE the kind + populate the trigger-only fields + clear the rule-only ones. The CHECK constraints validate the transition atomically.
§3

Routing collapse

The router today fires two store queries per event (rules then triggers). Post-unification it's one query and a single matcher loop.

Before · 2 queries per event
event: github:pr:ci_check_failed
↓
stores.TaskRules.GetEnabledForEvent(orgID, event_type)
↓ for each match
create unclaimed task
— then —
stores.Triggers.GetActiveForEvent(orgID, event_type)
↓ for each match
spawner.tryAutoDelegate(task, trigger)
After · 1 query
event: github:pr:ci_check_failed
↓
stores.EventHandlers.GetEnabledForEvent(orgID, event_type)
↓ for each match, switch on row.Kind
kind='rule' → create unclaimed task
kind='trigger' → create task (claimed by team's agent post-D-Claims) + dispatch
The order today is rule-first / trigger-second. Post-unification we preserve it by ordering the matcher loop by kind (rules before triggers) — same semantics, same observable behavior. The DB query returns both in one round trip.
§4

Migration design

Forward-only on both backends, single migration per backend: create the new table, backfill from both old tables, drop the old tables. No staged rollout — the store rewrite, router rewrite, and FE swap all land in the same PR, so nothing reads the old tables by the time the DROP runs.

Postgres

SQLite

Same shape minus the RLS section, and with one prerequisite step: add a unique index on prompts (id, org_id) so the composite FK can target it (SQLite, like Postgres, requires the FK column order to match an existing unique index or constraint on the parent).

-- (0) Parity prerequisite: SKY-269 added org_id columns via ALTER but
-- didn't add the composite UNIQUE constraints Postgres has on every
-- parent table that fields composite FKs. Audit confirmed: 8 parent
-- tables are missing UNIQUE(id, org_id) in SQLite vs Postgres. Eight
-- one-line indexes close the schema-shape gap with zero data risk —
-- id is already PK-unique on every table, so (id, org_id) is trivially
-- unique on existing data.
--
-- These indexes are the precondition for composite FKs. event_handlers
-- (this PR) uses one; future tables can declare composite FKs without
-- per-table prep. The 25+ existing child-table plain FKs are not
-- rebuilt in this PR — see §9 risk row for scope rationale.
CREATE UNIQUE INDEX IF NOT EXISTS prompts_id_org_unique           ON prompts (id, org_id);
CREATE UNIQUE INDEX IF NOT EXISTS projects_id_org_unique          ON projects (id, org_id);
CREATE UNIQUE INDEX IF NOT EXISTS entities_id_org_unique          ON entities (id, org_id);
CREATE UNIQUE INDEX IF NOT EXISTS events_id_org_unique            ON events (id, org_id);
CREATE UNIQUE INDEX IF NOT EXISTS tasks_id_org_unique             ON tasks (id, org_id);
CREATE UNIQUE INDEX IF NOT EXISTS runs_id_org_unique              ON runs (id, org_id);
CREATE UNIQUE INDEX IF NOT EXISTS pending_reviews_id_org_unique   ON pending_reviews (id, org_id);
CREATE UNIQUE INDEX IF NOT EXISTS curator_requests_id_org_unique  ON curator_requests (id, org_id);

The migration runs with PRAGMA foreign_keys=OFF around the DROP step (standard goose-managed dance) so the legacy task_rules.prompt_id + prompt_triggers.prompt_id FKs don't complicate the table drop.

Composite-FK parity, scoped pragmatically. Eight parent indexes close the schema-shape gap for free. New tables (this PR + future) can declare composite FKs natively. The 25+ existing SQLite child FKs that target plain parent(id) stay as-is in this PR — rewriting them requires SQLite's full table-rebuild dance per child, with no behavioral payoff at N=1 (single org means the plain FK already gives single-tenant integrity). A future child-FK parity sweep is bounded, optional, and not load-bearing for the unification.
ID preservation: both backfills copy id verbatim. A shipped system rule with id abc... in task_rules stays abc... in event_handlers. Anything outside this PR that references handler IDs (run rows via FK, audit logs) keeps working without rewrites.
§5

EventHandlerStore

One interface replaces TaskRuleStore + TriggerStore. Methods accept a kind filter where the caller distinguishes; the router's hot path doesn't.

type EventHandlerStore interface {
  // Seed inserts the v1 shipped handlers (system rules + system triggers
  // together) if they aren't already present. Idempotent.
  Seed(ctx context.Context, orgID string) error

  // Read paths.
  List(ctx context.Context, orgID string, kind string) ([]domain.EventHandler, error) // kind="" returns both
  Get(ctx context.Context, orgID, id string) (*domain.EventHandler, error)
  GetEnabledForEvent(ctx context.Context, orgID, eventType string) ([]domain.EventHandler, error) // the router's one query
  ListForPrompt(ctx context.Context, orgID, promptID string) ([]domain.EventHandler, error)

  // Write paths — kind required for Create / Promote; mutable fields
  // are per-kind so the handler endpoints can route by URL prefix.
  Create(ctx context.Context, orgID string, h domain.EventHandler) error
  Update(ctx context.Context, orgID string, h domain.EventHandler) error
  SetEnabled(ctx context.Context, orgID, id string, enabled bool) error
  Delete(ctx context.Context, orgID, id string) error
  Reorder(ctx context.Context, orgID string, ids []string) error // rule-only — UI ordering
}

Domain shape:

type EventHandler struct {
  ID                       string
  Kind                     string   // "rule" | "trigger"
  EventType                string
  ScopePredicateJSON       *string
  Enabled                  bool
  Source                   string   // "system" | "user"
  Name                     string   // rule: required; trigger: derived from prompt name in UI
  // kind="rule":
  DefaultPriority          *float64
  SortOrder                *int
  // kind="trigger":
  PromptID                 string
  BreakerThreshold         *int
  MinAutonomySuitability   *float64
  CreatedAt, UpdatedAt     time.Time
}

The old domain.TaskRule and domain.PromptTrigger types are deleted in this PR. Anywhere that used them now consumes domain.EventHandler directly; the per-kind fields are nullable and the caller narrows on Kind. No adapter shim — the compiler enforces the migration.

Why a single store + kind filter rather than two stores backed by one table: consumers like the router only ever want "all handlers matching this event," not a kind-partitioned slice. Two stores would duplicate the SQL bodies and force every new field to be added in two places. The kind filter is rarely used (only by the handler shims) and adds one CHECK at the SQL level.
§6

API + frontend

One unified endpoint family /api/event-handlers replaces /api/task-rules + /api/triggers. The kind is passed as a query param on reads and a body field on writes — there's no URL-prefix split because there's no longer a backend split to mirror. The two existing FE pages keep their split UX (the rules tab vs the triggers tab is still useful for users) — they both point at the unified endpoint with the right kind filter.

EndpointMaps to
GET /api/event-handlers?kind=ruleEventHandlers.List(orgID, kind="rule") — unfiltered kind= returns both
POST /api/event-handlersEventHandlers.Create(orgID, h) — h.kind required, validated against per-kind shape
PUT /api/event-handlers/{id}EventHandlers.Update(...) — kind is immutable here; promotion is a separate POST
POST /api/event-handlers/{id}/promoterule → trigger transition: validates incoming trigger-only fields, runs the single-UPDATE shape swap
POST /api/event-handlers/reorderEventHandlers.Reorder(orgID, ids) — rule-only (server rejects if any id is a trigger)

Frontend changes (in this PR)

  • Two existing page components stay split. RulesPage.tsx hits /api/event-handlers?kind=rule; TriggersPage.tsx hits ?kind=trigger. Same shape they have today — just one fetch URL change each.
  • One shared useEventHandlers(kind) hook replaces the existing useTaskRules + useTriggers hooks. The query-cache key uses the kind so both pages cache independently and a write on one doesn't invalidate the other.
  • Create/update payloads gain the explicit kind field. Forms stay split — the rules form posts {kind: "rule", ...}; trigger form posts {kind: "trigger", ...}.
  • Types (frontend/src/types.ts): existing TaskRule and PromptTrigger become EventHandler with discriminated-union narrowing on kind. Both page components consume the narrowed view via a type guard helper.
Frontend stays split by UX, unified at the API. Users still see two tabs (rules manage human-triage queues; triggers configure auto-delegation). The backend no longer pretends they're different things — and a future "unified handler config" page is a free move whenever the FE wants to take it.

Out of scope

  • Unified FE page. The split UX stays — rules tab + triggers tab. A future "all event handlers" page is a free move once the FE wants it; not in this PR.
  • claimed_by_agent_id column. D-Claims (SKY-261) adds this. Unification + claims in one PR would balloon the review surface; SKY-261 lands on the unified table afterward.
  • Vocab change. "rule" and "trigger" stay as the user-facing nouns. The unification is purely backend; users still create rules and create triggers.
  • Per-prompt default trigger — the ticket's "I wonder if we could share prompt_id and just make it a default" suggestion. Filed as a follow-up; not in this PR.

§9

Risks

RiskMitigation
Big migration; first PR after SKY-269 lands with the same dev team in flight — merge conflicts with SKY-261 work. SKY-261 hasn't started; sequencing this first means it inherits the unified table. Conflict surface is the spec doc + the agent test fixtures, both small.
Backfill copies row data but a subtle column mismatch (e.g. timezone handling on Postgres vs SQLite for created_at) could silently drop a field. Bootstrap-shape tests assert row-for-row equivalence between source tables and event_handlers (per-column, all rows) before the DROP runs in a separate test fixture that pauses the migration mid-flight.
System-row patterns differ slightly between task_rules + prompt_triggers (different namespace UUIDs in ShippedTaskRule.UUIDFor vs ShippedPromptTrigger.UUIDFor). Preserve both. The unified ShippedEventHandlers list assembles them with their existing derivations, so deterministic IDs survive across the migration without remapping.
The CHECK constraints reject mid-promote writes if the application tries to set kind='trigger' but hasn't yet populated all trigger-only fields. Promote is an explicit POST endpoint (/api/event-handlers/{id}/promote), not a generic PUT kind= swap. The handler validates the target shape before issuing one atomic UPDATE that populates trigger fields and clears rule fields together.
The router today processes rules and triggers in two separate matcher loops; collapsing into one might hide a subtle ordering invariant. Audit before merge: list every place the router compares "what fired" between the two passes. Today's behavior pins "rules fire first, then triggers fire against the resulting tasks." Preserve by ordering the unified matcher loop by kind.
Dropping the old tables in the same migration means no escape valve if a consumer outside this PR (a Go test, an unmigrated callsite) still hits the old table at runtime. Compiler catches Go callers — the old stores + types are deleted, so anything unswept fails the build. For the SQL layer: grep -r 'task_rules\|prompt_triggers' internal/ after the rewrite must return only the migration file itself.
SQLite parent-index sweep (8 new UNIQUE (id, org_id) indexes) touches tables outside the unification's strict scope — could collide with concurrent work on those tables. Indexes are additive and idempotent (IF NOT EXISTS); they don't change existing column semantics or rebuild tables. Existing child FKs continue to target plain parent(id). Conflict surface is the migration file's text only.
Adding parent unique indexes but not rewriting child FKs to composite leaves a half-fix: SQLite still lets a child row's org_id differ from the parent's org_id in principle. At N=1 there's only one org_id value in the entire database (the local sentinel), so divergence is structurally impossible. The full child-FK rewrite is a deferred parity sweep, not load-bearing for local-mode correctness.
