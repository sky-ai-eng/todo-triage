§1

Why per-resource interfaces

Two questions: why interfaces at all, and why split per resource instead of one big Store.

Why interfaces at all

  • Two backends. SQLite (local) and Postgres (multi). Today every callsite imports internal/db directly and calls package-level functions like db.FindOrCreateTask(database, ...). Adding a Postgres path means either (a) if/else inside every function — combinatorial mess — or (b) an interface boundary that picks one implementation at startup. (b) is the only sane shape.
  • RLS context plumbing. Postgres wants SELECT set_config('request.jwt.claims', '{...}', true) per transaction so RLS policies see the right user/org. That belongs in the storage layer, not in every handler.
  • Testing. Today every handler test boots a real SQLite. Per-resource interfaces + mockery-generated mocks give us fast in-memory handler tests; conformance suites still exercise real backends.

Why per-resource (not one big Store)

  • Standard Go. io.Reader, http.Handler, fmt.Stringer — small, focused interfaces accepted at consumer sites. "Accept interfaces, return structs." A monolithic Store with 190 methods isn't an interface; it's a typed bag.
  • Handler dependencies stay honest. A handler that only reads tasks accepts db.TaskStore. Its mockery-generated mock has 16 methods, not 190. Reviewing a PR, you can see exactly which resources a handler touches by its constructor signature.
  • Migrate wave by wave. Land PromptStore first, get both backends matching, ship it. Then RuleStore. Each store is its own self-contained refactor; the build stays green between waves.
This is mechanical, not architectural. The interfaces exist already, in spirit — every db.X function is a method-in-waiting. We're rearranging code, not designing new abstractions.

Old internal/db/<resource>.go files are repurposed, not deleted. internal/db/tasks.go — interface only

  "context"
  "time"

  "github.com/sky-ai-eng/triage-factory/internal/domain"

//go:generate mockery --name=TaskStore --output=./mocks --case=underscore --with-expecter

// TaskStore is the persistence boundary for tasks. orgID arguments are
// always required — local mode passes the literal LocalDefaultOrg,
// multi mode passes the request's org UUID. SQLite impl asserts
// orgID == LocalDefaultOrg; Postgres impl includes org_id in WHERE
// clauses as defense in depth and relies on RLS via SET LOCAL claims
// (set by WithTx).
type TaskStore interface {
  FindOrCreate(ctx context.Context, orgID, entityID, eventType, dedupKey, primaryEventID string, defaultPriority float64) (*domain.Task, bool, error)
  FindOrCreateAt(ctx context.Context, orgID, entityID, eventType, dedupKey, primaryEventID string, defaultPriority float64, createdAt time.Time) (*domain.Task, bool, error)
  Bump(ctx context.Context, orgID, taskID, eventID string) error
  Close(ctx context.Context, orgID, taskID, closeReason, closeEventType string) error
  CloseAllForEntity(ctx context.Context, orgID, entityID, closeReason string) (int, error)
  SetStatus(ctx context.Context, orgID, taskID, status string) error
  RecordEvent(ctx context.Context, orgID, taskID, eventID, kind string) error
  GetByID(ctx context.Context, orgID, taskID string) (*domain.Task, error)
  ListActiveByEntity(ctx context.Context, orgID, entityID string) ([]domain.Task, error)
  ListActiveByEntityAndType(ctx context.Context, orgID, entityID, eventType string) ([]domain.Task, error)
  Queued(ctx context.Context, orgID string) ([]domain.Task, error)
  ByStatus(ctx context.Context, orgID, status string) ([]domain.Task, error)
  CountConsecutiveFailedRuns(ctx context.Context, orgID, entityID, promptID string) (int, error)
  ActiveTaskRefsForEntities(ctx context.Context, orgID string, entityIDs []string) ([]PendingTaskRef, error)
  EntityIDsWithActiveTasks(ctx context.Context, orgID, source string) (map[string]struct{}, error)
}

// PendingTaskRef stays here — value type returned by stores.

queryer abstraction

// internal/db/sqlite/queryer.go (parallel in postgres)
package sqlite

type queryer interface {
  ExecContext(ctx context.Context, query string, args ...any) (sql.Result, error)
  QueryContext(ctx context.Context, query string, args ...any) (*sql.Rows, error)
  QueryRowContext(ctx context.Context, query string, args ...any) *sql.Row
}

Both *sql.DB and *sql.Tx satisfy queryer — same store code runs inside or outside a transaction.

switch runmode.Current() {
case runmode.Local:
  database, err := db.Open()
  // ... handle err ...

case runmode.Multi:
  adminConn, err := postgres.OpenAdmin(cfg.PG.AdminDSN)  // supabase_admin / superuser
  appConn,   _   := postgres.OpenApp(cfg.PG.AppDSN)       // authenticator + tf_app
  stores = postgres.New(adminConn, appConn)
}
// Pass stores.Tasks, stores.Runs, etc. to specific handlers.
Why two Postgres connection pools at startup: deploy-time operations (running goose migrations, seeding system_prompt_versions) write to roles that are REVOKEd from tf_app per D3. They run on the admin connection. Request traffic runs on tf_app. Local mode is single-conn; SQLite has no role concept. The bundle's Tx wrapper for request handlers always uses the app pool — admin-only operations call postgres.New's helpers directly during startup.

Transactions & RLS plumbing

Any handler that mutates rows in two stores atomically (creating a run + emitting an event, closing a task + recording the close event) needs a transaction. WithTx is the closure pattern.

Interface

// internal/db/store.go
type TxStores struct {
  // Same fields as Stores, each tx-bound.
  Tasks    TaskStore
  Runs     RunStore
  Entities EntityStore
  // ...
}

// TxRunner runs fn inside a single DB transaction. Postgres impl
// additionally calls SELECT set_config('request.jwt.claims', $1, true)
// so RLS policies see the right claims for this tx. SQLite impl ignores
// orgID/userID (asserts orgID == LocalDefaultOrg).
type TxRunner interface {
  WithTx(ctx context.Context, orgID, userID string, fn func(TxStores) error) error
}

SQLite impl

// internal/db/sqlite/tx.go
func (s *Store) WithTx(ctx context.Context, orgID, userID string, fn func(db.TxStores) error) error {
  if orgID != db.LocalDefaultOrg {
    return fmt.Errorf("sqlite WithTx: orgID must be %q, got %q", db.LocalDefaultOrg, orgID)
  }
  tx, err := s.conn.BeginTx(ctx, nil)
  if err != nil { return err }
  defer tx.Rollback()  // no-op after commit

  txStores := db.TxStores{
    Tasks:    newTaskStore(tx),
    Runs:     newRunStore(tx),
    Entities: newEntityStore(tx),
    // ... all 22 stores wired against the tx
  }
  if err := fn(txStores); err != nil { return err }
  return tx.Commit()
}

Postgres impl

// internal/db/postgres/tx.go
func (s *Store) WithTx(ctx context.Context, orgID, userID string, fn func(db.TxStores) error) error {
  tx, err := s.app.BeginTx(ctx, nil)
  if err != nil { return err }
  defer tx.Rollback()

  // Set the JWT claims that RLS policies + tf.current_user_id() /
  // tf.current_org_id() helpers read from. set_config(..., true)
  // scopes to this tx — does not leak to other pool connections.
  claims, err := json.Marshal(map[string]any{
    "sub":    userID,
    "org_id": orgID,
  })
  if err != nil { return err }
  if _, err := tx.ExecContext(ctx,
    `SELECT set_config('request.jwt.claims', $1, true)`, string(claims)); err != nil {
    return err
  }

  txStores := db.TxStores{ /* same wiring, all stores against tx */ }
  if err := fn(txStores); err != nil { return err }
  return tx.Commit()
}

Caller pattern

err := h.tx.WithTx(ctx, orgID, userID, func(t db.TxStores) error {
  run, err := t.Runs.Create(ctx, orgID, ...)
  if err != nil { return err }
  return t.Events.Insert(ctx, orgID, ...)
})
Where orgID and userID come from in D2: handlers in local mode pass db.LocalDefaultOrg and a synthetic local user constant. D7 introduces request-scoped auth context (authctx.ClaimsFromContext(ctx)) and rewrites handlers to extract claims from ctx. The TxRunner signature accepts them explicitly in D2 so the call sites are visible — D7 just changes where they come from, not the shape of WithTx.

Why WithTx isn't owned by individual stores: a handler that does "create run + emit event + bump counter" needs one transaction across three stores. If each store opened its own tx, you'd get partial-state on a mid-way failure. The handler knows the atomicity boundary; the storage layer doesn't. WithTx is that boundary.

§8

Both backends, day one

No stubs, no panic("D3 fills this"), no phase-1 wrappers. The whole point of doing this after D3 lands is that we can write real Postgres impls now.

What changes per backend

  • SQLite: SQL is moved verbatim from existing internal/db/<resource>.go into internal/db/sqlite/<resource>.go. Method signatures grow ctx + orgID (+ userID where relevant). Each method asserts orgID == db.LocalDefaultOrg at the top.
  • Postgres: SQL is written fresh against D3's schema. Reads include org_id = $N in WHERE as defense in depth (RLS also enforces). Writes set org_id explicitly. Vault calls use the vault_put_org_secret / vault_get_org_secret / vault_delete_org_secret SECURITY DEFINER helpers from D3. OCC uses the D3 update_project_knowledge function (raises SQLSTATE 40001). Composite FKs from D3 enforce cross-tenant integrity; tests assert SQLSTATE 23503 on cross-org refs.

Multi-only stores in SQLite

Stores that exist solely for multi-tenant features (Org, Membership, User, Session) still need SQLite impls. Three response patterns:

  • Synthetic default. OrgStore.GetByID(ctx, "default") returns {ID: "default", Name: "Local"}. OrgStore.ListAll(ctx) returns one entry. MembershipStore.RoleOf(ctx, "default", localUserID) returns "owner".
  • Sentinel error. SessionStore.Insert(...) in SQLite returns db.ErrNotApplicableInLocal. The auth path is gated behind runmode.Current() == Multi, so this is a safety net for code that escapes that gate.
  • No-op silently. MembershipStore.Add(...) in SQLite returns nil — nothing to add to, and the caller (org admin UI in D14) is multi-only anyway.

SQLite-only paths in Postgres

None. Anything SQLite does, Postgres does the same shape. The reverse direction (Postgres-only) is what produces the multi-only sentinel handling above.

"Every function written in both languages" is a hard rule. Conformance suite (§10) is the gate — every interface method has a test that runs against both backends. If a SQLite impl is missing, the build fails at var _ db.TaskStore = (*taskStore)(nil). If a Postgres impl is missing, the pgtest conformance run fails. No "we'll add Postgres later."
§9

System prompt seeder

The seeder ran on the request connection against a single global prompts table pre-D3. D3 changed both: prompts are per-org (composite PK (org_id, id)), and system_prompt_versions is REVOKEd from tf_app. D2 re-shapes the seeder to use stores + a per-org loop.

The new shape

// main.go, after migrations + before serving traffic
func seedSystemPrompts(ctx context.Context, adminStores db.Stores) error {
  orgs, err := adminStores.Orgs.ListAll(ctx)
  if err != nil { return err }
  for _, org := range orgs {
    for _, p := range shippedSystemPrompts {
      if err := adminStores.Prompts.SeedOrUpdate(ctx, org.ID, p); err != nil {
        return fmt.Errorf("seed %s into org %s: %w", p.ID, org.ID, err)
      }
    }
  }
  return nil
}

Backend differences

  • SQLite: OrgStore.ListAll returns one synthetic {"default"} entry; PromptStore.SeedOrUpdate writes to prompts + system_prompt_versions on the single connection. Behavior matches today.
  • Postgres: OrgStore.ListAll queries orgs on the admin connection (RLS is bypassed for supabase_admin). PromptStore.SeedOrUpdate writes via the admin tf_app-equivalent role that retains system_prompt_versions write privileges. Hash-based skip (SeedOrUpdateSystemPrompt from existing code) prevents re-write on identical content.

Bootstrap path for the first org (D7 ships first-org creation): immediately after creating the org, call seedSystemPrompts(ctx, adminStores) filtered to that one org. Same code path, scoped argument — no second seeder.

Failure modes if mis-wired: seeder run on the tf_app request pool → SQLSTATE 42501 (insufficient privilege) — pinned by TestSystemPromptSeeding_DeployActorOnly. Per-org loop missing → first org has prompts, subsequent orgs have empty prompts table → handler returns no system prompts → loud at any usage site. Both fail fast and obviously.

§11

Lint enforcement

A refactor like this only sticks if drift can't sneak in. Two rules; both go strict by wave 4.

Rule 1: Only impl packages access raw SQL

The old internal/db package-level functions (db.FindOrCreateTask, etc.) are deleted by the end of D2. The replacement check is that no file under internal/server/, internal/delegate/, internal/poller/, internal/curator/, internal/ai/, internal/tracker/, or any other non-impl package imports database/sql directly — only internal/db, internal/db/sqlite, internal/db/postgres, internal/db/dbtest, and internal/db/pgtest may.

# .golangci.yml
linters:
  enable:
    - forbidigo
linters-settings:
  forbidigo:
    forbid:
      - p: '^sql\.Open$'
        msg: 'use db.Open / db.OpenAt instead of raw sql.Open'
      - p: '^sql\.DB$'
        msg: 'depend on db store interfaces, not raw *sql.DB'
issues:
  exclude-rules:
    - path: 'internal/db/.*'
    - path: 'main\.go'
    - path: 'cmd/.*'

Path exclusions are intentionally permissive for the internal/db/ tree; everything inside that tree may use raw SQL freely. Everything outside it must go through interfaces.

Rule 2: Handlers can't take db.Stores

Grep-based CI check (scripts/lint-stores.sh):

#!/usr/bin/env bash
set -euo pipefail
if grep -rnE 'db\.Stores' internal/server/ internal/delegate/ internal/poller/ \
   --include='*.go' | grep -v '_test\.go'; then
  echo "FAIL: depend on specific db.TaskStore / db.RunStore / etc., not the bundle"
  exit 1
fi

The bundle is for main.go startup wiring only.

§12

Wave order

22 stores. Don't migrate them all at once. Six waves, each independently mergeable.

Wave 0 — skeleton

  • Create internal/db/sqlite/ and internal/db/postgres/ packages with empty Store structs.
  • Define db.Stores bundle, db.TxStores, db.TxRunner. All store fields are interfaces with no methods yet (waves 1–4 add methods).
  • sqlite.New(conn) and postgres.New(admin, app) compile and return valid bundles whose Tx wrappers work end-to-end (begin/commit, claims set).
  • Wire bundle in main.go via runmode.Current() dispatch.
  • Mockery config lands. Conformance scaffold (internal/db/dbtest/) lands.

Wave 1 — leaf stores

PromptStore, RuleStore, SecretStore, DashboardStore, ScoreStore, SwipeStore. Narrow surfaces. Both backends implemented. Conformance scaffold proven end-to-end on these.

Wave 2 — read paths

EntityStore, EventStore, ProjectStore, KBStore, RepoStore, FactoryStore. Higher volume but mostly reads.

Wave 3a — Tasks

TaskStore. Touched by ~30 callsites. The PendingTaskRef + EntityIDsWithActiveTasks shapes need careful translation in Postgres (composite indexes + RLS).

Wave 3b — Runs

RunStore alone. ~28 methods. Touches delegate spawner, run_messages, run_artifacts, run_worktrees. Estimated 5–7 day PR. If it grows beyond that, split internally by method group (lifecycle / message-stream / artifact-write).

Wave 3c — Memory + Curator + Reviews

MemoryStore, CuratorStore, ReviewStore. Touches the delegate path (memory) and the agent loop (curator + reviews).

Wave 4 — multi-only auth stores

OrgStore, MembershipStore, UserStore, SessionStore, PendingPRStore, PendingFiringStore. SQLite impls return synthetic local-mode values or ErrNotApplicableInLocal; Postgres impls real. Final wave; forbidigo path exemptions are removed.

Wave dependencies: Wave 0 unblocks 1. Waves 1, 2, 3a, 3b, 3c can run in parallel after Wave 0 (different resources, different callsites; build stays green between merges). Wave 4 must follow Wave 1 because the seeder consumes OrgStore.ListAll + PromptStore.SeedOrUpdate together.
§13

Implementation steps

  1. Wave 0 — skeleton PR Empty packages, empty bundle, WithTx wired with claims plumbing, mockery config, conformance scaffold. main.go picks the right backend via runmode.Current() + Postgres admin/app two-conn setup. Build passes; no functional change.
  2. System prompt seeder reshape Rewrite seedDefaultPrompts in main.go to iterate OrgStore.ListAll → PromptStore.SeedOrUpdate per org. SQLite returns one synthetic org; Postgres queries orgs. Both run on the admin connection bundle. Ships in Wave 1 alongside PromptStore.
  3. Wave 1 — leaf stores Define interfaces (PromptStore, RuleStore, SecretStore, DashboardStore, ScoreStore, SwipeStore). Move SQLite SQL into internal/db/sqlite/<resource>.go. Write Postgres SQL fresh against D3 schema. Wire bundle. Refactor affected handlers + non-server callsites to take the interface. Conformance tests for each. go generate mocks.
  4. Add lint rule + exemption baseline After wave 1 lands, add the forbidigo + scripts/lint-stores.sh checks. Exempt every path that hasn't been converted yet (waves 2/3/4 paths). Each subsequent wave removes its exemption.
  5. Wave 2 — read paths EntityStore, EventStore, ProjectStore, KBStore, RepoStore, FactoryStore. Same recipe.
  6. Wave 3a — TaskStore Move task SQL, write Postgres parallel. ~30 callsites; compile errors will surface all of them. Special attention to ActiveTaskRefsForEntities (batches via entityIDInChunkSize in SQLite — Postgres can drop the chunking or keep it for parity).
  7. Wave 3b — RunStore Largest. Move agent.go + run_worktrees.go SQL. Postgres impl handles run_messages, run_artifacts, takeover/release transitions, run_worktrees lifecycle. Coordinate with delegate spawner refactor (drop direct *sql.DB field, take db.RunStore + db.TxRunner).
  8. Wave 3c — Memory + Curator + Reviews MemoryStore, CuratorStore, ReviewStore. Curator's onEventRecorded callback (currently a package-level var in internal/db/events.go) needs a home — move to EventStore as a method SetOnRecorded(fn) or pass through ctx; pick whichever is cleaner once the wave starts.
  9. Wave 4 — multi-only stores OrgStore, MembershipStore, UserStore, SessionStore, PendingPRStore, PendingFiringStore. SQLite impls return defaults or ErrNotApplicableInLocal. Postgres impls real, RLS-aware. Remove all remaining forbidigo exemptions.
  10. Final cleanup Verify grep -rE '^func [A-Z]' internal/db/*.go returns zero results — all SQL has moved. The parent internal/db/<resource>.go files now hold only interfaces and value types. Verify forbidigo clean with no exemptions for non-impl packages.
§14

Risks

RiskMitigation
Refactor takes longer than estimate; multi-tenant work blocks behind it Six waves means downstream tickets unblock incrementally. Wave 1 unblocks D5 (SecretStore semantics). Wave 4 unblocks D7 (Sessions). D3 is already merged so the schema isn't a blocker.
Behavior drift between old internal/db.X and new SQLite impls SQLite SQL is moved verbatim, not rewritten. Conformance harness runs both backends against the same assertions; if the SQLite behavior changed in transit, the test fails. The existing test suite that already exercises internal/db.X functions stays in place during each wave, just rewired to call through the store.
Postgres impls have correctness bugs not caught by the conformance suite Conformance tests assert behavior on the happy path; the pgtest harness already covers RLS isolation, cross-org FK rejection, OCC, vault auth. Postgres-specific edge cases (SQLSTATE assertions, RLS bypass for admin) live in backend-specific test files alongside the conformance run.
Adding ctx + orgID + userID args breaks every callsite That's the point. Compile errors surface every site at once; IDE rename handles most. Bundle the rename per-wave so the build doesn't stay broken across waves.
Mockery + interface method additions cause stale mocks CI step: go generate ./... && git diff --exit-code internal/db/mocks/. Compile-time check at the top of each impl: var _ db.TaskStore = (*taskStore)(nil) fails the moment the interface gains a method that the impl doesn't have.
Wave 3b (Runs) is genuinely large — ~28 methods, touches delegate spawner If the PR grows beyond 5–7 days of work, split RunStore internally by method group: lifecycle (Create/Complete/Cancel/Release) → message stream (Append/List) → artifacts (Insert/GetPrimary) → worktrees (Create/Update/List). Each subgroup is a half-day PR.
Two Postgres connection pools at startup is operationally heavier Documented in §5 with rationale (deploy-time vs request-time roles per D3 grants). Local mode is single-conn. The split matches Postgres best practice for role-separated apps (Supabase, AWS RDS IAM auth all do this).
Package-level callbacks like onEventRecorded don't translate cleanly to per-impl stores Wave 3c (which owns EventStore) decides the shape — either a setter on the store, a constructor parameter, or a ctx-carried hook. Pick the one that doesn't leak into the interface signature; if all three do, the cleanest is constructor-injected.
Non-server packages (delegate, poller, tracker, etc.) also need to migrate, not just server The lint rule (§11) explicitly covers them. Each wave's refactor includes all callers, not just internal/server/. grep -lE 'database/sql' internal/{delegate,poller,tracker,curator,ai}/*.go goes empty by wave 4.

Code structure

  • PASS Every interface in §3 lives in internal/db/<resource>.go.
  • PASS Every interface has both a SQLite impl (internal/db/sqlite/<resource>.go) AND a Postgres impl (internal/db/postgres/<resource>.go). No stubs, no panics.
  • PASS db.Stores bundle exists and is constructed once in main.go via runmode.Current() dispatch.
  • PASS grep -rE '^func [A-Z]' internal/db/*.go returns zero results outside internal/db/sqlite/ and internal/db/postgres/.

Lint

  • PASS golangci-lint run clean with no path exemptions for any non-impl package.
  • PASS scripts/lint-stores.sh clean — no db.Stores field or param outside main.go.
  • PASS go generate ./... && git diff --exit-code internal/db/mocks/ produces no diff (mocks up-to-date).

Tests

  • PASS All existing tests pass — handler tests use mocks, end-to-end tests use real backends.
  • PASS Conformance suite in internal/db/dbtest/ exists with one RunXxxConformance per interface. SQLite + Postgres test files in their respective impl packages both invoke it.
  • PASS CI runs the full conformance suite against both backends. Postgres uses the pgtest testcontainer harness; skips cleanly when Docker isn't available.

Behavior

  • PASS ./triagefactory in local mode behaves identically to pre-D2: poll a repo, take in a PR, swipe a task, run an agent.
