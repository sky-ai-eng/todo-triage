D-Claims: task claim flow + run actor

Three nullable FKs. tasks.claimed_by_agent_id and tasks.claimed_by_user_id are mutually exclusive (CHECK) — together they form the "currently responsible" axis. runs.actor_agent_id is the immutable audit pointer stamped by the spawner. Status stays orthogonal but narrowed: it tracks genuine lifecycle (queued / snoozed / done / dismissed), with 'claimed' and 'delegated' dropped as denormalizations now that the claim columns carry responsibility. The "queue view" is a derived filter on unclaimed-and-active rows. No column on event_handlers — in v1 the kind discriminator is enough; v2 multi-agent picks its own shape.

§0

Phasing

One PR. Schema lands first, then the three claim handlers (trigger-create / user-delegate / user-takeover / user-claim) all write into the new columns, then the Board's per-member filter exposes the symmetry to users.

A
Schema
Three nullable FK columns: tasks.claimed_by_agent_id, tasks.claimed_by_user_id, runs.actor_agent_id. XOR CHECK on the two task columns. Parity prereq: UNIQUE (id, org_id) on agents.
B
Claim handlers
Router stamps agent claim on trigger match. User-delegate handler stamps agent claim on swipe / drag. User-claim handler stamps user claim on a queued task. Takeover flips agent → user. Spawner stamps run actor at spawn. Five callsites, one column family.
C
Board filter parity
Per-member filter chip — bot or any user — gives the same view shape: "tasks they're currently responsible for + tasks they've worked on." Two clauses, same query path, parameterized by member kind.

§2

Three orthogonal axes

The thing that makes the rest of this spec clean: status, responsibility, and audit live on different columns and never need to encode each other. The "queue view" is a derived filter of the first two; the third is write-once.

1. Lifecycle (existing)
tasks.status
Macro-states: queued, snoozed, done, dismissed. 'claimed' and 'delegated' are dropped in this PR — both were denormalizations of responsibility, which now lives on the claim cols. Does NOT enumerate "user-claimed-but-not-closed" — that's a claim-col concern.
2. Responsibility (new)
tasks.claimed_by_*_id
XOR — at most one
Who currently owns the work. NULL/NULL = unclaimed (in the queue). Set when a bot or human commits. Sticky past close — preserves "who finished this" for audit.
3. Audit (new)
runs.actor_agent_id
Per-run, immutable, "who actually ran this." Stamped by the spawner at run start. Survives all downstream config edits + handler deletion.
The queue view is a derived filter, not a status value. "What's in the queue" = WHERE claimed_by_agent_id IS NULL AND claimed_by_user_id IS NULL AND status NOT IN ('done', 'dismissed', 'snoozed'). Team-scoped via the existing team_id + visibility/RLS. A user claiming a queued task is a claim-column transition, not a status change. This is what makes the "user claims without involving the bot" case clean — no new status enum value needed.
Snooze is queue-only. The v0.7 invariant: status='snoozed' ↔ claimed_by_agent_id IS NULL AND claimed_by_user_id IS NULL. A snoozed task is a deferred unclaimed-queue entry — nothing more. Any claim-axis transition (auto-trigger fire, swipe-claim, swipe-delegate, factory drop, user→bot handoff) atomically wakes the task in the same UPDATE: SET ..., snooze_until = NULL, status = CASE WHEN status = 'snoozed' THEN 'queued' ELSE status END. The inverse — SnoozeTask on a claimed task — refuses with 409 and the audit row rolls back as part of the same tx, so the user sees no state change. Users wanting to defer work on a claimed task must explicitly requeue first. Cuts the snoozed-and-claimed incoherent state out of the model entirely, which closes the gap between drain/re-derive (both correctly skip snoozed) and the immediate auto-trigger fire path (used to silently fire on snoozed tasks).
§3

Schema delta

Three new columns. One CHECK constraint. One parity prerequisite. No status enum changes, no event_handlers changes.

event_handlers — unchanged
  • id, org_id, kind, ...
  • — no claim column; kind already discriminates
tasks — two claim cols + XOR
  • id, org_id, status, ...
  • claimed_by_agent_id UUID NULL · FK agents(id, org_id)
  • claimed_by_user_id UUID NULL · FK users(id)
runs — audit
  • id, org_id, task_id, ...
  • actor_agent_id UUID NULL · FK agents(id, org_id)

XOR CHECK on tasks

ALTER TABLE tasks ADD CONSTRAINT tasks_claim_xor
  CHECK (claimed_by_agent_id IS NULL OR claimed_by_user_id IS NULL);

"At most one is set." Both NULL = unclaimed (in the queue). Exactly one set = claimed by that entity. Both set = forbidden. This is the schema-level guarantee that backstops the application's claim-transition logic.

FK targets

  • claimed_by_agent_id + actor_agent_id: (agent_id, org_id) REFERENCES agents(id, org_id) ON DELETE SET NULL in Postgres (composite, after the parity prereq lands). Plain REFERENCES agents(id) ON DELETE SET NULL in SQLite (ALTER TABLE ADD COLUMN can't carry a composite FK; at N=1 there's one org so cross-org leakage is structurally impossible).
  • claimed_by_user_id: REFERENCES users(id) ON DELETE SET NULL. Users aren't org-scoped via composite-PK (the users table doesn't have (id, org_id) uniqueness — membership lives in org_memberships); RLS gates cross-org access at runtime.

Parity prereq on agents

SKY-260 built agents with (id) PRIMARY KEY + UNIQUE(org_id) (rationale at migrations-postgres/202605120003_agents.sql:16-24: "we never compose (agent_id, org_id) at the caller"). D-Claims is when that rationale stops holding.

ALTER TABLE agents ADD CONSTRAINT agents_id_org_unique UNIQUE (id, org_id);
-- SQLite parity:
CREATE UNIQUE INDEX IF NOT EXISTS agents_id_org_unique ON agents (id, org_id);
No status enum changes. The team-scoped queue framing collapses "queued" from "user-specific lifecycle state" to "derived filter on unclaimed-and-active." Adding a status like 'in_progress' or 'user_claimed' would be redundant — the claim columns already encode that, and the queue view filters them out automatically. See arch doc decision task-state-axes.
§4

State transitions

Every transition is small and explicit. The XOR CHECK is the safety net. Status never moves on claim or delegate — those gestures change responsibility (claim cols) only. Status only moves on genuine lifecycle changes: snooze, done, dismiss.

Transition claimed_by_agent_id claimed_by_user_id tasks.status run.actor_agent_id
Rule match (lands in queue) NULL NULL queued — (no run)
Trigger match (auto-delegate) org agent NULL queued (unchanged) org agent (stamped at spawn)
User claims queued task themselves NULL user queued (unchanged) — (no run yet)
User drags queued task → bot org agent NULL queued (unchanged) org agent (stamped at spawn)
User takes over a bot-running task NULL (cleared) user (set) queued (unchanged) org agent (unchanged — immutable)
User snoozes / completes / dismisses unchanged (sticky) unchanged (sticky) → snoozed / done / dismissed unchanged
Agent deleted (cascade) NULL (SET NULL) unchanged unchanged NULL on existing runs (SET NULL)
Sticky claims past close. When a task closes (status='done' or 'dismissed'), the claim columns are not cleared. They become the audit answer to "who was responsible when this finished?" Lifecycle's status + closed_at + close_reason tell the lifecycle story; claimed_by_* tell the responsibility story; runs.actor_agent_id tells the execution story. Three answers, three axes.
Takeover clears agent claim, sets user claim — same transaction. The XOR CHECK requires this be atomic. The existing takeover handler (which today flips run.status to taken_over + writes takeover_path) gains two extra UPDATE clauses on the task row in the same SQL statement.
§5

Migration design

Forward-only, one migration per backend. Parity unique on agents, three nullable columns, one XOR CHECK, two partial indexes. All new columns default to NULL on existing rows — the CHECK is trivially satisfied.

Postgres

-- (1) Agents parity: composite UNIQUE so the agent-id FKs can be composite.
ALTER TABLE agents ADD CONSTRAINT agents_id_org_unique UNIQUE (id, org_id);

-- (2) tasks.claimed_by_agent_id
ALTER TABLE tasks
  ADD COLUMN claimed_by_agent_id UUID,
  ADD CONSTRAINT tasks_claimed_agent_fkey
    FOREIGN KEY (claimed_by_agent_id, org_id)
    REFERENCES agents (id, org_id) ON DELETE SET NULL;

-- (3) tasks.claimed_by_user_id
ALTER TABLE tasks
  ADD COLUMN claimed_by_user_id UUID REFERENCES users(id) ON DELETE SET NULL;

-- (4) XOR CHECK
ALTER TABLE tasks ADD CONSTRAINT tasks_claim_xor
  CHECK (claimed_by_agent_id IS NULL OR claimed_by_user_id IS NULL);

-- (5) Partial indexes for the Board filter's hot reads
CREATE INDEX tasks_claimed_agent_idx ON tasks(claimed_by_agent_id)
  WHERE claimed_by_agent_id IS NOT NULL;
CREATE INDEX tasks_claimed_user_idx ON tasks(claimed_by_user_id)
  WHERE claimed_by_user_id IS NOT NULL;

-- (6) runs.actor_agent_id (audit)
ALTER TABLE runs
  ADD COLUMN actor_agent_id UUID,
  ADD CONSTRAINT runs_actor_agent_fkey
    FOREIGN KEY (actor_agent_id, org_id)
    REFERENCES agents (id, org_id) ON DELETE SET NULL;
CREATE INDEX runs_actor_agent_idx ON runs(actor_agent_id)
  WHERE actor_agent_id IS NOT NULL;

No RLS changes — the existing tasks + runs policies already gate on org_id = tf.current_org_id(); the new columns inherit row-level RLS.

SQLite

The three new columns go on via plain ADD COLUMN with inline column-level REFERENCES. To enforce the XOR CHECK on tasks, SQLite requires rebuilding the tasks table, as described below.

-- (1) Agents parity index.
CREATE UNIQUE INDEX IF NOT EXISTS agents_id_org_unique ON agents (id, org_id);

-- (2)-(4) Three ADD COLUMN statements with inline plain FKs.
ALTER TABLE tasks ADD COLUMN claimed_by_agent_id TEXT
  REFERENCES agents (id) ON DELETE SET NULL;
ALTER TABLE tasks ADD COLUMN claimed_by_user_id TEXT
  REFERENCES users (id) ON DELETE SET NULL;
ALTER TABLE runs ADD COLUMN actor_agent_id TEXT
  REFERENCES agents (id) ON DELETE SET NULL;

-- (5) XOR CHECK. SQLite supports ALTER TABLE ADD CHECK in 3.37+ via the
-- standard rebuild-with-new-constraint dance (the 12-step or via ALTER
-- TABLE ... ADD CHECK if available); pick whichever the codebase's
-- existing migrations use for tasks.
PRAGMA foreign_keys=OFF;
-- rebuild tasks with the added CHECK constraint
CREATE TABLE tasks_new (... /* original cols + new claim cols + new CHECK */ ...);
INSERT INTO tasks_new SELECT ... FROM tasks;
DROP TABLE tasks;
ALTER TABLE tasks_new RENAME TO tasks;
PRAGMA foreign_keys=ON;

-- (6) Partial indexes — same as Postgres.
CREATE INDEX IF NOT EXISTS tasks_claimed_agent_idx ON tasks(claimed_by_agent_id)
  WHERE claimed_by_agent_id IS NOT NULL;
CREATE INDEX IF NOT EXISTS tasks_claimed_user_idx ON tasks(claimed_by_user_id)
  WHERE claimed_by_user_id IS NOT NULL;
CREATE INDEX IF NOT EXISTS runs_actor_agent_idx ON runs(actor_agent_id)
  WHERE actor_agent_id IS NOT NULL;
SQLite tasks rebuild for the CHECK. SQLite can't add a table-level CHECK constraint without rebuilding the table. The standard 12-step dance applies, with -- +goose NO TRANSACTION for the PRAGMA foreign_keys flip. Runs doesn't need a rebuild (no CHECK).
§6

Router · spawner · claim handlers

Five write sites for the new columns, all going through the same column family. Each is a small addition to existing flows.

1. Router — trigger match stamps agent claim at task creation

// internal/routing/router.go — in task construction.
var claimedAgentID string
if handler.Kind == domain.EventHandlerKindTrigger {
    a, err := r.stores.Agents.GetForOrg(ctx, orgID)
    if err == nil && a != nil {
        claimedAgentID = a.ID
    }
}
task := &domain.Task{
    OrgID:            orgID,
    EntityID:         entityID,
    EventType:        evt.EventType,
    DedupKey:         derivedDedupKey,
    PrimaryEventID:   evt.ID,
    ClaimedByAgentID: claimedAgentID,  // "" if rule, agent.ID if trigger
}

2. Spawner — stamps run.actor_agent_id at run start

// internal/delegate/spawner.go — inside Delegate, where runs row is INSERTed.
actorAgentID := task.ClaimedByAgentID
if actorAgentID == "" {
    // User-claimed task (user-claim ran via "run this prompt now" UI) or
    // a rule-task with no claim that's being one-off delegated — fall
    // back to org's agent.
    if a, err := s.stores.Agents.GetForOrg(ctx, task.OrgID); err == nil && a != nil {
        actorAgentID = a.ID
    }
}
// INSERT runs(..., actor_agent_id) VALUES (..., $N)

3. User-delegate handler — drag-to-bot / "Delegate"

The HTTP handler that handles "delegate this task to the bot" (swipe-right / drag-to-Agent-column / factory drop on runs-tray) stamps the agent claim BEFORE attempting the spawn. Status is not touched — the task remains 'queued' on the lifecycle axis; only the claim col flips. A failed spawn doesn't unstamp the claim: commitment is real even when execution didn't fire, and the Board renders a "delegate failed, retry" affordance on the bot-claimed card.

UPDATE tasks
   SET claimed_by_agent_id = $1,        -- org's agent.id
       claimed_by_user_id  = NULL     -- XOR safety; usually already NULL
 WHERE id = $2 AND org_id = $3
   AND status NOT IN ('done', 'dismissed', 'snoozed');

4. User-claim handler — "I'll take this myself"

A new endpoint (or extension of an existing claim/take action — TBD with the UI). Pure claim-col write, no status change:

UPDATE tasks
   SET claimed_by_user_id  = $1,        -- current user.id
       claimed_by_agent_id = NULL
 WHERE id = $2 AND org_id = $3
   AND claimed_by_agent_id IS NULL   -- safety: don't steal from the bot
   AND claimed_by_user_id  IS NULL;  -- safety: don't steal from another user

5. Takeover handler — user grabs an active bot run

The existing takeover flow (which moves the worktree, flips runs.status = 'taken_over') extends with the task-claim flip:

UPDATE tasks
   SET claimed_by_agent_id = NULL,    -- bot loses claim
       claimed_by_user_id  = $1         -- takeover user.id
 WHERE id = $2 AND org_id = $3
   AND claimed_by_agent_id IS NOT NULL; -- only if bot currently claims
-- runs.actor_agent_id stays — that run was run by the bot. Audit truth.
Credentials still come from the keychain in local v1. The spawner stamps actor_agent_id for audit but doesn't yet resolve GitHub credentials through the agent row. Threading agent.github_app_installation_id / agent.github_pat_user_id into credential resolution is the natural follow-up — out of scope here.
§7

API + frontend

The headline UX deliverable is the per-member Board filter — same view shape for the bot and any human teammate.

Board's per-member filter

Filter chips at the top of the Board, one per team member (including the bot). Clicking a chip parameterizes the same SQL shape:

-- "Tasks for member X" — same query, two parameter sets.
SELECT ... FROM tasks t
WHERE t.org_id = $org_id AND t.team_id = $team_id
  AND (
    -- Currently responsible (active only)
    (t.status NOT IN ('done', 'dismissed', 'snoozed') AND (
      ($kind = 'user'  AND t.claimed_by_user_id  = $member_id) OR
      ($kind = 'agent' AND t.claimed_by_agent_id = $member_id)
    ))
    OR
    -- Has ever acted on (all time, including closed)
    ($kind = 'user'  AND (
       t.creator_user_id    = $member_id OR
       t.claimed_by_user_id  = $member_id))
    OR
    ($kind = 'agent' AND (
       t.claimed_by_agent_id = $member_id OR
       EXISTS (SELECT 1 FROM runs r
               WHERE r.task_id = t.id AND r.actor_agent_id = $member_id)))
  );

The two clauses ("currently responsible" + "has ever acted on") compose into "everything this member has touched." User and bot are symmetric in shape — the parameters change, the structure doesn't.

Task + run DTOs

  • Task DTO gains claimed_by_agent_id?: string | null, claimed_by_user_id?: string | null, and JOIN-resolved display fields (claimed_by_agent_name, claimed_by_user_name) for the Board's Agent column.
  • Run DTO gains actor_agent_id?: string | null + actor_agent_name?: string. AgentCard.tsx renders "Ran as: {name}" when populated.

What does NOT change in the UI

  • No new "claim this trigger" editor checkbox. Trigger configuration is unchanged — the auto-claim is implicit in kind='trigger'.
  • No new agent-picker. v1 has one agent per org; the Board's bot chip reads the org's only agent.
  • No task-status enum surface change. Status badges read the existing values.

Out of scope

  • event_handlers.claimed_by_agent_id — deliberately deferred to v2. In v1, kind='trigger' + UNIQUE(org_id) on agents encodes the entire claim configuration. Adding a per-handler claim column now bets on one of three plausible v2 shapes (per-handler explicit / per-team derived via team_agents / per-prompt) before we have multi-agent requirements.
  • Spawner credential routing through agents. Today the spawner reads the keychain; after this PR it still does. Threading agent.github_app_installation_id / agent.github_pat_user_id into credential resolution is the natural follow-up but a behavior change worth its own ticket.
  • Multi-agent-per-org. v1 has UNIQUE(org_id) on agents — one bot per org.
  • Per-member Board filter UI + JOIN-resolved claim display names. The schema + handlers in this PR write claim columns + actor pointers; the chip UX and the agent/user-name lookup that lights up the Board's Agent column are a follow-up FE PR. Scope cut deliberate — this PR is the load-bearing schema + state-machine work, the UX layer comes next.
  • Cross-team filter / multi-team Board. SKY-262 (D-TeamDefault) shipped the team-scoped queue: every task now has team_id NOT NULL and the Board's queue view is a team-scoped derived filter. Rendering work from teams the caller doesn't belong to (cross-team admin overviews) is a separate UX call.
  • Denormalized agent_display_name on runs — if an agent is deleted, the display name disappears too. Add later if audit retention becomes a hard requirement.
  • Filter chips + sorting controls themselves. The Board query in §7 is the load-bearing part; chip UX (which chips show, ordering, persistence) is a cosmetic follow-up.

§10

Risks

RiskMitigation
Concurrent claim attempts — two users (or user + auto-trigger) try to claim the same queued task at the same moment. Each claim handler's UPDATE includes guard clauses (WHERE claim cols currently NULL) and a check of the number of rows affected. Lost races return a 409 with "task already claimed." The XOR CHECK is the schema-level backstop against simultaneous double-write.
v2 multi-agent picks a shape that doesn't fit "task claim is just the org's agent" (e.g., per-team via team_agents, or per-prompt). The resolution lives in one function in the router. v2 rewrites that function — anywhere from per-team derivation to per-handler explicit — and the downstream task column absorbs the result unchanged. The column shape is robust to all plausible v2 designs because it stores the resolved answer, not the resolution mechanism.
SQLite's ADD COLUMN ... REFERENCES creates plain (single-column) FKs on the three new columns; cross-org integrity rests on application code only in SQLite. At N=1 in local mode there's exactly one org_id value in the entire database; cross-org leakage is structurally impossible. The composite-FK promotion is part of the broader SQLite child-FK parity sweep deferred from SKY-259 — bounded, optional, not load-bearing.
The SQLite tasks rebuild for the XOR CHECK is the largest table-mod in this migration; risk of data loss on an interrupted upgrade. Goose runs each migration in its own transaction by default; the explicit NO TRANSACTION directive only wraps the PRAGMA foreign_keys flip, and the table-rebuild itself is inside BEGIN/COMMIT manually. Standard SQLite-rebuild pattern, used elsewhere in the codebase (e.g., SKY-259's event_handlers migration). Bootstrap-shape test confirms post-migration row count matches pre-migration.
If Agents.GetForOrg returns nil (bot not bootstrapped), trigger matches still create tasks but with claimed_by_agent_id = NULL — looks like a rule match to downstream consumers. Acceptable. Bootstrap creates the local-mode agent at first launch; the nil case is a transient seam between DB init and agent bootstrap. The Board's Agent column shows "—" in that window; once the agent row exists, subsequent triggers stamp correctly. Multi-mode installs that complete the founder flow always have an agent row before any task fires.
Sticky claims past close mean an old closed task that referenced a deleted user retains a NULL claimed_by_user_id (SET NULL behavior) and the Board renders "—" in its "claimed by" column. By design. The display name is JOIN-resolved; if the user row is gone, there's no name to show. If audit retention past user deletion ever becomes a hard requirement, denormalize the display name onto the task at claim time — out of scope here.
Takeover's two-table UPDATE (runs + tasks) needs to be atomic — partial failure leaves the bot's claim cleared but the takeover not recorded on the run, or vice versa. Both UPDATEs go in the same Go transaction. The existing takeover handler already uses a transaction for the runs-side work (worktree move + run.status flip); the tasks-side UPDATE joins that transaction.
§11

Acceptance

  • PASS Fresh install + upgrade: tasks + runs gain the new columns; both default to NULL on all existing rows. XOR CHECK passes trivially.
  • PASS Trigger match: task has claimed_by_agent_id = org's agent, claimed_by_user_id = NULL; resulting run has actor_agent_id = org's agent.
  • PASS Rule match: task has both claim cols NULL; status = queued; appears in the queue view.
  • PASS User claims a queued task: claimed_by_user_id = user, claimed_by_agent_id = NULL, status unchanged. Task drops out of the queue view. No run spawned.
  • PASS User drags queued task → bot: claimed_by_agent_id = org's agent, status unchanged (stays 'queued'), run spawns with actor_agent_id = org's agent.
  • PASS User takes over an active bot run: claimed_by_agent_id → NULL, claimed_by_user_id → user (atomic); runs.actor_agent_id unchanged.
  • PASS Task close (any path): claim columns sticky — preserve the responsibility audit alongside closed_at + close_reason.
  • PASS Per-member Board filter: same query shape returns sensible result sets for both user and bot chips in a synthetic scenario (mixed rule, trigger, user-claim, takeover history).
  • PASS XOR CHECK rejection: direct SQL attempt to set both claim cols on the same row fails with the CHECK error.
  • PASS Concurrent claim race: two simultaneous claim attempts on the same task; one succeeds, the other returns 409. No corrupt state, no XOR violation.
  • TEST Conformance: round-trip of claim cols on Task; round-trip of actor on AgentRun.
  • TEST Router + spawner + handler tests pin every transition in §4's state-transition matrix.
  • TEST Migration shape test: agents UNIQUE(id, org_id), three new columns, XOR CHECK present, partial indexes present.
