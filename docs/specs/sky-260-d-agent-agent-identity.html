D-Agent — Agent identity primitive

The bot is a first-class member of every team, distinct from human users. This ticket adds the agents + team_agents tables, the per-resource stores that wrap them, the local-install bootstrap, and the multi-mode org-create bootstrap function. The team/agent reframe (locked decision agent-identity) is finally load-bearing schema — every downstream multi-mode ticket from D-Claims onward references one of these tables.

§1

Why now

D-Agent unblocks four downstream tickets that all need the agents table to exist. It's also a prerequisite for SKY-259 (the event_handlers unification) — the unified table will stamp claimed_by_agent_id on trigger-kind rows, and that column can't be added until the agents table is referenceable.

DownstreamWhat it needs from D-Agent
SKY-261 D-Claimstasks.claimed_by_agent_id UUID REFERENCES agents(id) + runs.actor_agent_id UUID NOT NULL REFERENCES agents(id). The router stamps claimed_by_agent_id = team_agents.agent_id for the firing team — needs both tables present.
SKY-262 D-TeamDefaulttf.user_in_team(team_id) RLS helper. This ticket adds it (needed for team_agents policies); D-TeamDefault then reuses it across the broader policy sweep.
SKY-263 D11 GH Appagents.github_app_installation_id is where the App install id lands; that column is added now (nullable, no writes yet).
SKY-264 D-PredicateNo direct schema dependency, but the "one-person team boolean toggle" UI assumes the team has an agent membership row to consult — D-Agent guarantees there is one.
SKY-259 event_handlersThe unified table's auto_claim_by_agent_id column FKs agents(id). Without D-Agent, the unification migration would have to invent a placeholder agent row or skip the FK.

Why before D-Claims, not bundled with it

The two tickets touch different surfaces — D-Agent adds two tables + their stores; D-Claims modifies tasks + runs + the router. Bundling them would mean one PR with ~6 migrations, the entire claim state machine rewrite, AND new tests for both. Splitting keeps each PR reviewable. D-Agent is purely additive (zero existing code paths change behavior); D-Claims is the riskier rewrite.

Why before SKY-259

The conversation earlier in the session settled on SKY-260 → SKY-259 → SKY-261: do the additive agent work first, then unify task_rules + prompt_triggers, then add claim semantics to the unified table once instead of twice. D-Agent doesn't depend on SKY-259; SKY-259 depends on D-Agent.

§2

Current state

The bot identity today is implicit: "whoever's PAT is in the keychain." There is no row anywhere that represents "the agent" as an entity with its own configuration. Every downstream feature that needs to know about the agent (claim semantics, per-team enable, model selection, audit) currently has to reach for either the lone user (local) or the org's GitHub App via a non-existent integration (multi).

What's already in the schema (don't re-add)

The SKY-247 Postgres baseline already ships orgs, teams, users, org_memberships, memberships, the org_role + membership_role enums, and the helpers tf.user_has_org_access / tf.user_is_org_admin / tf.user_owns_org. SQLite has none of these (synthetic single-tenant). Both backends have the resource tables (prompts, projects, task_rules, prompt_triggers, tasks, runs, ...) that D-Claims will later modify.

What's missing and exactly where it lands: two tables (agents, team_agents) + one helper (tf.user_in_team) + two RLS policies (one each per table) on the Postgres side; two parallel tables (no RLS) on the SQLite side. That's the whole schema delta for this ticket.
§3

Target state

Two new tables, parallel between backends. Postgres carries the full schema with FKs into orgs/teams/users; SQLite carries the same shape minus tenancy-table FKs (those tables don't exist locally).

Postgres migration 202605120003_agents.sql (working name; rev per merge day)

-- +goose Up
-- ===== agents (workload identity) =====
-- One row per org. NOT a row in users — agents have no auth.users entry,
-- no session, no JWT. Writes go through the admin pool (BYPASSRLS) or via
-- workload code paths; the agent never appears as the JWT subject on the
-- RLS-active app pool.

CREATE TABLE agents (
  id                              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id                          UUID NOT NULL REFERENCES orgs(id) ON DELETE CASCADE,
  display_name                    TEXT NOT NULL DEFAULT 'Triage Factory Bot',
  default_model                   TEXT,                          -- e.g. 'claude-opus-4-7'
  default_autonomy_suitability    REAL,                          -- minimum score for auto-claim via triggers
  github_app_installation_id      TEXT,                          -- D11 populates; nullable until then
  github_pat_user_id              UUID REFERENCES users(id) ON DELETE SET NULL,
  jira_service_account_id         TEXT,                          -- v2; nullable
  created_at                      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at                      TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE (org_id)                                                  -- one bot per org
);
CREATE INDEX agents_org_idx ON agents(org_id);

-- Team-level membership for the agent. Default-enabled at team creation;
-- per-team overrides (model, autonomy) let different teams in the same
-- org run the bot with different settings.

CREATE TABLE team_agents (
  team_id                          UUID NOT NULL REFERENCES teams(id) ON DELETE CASCADE,
  agent_id                         UUID NOT NULL REFERENCES agents(id) ON DELETE CASCADE,
  enabled                          BOOLEAN NOT NULL DEFAULT TRUE,
  per_team_model                   TEXT,                          -- NULL → agent.default_model
  per_team_autonomy_suitability    REAL,                          -- NULL → agent.default_autonomy_suitability
  added_at                         TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (team_id, agent_id)
);
CREATE INDEX team_agents_team_idx  ON team_agents(team_id);
CREATE INDEX team_agents_agent_idx ON team_agents(agent_id);

-- ===== tf.user_in_team helper =====
-- Used here for team_agents RLS; reused by D-TeamDefault across the
-- broader per-resource RLS sweep. SECURITY DEFINER so the lookup
-- bypasses memberships' own RLS (no recursion).

-- +goose StatementBegin
CREATE OR REPLACE FUNCTION tf.user_in_team(target_team UUID) RETURNS BOOLEAN
LANGUAGE sql STABLE SECURITY DEFINER
SET search_path = pg_catalog, public
AS $$
  SELECT EXISTS (
    SELECT 1 FROM memberships
    WHERE user_id = tf.current_user_id() AND team_id = target_team
  );
$$;
-- +goose StatementEnd
REVOKE ALL ON FUNCTION tf.user_in_team FROM PUBLIC;
GRANT EXECUTE ON FUNCTION tf.user_in_team TO tf_app;

-- ===== RLS policies =====

ALTER TABLE agents      ENABLE ROW LEVEL SECURITY;
ALTER TABLE team_agents ENABLE ROW LEVEL SECURITY;

-- agents: org-wide read (any member can see "who is our bot"),
-- admin-only writes (rename, model, credential changes are sensitive).
-- Writes split into separate INSERT/UPDATE/DELETE policies rather than
-- one FOR ALL so future tickets can re-gate any single op (e.g. tighten
-- DELETE) without touching the others — matches the prompt_triggers
-- split from 202605120001.
--
-- INSERT + UPDATE WITH CHECK additionally enforce that
-- github_pat_user_id (when populated) references a user with an
-- org_memberships row in this agent's org (migration 202605120005).
-- Without that constraint, an admin in org A could write
-- agents.github_pat_user_id = bob-from-org-B; Vault wrappers gate the
-- actual PAT lookup by tf.current_org_id() so cross-org credential
-- theft isn't reachable, but the row's integrity is wrong and we
-- refuse the write directly as defense in depth.
CREATE POLICY agents_select ON agents
  FOR SELECT TO tf_app USING (org_id = tf.current_org_id() AND tf.user_has_org_access(org_id));

CREATE POLICY agents_insert ON agents FOR INSERT
  WITH CHECK (
    org_id = tf.current_org_id()
    AND tf.user_is_org_admin(org_id)
    AND (github_pat_user_id IS NULL
         OR EXISTS (SELECT 1 FROM org_memberships
                      WHERE user_id = github_pat_user_id AND org_id = agents.org_id))
  );

CREATE POLICY agents_update ON agents FOR UPDATE
  USING      (org_id = tf.current_org_id() AND tf.user_is_org_admin(org_id))
  WITH CHECK (
    org_id = tf.current_org_id()
    AND tf.user_is_org_admin(org_id)
    AND (github_pat_user_id IS NULL
         OR EXISTS (SELECT 1 FROM org_memberships
                      WHERE user_id = github_pat_user_id AND org_id = agents.org_id))
  );

CREATE POLICY agents_delete ON agents FOR DELETE
  USING (org_id = tf.current_org_id() AND tf.user_is_org_admin(org_id));

-- team_agents: team-member read + write (toggling the bot for your own
-- team is a team-member power, not an admin power; rename/credential
-- changes happen at the agents level above and are admin-gated there).
-- Every policy pins to (1) the active request org via
-- tf.team_in_current_org(team_id) and (2) team membership via
-- tf.user_in_team(team_id). Without the org pin, a user with
-- memberships in multiple orgs could operate on rows in org B while
-- their JWT claims org_id = A — same shape as the
-- teams_select / memberships_select t.org_id = tf.current_org_id()
-- pins in the baseline. INSERT + UPDATE WITH CHECK additionally
-- require the referenced agent's org_id = tf.current_org_id() so a
-- team member can't write a row pointing at another org's agent
-- (migrations 202605120004 + 202605120005).
CREATE POLICY team_agents_select ON team_agents
  FOR SELECT TO tf_app
  USING (tf.team_in_current_org(team_id) AND tf.user_in_team(team_id));

CREATE POLICY team_agents_insert ON team_agents FOR INSERT
  WITH CHECK (
    tf.team_in_current_org(team_id) AND tf.user_in_team(team_id)
    AND EXISTS (SELECT 1 FROM agents a
                 WHERE a.id = agent_id AND a.org_id = tf.current_org_id())
  );

CREATE POLICY team_agents_update ON team_agents FOR UPDATE
  USING      (tf.team_in_current_org(team_id) AND tf.user_in_team(team_id))
  WITH CHECK (
    tf.team_in_current_org(team_id) AND tf.user_in_team(team_id)
    AND EXISTS (SELECT 1 FROM agents a
                 WHERE a.id = agent_id AND a.org_id = tf.current_org_id())
  );

CREATE POLICY team_agents_delete ON team_agents FOR DELETE
  USING (tf.team_in_current_org(team_id) AND tf.user_in_team(team_id));

Why SQLite carries the credential columns even though they're always NULL locally: store interface symmetry. Both backends return the same domain.Agent struct; consumers (D11's spawner credential picker, D-Claims' router) read the same fields. The local path falls through the credential-source-is-NULL branch to "fetch PAT from keychain"; multi mode picks between App and PAT-borrow. One branch, two table shapes.

SQLite domain row → Postgres domain row

SQLite stores id as TEXT; Postgres as UUID. The store interface accepts string everywhere — same convention as TaskRuleStore / TriggerStore from SKY-246. Postgres impls validate via isValidUUID (helper from internal/db/postgres/uuid.go) and no-op Get/Update/Delete on invalid IDs to match SQLite's TEXT-keyed behavior.

Stores

Same per-resource pattern as the SKY-246 wave 1 stores. Each interface ships with SQLite + Postgres impls under internal/db/{sqlite,postgres}/, plus a shared conformance suite under internal/db/dbtest/.

AgentStore (internal/db/agents_store.go, new)

package db

import (
  "context"
  "github.com/sky-ai-eng/triage-factory/internal/domain"
)

// AgentStore is the workload-identity store. One row per org. Writes are
// admin-gated in Postgres via the agents_insert/agents_update/agents_delete
// policies; SQLite's assertLocalOrg pins identity to the synthetic
// single-org world.
type AgentStore interface {
  // GetForOrg returns the single agent row for the org, or nil if not
  // yet bootstrapped. Stores call this on every credential lookup.
  GetForOrg(ctx context.Context, orgID string) (*domain.Agent, error)

  // Create inserts the org's agent row. Idempotent on (org_id) — duplicate
  // calls return the existing row's id and no error (so org-create
  // bootstrap is safe to re-run on partial-success recovery).
  Create(ctx context.Context, orgID string, a domain.Agent) (string, error)

  // Update changes display_name, default_model, default_autonomy_suitability,
  // credential FKs. Admin-only in Postgres (RLS enforces). No-ops on invalid UUID.
  Update(ctx context.Context, orgID string, a domain.Agent) error

  // SetGitHubAppInstallation is the D11 entry point. Kept narrow so the
  // App install flow doesn't accidentally touch unrelated fields.
  SetGitHubAppInstallation(ctx context.Context, orgID, agentID, installationID string) error

  // SetGitHubPATUser is the fallback / local entry point.
  SetGitHubPATUser(ctx context.Context, orgID, agentID, userID string) error
}

TeamAgentStore (internal/db/team_agents_store.go, new)

type TeamAgentStore interface {
  // GetForTeam returns the team_agents row, or nil if not present.
  // Routing calls this to decide trigger fire vs. unclaimed fallback.
  GetForTeam(ctx context.Context, orgID, teamID string) (*domain.TeamAgent, error)

  // AddForTeam inserts a default-enabled row. Called from the team-create
  // bootstrap function (BootstrapTeamAgent). Idempotent on (team_id, agent_id).
  AddForTeam(ctx context.Context, orgID, teamID, agentID string) error

  // SetEnabled toggles the bot on or off for a single team. Team-member-
  // writable (not admin-only) per the locked decision in the arch doc.
  SetEnabled(ctx context.Context, orgID, teamID, agentID string, enabled bool) error

  // SetOverrides writes per-team model + autonomy overrides; NULL strings
  // / nil pointers clear the override and fall back to agent defaults.
  SetOverrides(ctx context.Context, orgID, teamID, agentID string, model *string, autonomy *float64) error

  // Remove deletes the membership entirely. Rare — only used if an
  // admin wants to fully unhook the bot from a team rather than disable.
  Remove(ctx context.Context, orgID, teamID, agentID string) error

  // ListForOrg returns every (team, enabled) pair for the org's agent —
  // used by D-Claims' router to decide which firing-team triggers auto-
  // claim and the admin UI's "per-team config" table.
  ListForOrg(ctx context.Context, orgID string) ([]domain.TeamAgent, error)
}

§5

Bootstrap

Three entry points: local install, multi-mode org create, multi-mode team create. The first runs in this PR; the other two ship as functions ready for callers — actual signup / org-create handlers are D5/D6/D7 work. Calling the functions is left to those tickets.

Local install bootstrap

Runs from main.go after db.Migrate + alongside seedDefaultPrompts. New helper in seed.go (or a sibling file — recommend bootstrap.go since this isn't a "shipped row" seed, it's an install-time row insert).

// internal/db/bootstrap.go (package db, not the per-backend impls)
// Idempotent — re-runs across boots leave existing rows intact.
// Both backends use db.BootstrapAgentID(orgID) as the row id (a
// deterministic UUIDv5 per orgID) — Agent.ID on the input is ignored
// by both impls so cross-backend re-runs land on the same row.
func BootstrapLocalAgent(ctx context.Context, stores Stores) error {
  agentID, err := stores.Agents.Create(ctx, runmode.LocalDefaultOrg, domain.Agent{
    DisplayName: "Triage Factory Bot",
  })
  if err != nil { return err }

  // One team in local mode (synthetic LocalDefaultTeamID = "default").
  return stores.TeamAgents.AddForTeam(ctx, runmode.LocalDefaultOrg, LocalDefaultTeamID, agentID)
}

Called from main.go's startup block, right after seedDefaultPrompts(...):

seedDefaultPrompts(database, stores.Prompts, stores.Triggers)
if err := BootstrapLocalAgent(ctx, stores); err != nil {
  log.Printf("[bootstrap] warning: local agent: %v", err)
}

Multi-mode org-create bootstrap (function only — handler is D5/D7)

// internal/db/bootstrap.go (new)
// Called from the future org-create handler (D7) AFTER the orgs row is
// inserted and BEFORE any team is created for the org. Uses the admin
// pool internally because no JWT claims exist yet (org_memberships row
// for the founder is also inserted in the same handler transaction).
func BootstrapAgentForOrg(ctx context.Context, stores Stores, orgID string) (string, error) {
  return stores.Agents.Create(ctx, orgID, domain.Agent{
    DisplayName: "Triage Factory Bot",
  })
}

// Called from the future team-create handler (D7) AFTER the teams row
// is inserted. Looks up the org's agent and inserts the team_agents row.
func BootstrapTeamAgent(ctx context.Context, stores Stores, orgID, teamID string) error {
  agent, err := stores.Agents.GetForOrg(ctx, orgID)
  if err != nil { return err }
  if agent == nil {
    return fmt.Errorf("bootstrap: org %s has no agent — call BootstrapAgentForOrg first", orgID)
  }
  return stores.TeamAgents.AddForTeam(ctx, orgID, teamID, agent.ID)
}
Why these are functions, not handlers: the org-create + team-create HTTP handlers don't exist yet (D5/D7). Building them inside this PR would bloat the diff and couple to auth work that hasn't landed. Defining the bootstrap functions now means the handler tickets just call them — clean handoff.
Why package-level functions instead of methods on Stores: earlier drafts of this spec attached bootstrap as (s *Stores) Bootstrap... methods. The implementation moved them to package-level functions taking db.Stores by value because (a) handlers depend on the specific store interfaces they need rather than the whole bundle, and (b) the bootstrap functions don't need to mutate Stores — they just call into the store interfaces it carries. Both shapes work; the package-level form reads better at the call site (db.BootstrapLocalAgent(ctx, stores) vs stores.BootstrapLocalAgent(ctx)) and keeps Stores a pure DI bundle.

Admin-pool routing for bootstrap

Bootstrap functions in multi mode must use the admin pool (BYPASSRLS), not the app pool. Reasoning: the founder's JWT exists in the org-create flow, but their org_memberships row is being inserted in the same transaction — at the moment we insert the agents row, tf.user_is_org_admin(org_id) returns false (no membership row yet), so the agents_insert RLS policy would refuse. Same pattern as PromptStore.Seed and TriggerStore.Seed land via the admin pool.

The Postgres AgentStore.Create routes unconditionally through the admin pool — there is no separate CreateSystem method. Justification: the only legitimate caller in v1 is bootstrap, and a public Create that routes through the app pool would expose an admin-only surface (agents_insert is admin-gated). If a future ticket needs user-initiated agent creation (e.g. admin UI creating extra agents per team), it adds a separate method then. Earlier drafts of this spec proposed CreateSystem as the explicit "system context" variant; the implementation simplified to one Create with admin-pool routing because the second method had no caller.

Existing installs (v1.10.1 → current)

Forward-only, idempotent, invisible. There is no backfill migration and no manual step. Three pieces line up to make this work:

  • Migrations only create empty tables. The new SQLite + Postgres migrations land the schema; they don't insert rows. Pre-runner installs are still refused by SKY-245's ErrPreRunnerInstall gate pointing at v1.10.1 as the intermediate hop; legacy-runner-populated installs roll forward cleanly because the migration is purely additive.
  • Startup bootstrap inserts the row. BootstrapLocalAgent runs every boot. The store impl uses INSERT OR IGNORE on a stable id (deterministic UUID5), so the first post-upgrade boot inserts; every subsequent boot is a no-op. Same for BootstrapTeamAgent on the synthetic 'default' team.
  • Existing data stays untouched. Tasks, runs, task_rules, prompt_triggers — none reference agents in this ticket. Their PAT remains in the keychain; the spawner's credential lookup keeps falling through to it because agents.github_pat_user_id is NULL in local mode. The agents row is metadata that becomes load-bearing only when D-Claims wires claimed_by_agent_id + actor_agent_id (which lands with its own backfill — existing tasks default to claim-NULL until a human or trigger claims them).

Net result for a v1.10.1 user upgrading: one new row in agents, one in team_agents, zero behavioral change until later tickets surface them.

§6

RLS pattern

Two tables, four policies, one helper. The helper (tf.user_in_team) is the only new building block — D-TeamDefault reuses it across the broader sweep.

TableOperationGateWhy
agents SELECT tf.user_has_org_access(org_id) Any org member can see who their org's bot is — display name, model, credential source. Helps the human triage UI surface "the bot is currently set to model X."
INSERT / UPDATE / DELETE tf.user_is_org_admin(org_id) Bot identity is org-wide. Rename, model change, credential rotation all need admin sign-off. Bootstrap bypasses via the admin pool.
team_agents SELECT tf.user_in_team(team_id) A team's bot config is the team's business. Other teams in the same org can't see whether the bot is enabled or what model it's using.
INSERT / UPDATE / DELETE tf.user_in_team(team_id) Any team member can toggle the bot for their team or override the model. The decision in the arch doc (agent-identity locked decision): per-team bot enable is a team-member power, not an admin power — same justification as "any team member can configure their own queue."

Bootstrap pool routing

Both bootstrap functions (BootstrapAgentForOrg, BootstrapTeamAgent) run on the admin pool because at signup time the founder's org_memberships + memberships rows are being inserted in the same transaction as the agents row. By the time the test asserting RLS runs, the rows exist and gating works correctly.

Edge case: if a non-admin team member calls SetEnabled(enabled=false) on every team in the org, they've effectively disabled the bot org-wide without admin approval. Not a bug — that's the team-member-power decision; if the org wants to prevent it, the workaround is "the user isn't on every team" which is enforced by membership. The arch doc accepts this trade.

Risks & open questions

Open questions for the owner

QuestionDefault if no answerWhy it matters
Should SetGitHubAppInstallation + SetGitHubPATUser be separate methods, or a single SetCredentialSource with a discriminator? Separate methods. The discriminator type is small (App vs PAT) and the call sites are distinct (D11 vs the local-install code). Two narrow methods read cleaner at the call site than one method + a switch. Both methods enforce the "exactly one populated" invariant by clearing the other field as a side effect. A single combined method would have to take both arguments (one nil) which is more error-prone.
Is agents.id = 'local-agent' too cute? Should we use a deterministic UUID instead? Resolved (in implementation): deterministic UUID via db.BootstrapAgentID(orgID) on both backends. Agent.ID on the input is ignored — letting callers override creates a foot-gun where SQLite (no UNIQUE(org_id)) would write a row that GetForOrg's deterministic lookup couldn't reach. If a local install ever upgrades to multi (which the arch doc says they don't, but) or if local-mode data ends up in a Postgres dev env, a TEXT id like "local-agent" would clash with Postgres's UUID-typed PK. Stable UUID avoids the trap.
Does team_agents need an updated_at column? No. added_at captures creation; updates flip enabled + overrides, all of which are reflected via the parent agents.updated_at. If we want a per-team audit later, add it then. Less surface to keep coherent. The architecture's pattern is "the parent row carries the timestamp; the link table carries the join."
Should BootstrapTeamAgent error or no-op when the org has no agent row? Error. Calling team-bootstrap before org-bootstrap is a sequencing bug in the caller (D5/D7 handler), and silent no-op would leave teams with no bot — surprising to debug. Loud failure beats silent skip when the calling code is downstream from us.
RLS on team_agents writes — should the toggle be team-member-writable or admin-only? Team-member-writable. Matches the locked architectural decision; the upside is "teams own their queue + their bot" and the downside (a team member maliciously toggling) is bounded by membership scope. If we lock it to admin-only, every team needs an admin to flip the bot off, which is operationally annoying for orgs with many teams.
