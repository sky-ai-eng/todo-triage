§1

Why now

D1 (goose) and D2 (per-resource store interfaces) need a real Postgres schema to work against — D2's whole acceptance is "tests pass against both SQLite and Postgres impls." Without D3, the Postgres side of D2 has nothing to compile against and the `internal/store/postgres/*` files are pure speculation.

D3 is also the first ticket where multi-tenant isolation becomes a thing the code can express. The architecture stance — service-role connection + SET LOCAL request.jwt.claims + RLS on every org-scoped table — only exists if the schema has the policies attached. D3 lays the floor: every downstream multi-mode ticket (D6 GoTrue, D7 auth middleware, D9 org-scoping pass) reads from this schema.

Carved out: nothing here touches Go callsites. D3 is purely Postgres DDL + roles + policies. D2's Postgres implementations land in their own PR; D4 (mode flag) decides when to open the Postgres connection; D6 (GoTrue) deploys auth on top of this schema.
§2

Current state

  • internal/db/migrations-sqlite/202605090001_baseline.sql — the consolidated SQLite baseline that D1 landed.
  • internal/db/migrations-postgres/ — empty tree with a placeholder README, scaffolded by D1 so the dialect router has a destination.
  • internal/db/migrations.go's detectDialect hardcodes "sqlite3"; migrationsFor("postgres") resolves to the empty tree without erroring.
  • No Postgres driver in go.mod. No connection-string handling for non-SQLite. No code path that opens a Postgres DB.

So the runner, dialect routing, and embed plumbing are all in place. D3 fills the tree; D6 (GoTrue) and D4 (mode flag) decide when to point a real connection at it.

§3

Target state

One goose baseline migration in internal/db/migrations-postgres/ brings a fresh Postgres up to a multi-tenant-ready schema. goose.Up against an empty Postgres DB produces:

  • Roles: tf_app (the role the application will connect as — no superuser, no BYPASSRLS).
  • Extensions: pgcrypto (for gen_random_uuid()), supabase_vault (for Vault wrappers).
  • Schemas: tf (RLS helper functions live here), auth stub (a minimal auth.users placeholder so our FKs resolve when GoTrue isn't running yet — see §4 callout).
  • Tenancy tables: orgs, teams, memberships, org_invites, users (mirror of auth.users), sessions, integrations, org_limits, project_knowledge.
  • Vault wrappers: three SECURITY INVOKER functions (vault_put_org_secret, vault_get_org_secret, vault_delete_org_secret) + revoked-from-PUBLIC + granted to tf_app.
  • RLS helpers: tf.current_user_id(), tf.current_org_id(), tf.user_has_org_access(target_org).
  • TF tables (multi shape): every table from the SQLite baseline recreated with org_id NOT NULL REFERENCES orgs(id) ON DELETE CASCADE + (where applicable) creator_user_id, visibility, team_id baked in. SQLite-isms translated: TEXT PRIMARY KEY → UUID PRIMARY KEY DEFAULT gen_random_uuid(), TEXT JSON columns → JSONB, BOOLEAN stays, DATETIME → TIMESTAMPTZ, autoincrement → BIGSERIAL.
  • RLS policies on every org-scoped table: FOR SELECT USING (org_id = tf.current_org_id() AND ...), FOR INSERT WITH CHECK (...), etc.

Local SQLite migrations are untouched. The two trees never interact at runtime — migrationsFor picks one based on dialect.

§4

Tenancy schema

Identity, orgs, teams, memberships, sessions, integrations. Verbatim translation of the architecture doc §5 schema with light commentary on why each choice.

Auth schema dependency

The GoTrue split. Our users table FKs id → auth.users(id) ON DELETE CASCADE. GoTrue (D6) creates the auth schema on first boot; in production deployments GoTrue starts before the TF web tier so auth.users exists. For tests and standalone Postgres bootstrap, the migration creates a minimal auth.users stub (CREATE SCHEMA IF NOT EXISTS auth; CREATE TABLE IF NOT EXISTS auth.users (id UUID PRIMARY KEY, email TEXT)) so the FK resolves. When GoTrue layers in, its full schema is a superset — the stub columns are forward-compatible.

Schema sketch

-- tf-namespaced helpers + a minimal auth stub for standalone bootstrap
CREATE SCHEMA IF NOT EXISTS tf;
CREATE SCHEMA IF NOT EXISTS auth;
CREATE TABLE IF NOT EXISTS auth.users (
  id    UUID PRIMARY KEY,
  email TEXT
);

-- our user mirror — FK to auth.users for cascade-on-delete
CREATE TABLE users (
  id            UUID PRIMARY KEY REFERENCES auth.users(id) ON DELETE CASCADE,
  display_name  TEXT,
  avatar_url    TEXT,
  created_at    TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE TABLE orgs (
  id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  slug         TEXT NOT NULL UNIQUE,
  name         TEXT NOT NULL,
  created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
  deleted_at   TIMESTAMPTZ
);

CREATE TABLE teams (
  id      UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  org_id  UUID NOT NULL REFERENCES orgs(id) ON DELETE CASCADE,
  slug    TEXT NOT NULL,
  name    TEXT NOT NULL,
  UNIQUE (org_id, slug)
);

CREATE TYPE membership_role AS ENUM ('owner', 'admin', 'member', 'viewer');

CREATE TABLE memberships (
  user_id    UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  team_id    UUID NOT NULL REFERENCES teams(id) ON DELETE CASCADE,
  role       membership_role NOT NULL DEFAULT 'member',
  created_at TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, team_id)
);

-- sessions: opaque ID, AES-GCM-encrypted JWT/refresh (key = TF_SESSION_KEY)
CREATE TABLE sessions (
  id                UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id           UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  jwt_enc           BYTEA NOT NULL,
  jwt_nonce         BYTEA NOT NULL,
  refresh_token_enc BYTEA NOT NULL,
  refresh_nonce     BYTEA NOT NULL,
  jwt_expires_at    TIMESTAMPTZ NOT NULL,
  expires_at        TIMESTAMPTZ NOT NULL,
  created_at        TIMESTAMPTZ NOT NULL DEFAULT now(),
  last_seen_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  revoked_at        TIMESTAMPTZ,
  user_agent        TEXT,
  ip_addr           INET,
  CHECK (expires_at > created_at),
  CHECK (jwt_expires_at <= expires_at)
);
CREATE INDEX sessions_user_idx    ON sessions(user_id)    WHERE revoked_at IS NULL;
CREATE INDEX sessions_expires_idx ON sessions(expires_at) WHERE revoked_at IS NULL;

-- integrations + org_limits + project_knowledge — see arch doc §5 verbatim

Why UUID over BIGSERIAL: primary keys are emitted to Vault paths (org/<uuid>/<key>), JWT claims, and external integration metadata. UUIDs prevent ID-guessing attacks and make cross-system references unambiguous. Performance overhead vs sequential IDs is acceptable at the scale this targets.

Why JSONB for integrations.config: shape varies per integration (github_app's installation_id vs jira_oauth's site URL); a JSONB column with key extraction at query time beats a sparse columnar layout, and pg's JSONB indexing handles selective lookups when needed.

§5

Vault wrappers

Three SECURITY INVOKER functions in public that the Go backend calls. Org-prefixed naming (org/<uuid>/<key>) makes leakage between orgs structurally impossible at the Vault layer.

CREATE EXTENSION IF NOT EXISTS supabase_vault WITH SCHEMA vault;

CREATE OR REPLACE FUNCTION public.vault_put_org_secret(
  p_org_id     UUID,
  p_key        TEXT,
  p_secret     TEXT,
  p_description TEXT DEFAULT NULL
) RETURNS UUID
LANGUAGE plpgsql SECURITY INVOKER
SET search_path TO public, vault
AS $$
DECLARE
  v_full_name TEXT := 'org/' || p_org_id::text || '/' || p_key;
  v_existing  UUID;
BEGIN
  SELECT id INTO v_existing FROM vault.decrypted_secrets WHERE name = v_full_name;
  IF v_existing IS NOT NULL THEN
    PERFORM vault.update_secret(v_existing, p_secret, v_full_name, p_description);
    RETURN v_existing;
  END IF;
  RETURN vault.create_secret(p_secret, v_full_name, p_description);
END;
$$;

-- vault_get_org_secret + vault_delete_org_secret — same shape, see arch doc §5

REVOKE ALL ON FUNCTION public.vault_put_org_secret    FROM PUBLIC;
REVOKE ALL ON FUNCTION public.vault_get_org_secret    FROM PUBLIC;
REVOKE ALL ON FUNCTION public.vault_delete_org_secret FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.vault_put_org_secret    TO tf_app;
GRANT EXECUTE ON FUNCTION public.vault_get_org_secret    TO tf_app;
GRANT EXECUTE ON FUNCTION public.vault_delete_org_secret TO tf_app;
Why SECURITY INVOKER, not DEFINER: the function runs with the caller's privileges. tf_app can call it; nothing else can (because of the REVOKE/GRANT). DEFINER would let anyone with EXECUTE bypass the row-level Vault gates that pgsodium enforces. INVOKER keeps the security model uniform — whatever privileges we grant tf_app are exactly what it has.
Sessions stay app-layer-encrypted, not Vault. The sessions table holds AES-GCM ciphertext (key = TF_SESSION_KEY env var) directly in BYTEA columns. Vault is too high-frequency for per-request RPC overhead. Org secrets (PATs, API tokens) go through Vault because they're written once per integration setup and read at most a few times per minute.
§6

RLS pattern

Three helper functions live in tf schema; every org-scoped table gets ENABLE ROW LEVEL SECURITY + per-operation policies. The application sets request.jwt.claims per-transaction via SET LOCAL; helpers extract user_id and org_id from there.

Helpers

CREATE FUNCTION tf.current_user_id() RETURNS UUID AS $$
  SELECT NULLIF(current_setting('request.jwt.claims', true)::jsonb ->> 'sub', '')::uuid;
$$ LANGUAGE SQL STABLE;

CREATE FUNCTION tf.current_org_id() RETURNS UUID AS $$
  SELECT NULLIF(current_setting('request.jwt.claims', true)::jsonb ->> 'org_id', '')::uuid;
$$ LANGUAGE SQL STABLE;

CREATE FUNCTION tf.user_has_org_access(target_org UUID) RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM memberships m
    JOIN teams t ON t.id = m.team_id
    WHERE m.user_id = tf.current_user_id() AND t.org_id = target_org
  );
$$ LANGUAGE SQL STABLE;

Policy template

ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;

CREATE POLICY tasks_select ON tasks
  FOR SELECT USING (
    org_id = tf.current_org_id()
    AND creator_user_id = tf.current_user_id()    -- v1: tasks user-scoped
  );

CREATE POLICY tasks_insert ON tasks
  FOR INSERT WITH CHECK (
    org_id = tf.current_org_id()
    AND creator_user_id = tf.current_user_id()
    AND tf.user_has_org_access(org_id)
  );

-- update + delete policies follow the same shape; full enumeration in the migration

TF tables in multi shape

Every table from the SQLite baseline gets a Postgres counterpart with multi-tenant columns baked in from inception. Translation is mostly mechanical, but a few SQLite-isms need explicit decisions.

Translation rules

SQLitePostgres
TEXT PRIMARY KEY (UUID strings)UUID PRIMARY KEY DEFAULT gen_random_uuid()
INTEGER PRIMARY KEY AUTOINCREMENTBIGSERIAL PRIMARY KEY (or BIGINT GENERATED ALWAYS AS IDENTITY)
TEXT (JSON payloads)JSONB
DATETIMETIMESTAMPTZ
BOOLEAN DEFAULT 0BOOLEAN NOT NULL DEFAULT FALSE
BLOBBYTEA
Partial UNIQUE INDEX (e.g. one-per-run primary)Same syntax — partial indexes are core PG, not a SQLite extension
CHECK (id = 1) singleton (settings)Drop — multi mode has no per-binary settings table; settings live per-org

Multi-tenant columns by category

TablesColumns added on top of the SQLite shape
tasks, runs, prompts, projects, task_rules, prompt_triggers, swipe_events, pending_firings org_id UUID NOT NULL, creator_user_id UUID NOT NULL, visibility TEXT NOT NULL DEFAULT 'private', team_id UUID NULL REFERENCES teams(id) ON DELETE SET NULL
entities, events, entity_links, task_events, repo_profiles, poller_state org_id UUID NOT NULL only — no creator scope (these are org-shared infrastructure)
run_artifacts, run_messages, run_memory, run_worktrees, pending_prs, pending_reviews, pending_review_comments, curator_messages, curator_pending_context org_id UUID NOT NULL denormalized from the parent (runs / pending_reviews / curator_requests) for RLS-policy efficiency — avoids policies that have to JOIN to figure out access
system_prompt_versions, events_catalog Global tables — no org_id. Read-only for tf_app; written only by the migration / seeder. RLS not enabled.
settings Dropped from multi schema — settings are per-org (a future ticket) rather than per-binary.

Every org_id column references orgs(id) ON DELETE CASCADE. Deleting an org takes its entire data graph with it (modulo soft-delete on orgs.deleted_at for the 30-day retention window).

§8

Test rig

D3 is the first ticket that needs Postgres at runtime. Picking the test rig now sets the pattern D2 (Postgres impls) and every later multi-mode ticket inherits.

Image: supabase/postgres:15.1.0.147

Pinned to the version the production compose stack uses (architecture §5). It bundles supabase_vault, pgsodium, pgcrypto, pgvector, pg_graphql, etc. — vanilla Postgres + manually-installed extensions is fragile because the Vault extension's pgsodium key bootstrap is finicky. Using the same image as production also collapses test-vs-prod parity questions.

Harness: testcontainers-go

Spins ephemeral Postgres containers per test package via testcontainers/testcontainers-go. One container per test binary (cached across go test invocations within the same package), with tc.Reset() between tests — way faster than per-test container churn. The harness lives in internal/db/pgtest/ so any package that needs Postgres can import it.

// internal/db/pgtest/harness.go (sketch)
func NewPostgres(t *testing.T) *sql.DB {
  ctx := context.Background()
  pg, err := postgres.Run(ctx,
    "supabase/postgres:15.1.0.147",
    postgres.WithDatabase("tf_test"),
    postgres.WithUsername("tf"),
    postgres.WithPassword("tf"),
    testcontainers.WithWaitStrategy(wait.ForLog("database system is ready")),
  )
  if err != nil { t.Fatal(err) }
  t.Cleanup(func() { pg.Terminate(ctx) })

  dsn, _ := pg.ConnectionString(ctx, "sslmode=disable")
  db, _ := sql.Open("pgx", dsn)
  if err := db.MigrateForTest(); err != nil { t.Fatal(err) }
  return db
}

CI

GitHub Actions runner already has Docker available; testcontainers-go uses the host daemon. No service-container declaration needed — the test process boots its own Postgres. First-run pulls the image (~400MB); subsequent runs hit the layer cache.

Why not GHA service container: service containers can't easily install supabase_vault on a vanilla postgres image (requires pgsodium extension at compile time). Using the supabase/postgres image as a service container is doable, but testcontainers gives us better per-test isolation and lets local dev run the same path without docker-compose.
pgx, not lib/pq: the modern Postgres driver. Better connection pooling, native LISTEN/NOTIFY, COPY support. Add github.com/jackc/pgx/v5 + the stdlib sub-package so database/sql works.

§10

Risks & open questions

Open questions for the owner

QuestionDefault if no answerWhy it matters
Does GoTrue own auth.users, or do we stub it? Stub a minimal auth.users in our migration. GoTrue's full schema is a superset of our stub. If we don't stub, fresh-Postgres bootstrap requires GoTrue to run first — couples test setup to GoTrue and complicates D6's sequencing.
Does the goose migration CREATE ROLE tf_app, or is that a deployment-script concern? Migration creates the role idempotently (DO $$ BEGIN IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'tf_app') THEN CREATE ROLE tf_app NOLOGIN; END IF; END $$;). Postgres roles are cluster-global, not per-DB. If the migration creates them, two parallel goose.Up against the same cluster could conflict — the IF NOT EXISTS guard handles that.
Migration as one big file, or split (tenancy / vault / RLS-helpers / RLS-policies / TF-tables)? One big file. Mirrors D1's single-baseline approach, easier to review as a unit, no inter-file ordering concerns. Splitting buys reviewability per concern but introduces ordering fragility (Vault wrappers reference orgs(id), so they can't run before tenancy).
First-org bootstrap mechanism Out of scope for D3. D7 (auth middleware) detects empty orgs on first user signup and creates one. D3 just guarantees the schema supports it. Could land here as a Postgres trigger — but triggers running app logic are a smell, and D7's middleware already touches the signup path.
Test rig: testcontainers-go vs supabase/postgres GHA service container testcontainers-go (lets local dev run the same path; better per-test isolation; supabase/postgres image bundles everything). GHA service container is simpler in CI but doesn't help local dev, and supabase/postgres's pgsodium bootstrap doesn't always survive a stripped-down service-container env.

Risks & rollback

RiskMitigationRollback
An RLS policy is wrong → user sees other orgs' data, or can't see their own Cross-org isolation tests for every org-scoped resource. Tests run in CI; failures block merge. Patch the policy in a follow-up migration; DROP POLICY foo ON tasks; CREATE POLICY ... in a forward-only migration.
Vault wrapper grants drift → unprivileged role can call wrappers without going through application auth Migration explicitly REVOKEs from PUBLIC and GRANTs only to tf_app. Test that calling as a non-tf_app role errors. Re-run the REVOKE/GRANT pair in a follow-up migration.
SQLite-vs-Postgres schema drifts (e.g. a future migration only updates one tree) Conformance tests in D2 run identical assertions against both backends; a column added to one and not the other breaks the suite. —

The auth.users stub conflicts with what GoTrue creates on first boot Stub uses CREATE TABLE IF NOT EXISTS with only id UUID PRIMARY KEY and email TEXT — both columns exist in GoTrue's full schema. GoTrue's bootstrap migrations should ALTER-add their additional columns idempotently. If GoTrue refuses to layer in, drop the stub from a follow-up migration and require GoTrue-first deployment.
