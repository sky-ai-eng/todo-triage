§1

Why now

D1 (goose) and D2 (per-resource store interfaces) need a real Postgres schema to work against — D2's whole acceptance is "tests pass against both SQLite and Postgres impls." Without D3, the Postgres side of D2 has nothing to compile against and the `internal/store/postgres/*` files are pure speculation.

D3 is also the first ticket where multi-tenant isolation becomes a thing the code can express. The architecture stance — service-role connection + SET LOCAL request.jwt.claims + RLS on every org-scoped table — only exists if the schema has the policies attached. D3 lays the floor: every downstream multi-mode ticket (D6 GoTrue, D7 auth middleware, D9 org-scoping pass) reads from this schema.

Carved out: nothing here touches Go callsites. D3 is purely Postgres DDL + roles + policies. D2's Postgres implementations land in their own PR; D4 (mode flag) decides when to open the Postgres connection; D6 (GoTrue) deploys auth on top of this schema.

Current state

§3

Target state

One goose baseline migration in internal/db/migrations-postgres/ brings a fresh Postgres up to a multi-tenant-ready schema. goose.Up against an empty Postgres DB produces:

  • Roles: tf_app (NOLOGIN, NOINHERIT, BYPASSRLS=false) — the role the app's transactions execute under. The migration also GRANT tf_app TO authenticator (the image's pre-shipped LOGIN role) so the connection-as-authenticator + SET LOCAL ROLE tf_app pattern works. See §4 for the full ceremony.
  • Extensions: supabase_vault, pgsodium, pgcrypto, pg_graphql, pgjwt, uuid-ossp are already loaded by the supabase/postgres image. Our migration calls CREATE EXTENSION IF NOT EXISTS supabase_vault defensively but it's a no-op in practice.
  • Schemas: tf (RLS helper functions live here). auth + vault + storage + graphql + realtime + extensions are pre-created by the image — we don't touch them beyond FK references.
  • Tenancy tables: orgs, teams, memberships, org_invites, users (mirror of auth.users; FK to auth.users(id) resolves natively because the image ships the full GoTrue schema), sessions, integrations, org_limits, project_knowledge.
  • Settings tables: org_settings, team_settings, user_settings (columnar, one row per scope-key) plus jira_project_status_rules for per-project Jira workflow vocabularies. See §8 for the column mapping.
  • Vault wrappers: three SECURITY INVOKER functions (vault_put_org_secret, vault_get_org_secret, vault_delete_org_secret) + revoked-from-PUBLIC + granted to tf_app.
  • RLS helpers: tf.current_user_id(), tf.current_org_id(), tf.user_has_org_access(target_org).
  • TF tables (multi shape): every table from the SQLite baseline recreated with org_id NOT NULL REFERENCES orgs(id) ON DELETE CASCADE + (where applicable) creator_user_id, visibility, team_id baked in. SQLite-isms translated per §8. Indexes replicated with org_id as the leading column on org-scoped tables.
  • RLS policies on every org-scoped table: FOR SELECT USING (org_id = tf.current_org_id() AND ...), FOR INSERT WITH CHECK (...), etc. Test harness exercises them as tf_app (RLS active), not as postgres superuser (RLS bypassed).
  • Reference data baked in: events_catalog seeded via INSERTs in the migration itself (mirrors supabase convention; avoids requiring app-layer seeding privileges on a table tf_app has no INSERT grant on). Same treatment for system_prompt_versions if shipping with system prompts pre-loaded.

D4a is merged but D3 doesn't import it. internal/runmode is the deployment-posture flag; D3 is pure SQL DDL with no Go integration. D4b will be the first runmode consumer (paths package). Leaving the runmode reference here only as forward-context.

Local SQLite migrations are untouched. The two trees never interact at runtime — migrationsFor picks one based on dialect.

§4

Roles & connection pattern

The supabase-idiomatic authenticator + SET LOCAL ROLE pattern, adapted for our Go-backend-on-Postgres setup. One LOGIN role at the connection level (image-shipped authenticator); one NOLOGIN role at the transaction level (tf_app) that RLS policies actually gate against. Migrations and tests both run as postgres superuser, which bypasses RLS — that's intentional for setup, but it means RLS-asserting tests must SET LOCAL ROLE tf_app or the assertions are silently meaningless.

Migration-time grants

-- Idempotent role creation (cluster-global, so guard).
DO $$
BEGIN
  IF NOT EXISTS (SELECT FROM pg_roles WHERE rolname = 'tf_app') THEN
    CREATE ROLE tf_app NOLOGIN NOINHERIT;
  END IF;
END
$$;

-- Let the LOGIN role switch into tf_app.
GRANT tf_app TO authenticator;

-- Schema usage.
GRANT USAGE ON SCHEMA public, tf TO tf_app;

-- Table-level grants on TF data tables — defaults applied at table creation.
ALTER DEFAULT PRIVILEGES IN SCHEMA public
  GRANT SELECT, INSERT, UPDATE, DELETE ON TABLES TO tf_app;

-- Read-only on global reference tables (events_catalog, system_prompt_versions).
-- Migration writes them; tf_app only reads.
REVOKE INSERT, UPDATE, DELETE ON events_catalog, system_prompt_versions FROM tf_app;

-- Vault wrapper EXECUTE (REVOKEd from PUBLIC; see §6).
GRANT EXECUTE ON FUNCTION public.vault_put_org_secret, public.vault_get_org_secret, public.vault_delete_org_secret TO tf_app;

Per-transaction ceremony (D2's WithTx)

// internal/store/postgres/tx.go (sketch — D2's territory, included here for context)
func WithTx(ctx context.Context, db *sql.DB, fn func(s TxStores) error) error {
  tx, err := db.BeginTx(ctx, nil)
  if err != nil { return err }
  defer tx.Rollback()

  // 1. Drop into tf_app — no superuser, no BYPASSRLS, RLS is now active.
  if _, err := tx.ExecContext(ctx, `SET LOCAL ROLE tf_app`); err != nil {
    return err
  }

  // 2. Set the JWT claims so RLS helpers (tf.current_user_id, tf.current_org_id) can read them.
  claims := ctxClaims(ctx)
  payload, _ := json.Marshal(claims)
  if _, err := tx.ExecContext(ctx,
    `SELECT set_config('request.jwt.claims', $1, true)`, string(payload)); err != nil {
    return err
  }

  // 3. Run the user's transaction body — RLS now in effect.
  if err := fn(newTxStores(tx)); err != nil { return err }
  return tx.Commit()
}

The SET LOCAL ROLE and SET LOCAL request.jwt.claims are both transaction-scoped — when the tx commits/rolls back, they reset on the next checkout from the connection pool. That's the load-bearing property: pgbouncer in transaction mode can multiplex many requests over one underlying connection, and each one sees a clean role + claims state.

Test rig implications

The test harness needs two connection paths against the same testcontainer:

  • Migration / setup connection — connects as postgres (superuser). Runs goose.Up, seeds tenancy data (orgs, users, memberships), creates JWT-claim values for tests. RLS is bypassed under this role; that's correct for setup.
  • Assertion connection — connects as authenticator with the right credentials, then SET LOCAL ROLE tf_app + set_config('request.jwt.claims', …) per test. RLS is active. Cross-org isolation tests run here.

Conflating these — running RLS assertions through the superuser connection — silently no-ops the policies and produces false-pass results. The harness exposes both connections explicitly so test authors can't accidentally grab the wrong one.

Why NOINHERIT on tf_app: NOINHERIT means privileges granted to tf_app aren't automatically inherited by roles that have tf_app as a member (like authenticator). The privileges only apply once the inheriting role explicitly does SET ROLE tf_app. Without this, authenticator would silently inherit tf_app's grants by default, and connections that forgot to SET LOCAL ROLE would still work — masking the bug rather than surfacing it. NOINHERIT makes "forgot to SET ROLE" a permission denial, which is what we want.

Tenancy schema

Identity, orgs, teams, memberships, sessions, integrations. Verbatim translation of the architecture doc §5 schema with light commentary on why each choice.

What the image gives us free

Verified against supabase/postgres:15.1.0.147: the image's init scripts pre-create the full GoTrue auth schema (auth.users, auth.refresh_tokens, auth.audit_log_entries, auth.instances, auth.schema_migrations) before any service connects. Our users.id REFERENCES auth.users(id) ON DELETE CASCADE resolves natively at goose.Up time. No stub needed. When GoTrue boots, it owns the schema going forward and our FKs continue to work because the image-installed schema matches what GoTrue expects.

-- integrations + org_limits + project_knowledge — see arch doc §5 verbatim

Why UUID over BIGSERIAL: primary keys are emitted to Vault paths (org/<uuid>/<key>), JWT claims, and external integration metadata. UUIDs prevent ID-guessing attacks and make cross-system references unambiguous. Performance overhead vs sequential IDs is acceptable at the scale this targets.

Why JSONB for integrations.config: shape varies per integration (github_app's installation_id vs jira_oauth's site URL); a JSONB column with key extraction at query time beats a sparse columnar layout, and pg's JSONB indexing handles selective lookups when needed.

§6

Vault wrappers

Three SECURITY INVOKER functions in public that the Go backend calls. Org-prefixed naming (org/<uuid>/<key>) makes leakage between orgs structurally impossible at the Vault layer.

-- supabase_vault is pre-loaded by the image; this is defensive only
CREATE EXTENSION IF NOT EXISTS supabase_vault WITH SCHEMA vault;

CREATE OR REPLACE FUNCTION public.vault_put_org_secret(
  p_org_id     UUID,
  p_key        TEXT,
  p_secret     TEXT,
  p_description TEXT DEFAULT NULL
) RETURNS UUID
LANGUAGE plpgsql SECURITY INVOKER
SET search_path TO public, vault
AS $$
DECLARE
  v_full_name TEXT := 'org/' || p_org_id::text || '/' || p_key;
  v_existing  UUID;
BEGIN
  SELECT id INTO v_existing FROM vault.decrypted_secrets WHERE name = v_full_name;
  IF v_existing IS NOT NULL THEN
    PERFORM vault.update_secret(v_existing, p_secret, v_full_name, p_description);
    RETURN v_existing;
  END IF;
  RETURN vault.create_secret(p_secret, v_full_name, p_description);
END;
$$;

-- vault_get_org_secret + vault_delete_org_secret — same shape, see arch doc §5

REVOKE ALL ON FUNCTION public.vault_put_org_secret    FROM PUBLIC;
REVOKE ALL ON FUNCTION public.vault_get_org_secret    FROM PUBLIC;
REVOKE ALL ON FUNCTION public.vault_delete_org_secret FROM PUBLIC;
GRANT EXECUTE ON FUNCTION public.vault_put_org_secret    TO tf_app;
GRANT EXECUTE ON FUNCTION public.vault_get_org_secret    TO tf_app;
GRANT EXECUTE ON FUNCTION public.vault_delete_org_secret TO tf_app;
Why SECURITY INVOKER, not DEFINER: the function runs with the caller's privileges. tf_app can call it; nothing else can (because of the REVOKE/GRANT). DEFINER would let anyone with EXECUTE bypass the row-level Vault gates that pgsodium enforces. INVOKER keeps the security model uniform — whatever privileges we grant tf_app are exactly what it has.
Sessions stay app-layer-encrypted, not Vault. The sessions table holds AES-GCM ciphertext (key = TF_SESSION_KEY env var) directly in BYTEA columns. Vault is too high-frequency for per-request RPC overhead. Org secrets (PATs, API tokens) go through Vault because they're written once per integration setup and read at most a few times per minute.
§7

RLS pattern

Three helper functions live in tf schema; every org-scoped table gets ENABLE ROW LEVEL SECURITY + per-operation policies. The application sets request.jwt.claims per-transaction via SET LOCAL; helpers extract user_id and org_id from there.

Helpers

CREATE FUNCTION tf.current_user_id() RETURNS UUID AS $$
  SELECT NULLIF(current_setting('request.jwt.claims', true)::jsonb ->> 'sub', '')::uuid;
$$ LANGUAGE SQL STABLE;

CREATE FUNCTION tf.current_org_id() RETURNS UUID AS $$
  SELECT NULLIF(current_setting('request.jwt.claims', true)::jsonb ->> 'org_id', '')::uuid;
$$ LANGUAGE SQL STABLE;

CREATE FUNCTION tf.user_has_org_access(target_org UUID) RETURNS BOOLEAN AS $$
  SELECT EXISTS (
    SELECT 1 FROM memberships m
    JOIN teams t ON t.id = m.team_id
    WHERE m.user_id = tf.current_user_id() AND t.org_id = target_org
  );
$$ LANGUAGE SQL STABLE;

Policy template

ALTER TABLE tasks ENABLE ROW LEVEL SECURITY;

CREATE POLICY tasks_select ON tasks
  FOR SELECT USING (
    org_id = tf.current_org_id()
    AND creator_user_id = tf.current_user_id()    -- v1: tasks user-scoped
  );

CREATE POLICY tasks_insert ON tasks
  FOR INSERT WITH CHECK (
    org_id = tf.current_org_id()
    AND creator_user_id = tf.current_user_id()
    AND tf.user_has_org_access(org_id)
  );

-- update + delete policies follow the same shape; full enumeration in the migration

The curator-chat scoping deserves an explicit callout: each user has their own ongoing conversation with the curator about a project (own creator_user_id on curator_requests; children denormalize org_id from the parent). Both Alice and Bob can teach the curator about the same project; their chats stay separate but the resulting writes to project_knowledge land in the shared, org-visible table with last-write-wins on conflicts. project_knowledge.last_updated_by retains the run id so the chronology is auditable. v2 can introduce shared/team-visible curator chats if usage shows people want them — same elevation pattern as memories.

Service-role caveat: The Postgres role TF connects as (tf_app) must NOT be a superuser and must NOT have BYPASSRLS. Both bypass row-level security. The migration role (used by goose) can be more privileged because it never serves user requests — but at minimum it needs CREATE on the relevant schemas, plus permission to CREATE EXTENSION.

Every org_id column references orgs(id) ON DELETE CASCADE. Deleting an org takes its entire data graph with it (modulo soft-delete on orgs.deleted_at for the 30-day retention window).

RLS posture per scope:

  • org_settings — readable by any user with org membership; writable only by org owners/admins (enforced via memberships role check in the policy).
  • team_settings — readable by team members; writable by team admins.
  • user_settings — readable + writable only by the owning user.
  • jira_project_status_rules — readable by any user with org membership; writable by org admins. Status rules are global per-Jira-project, not per-team — multiple teams tracking the same project share one rule set.

Deferred to a follow-up: the SettingsResolver application layer (which precedence layer wins per key, the merge logic, the UI surface). The schema gives us room without prescribing application behavior.

Indexes

Every SQLite index from the consolidated baseline gets a Postgres counterpart. On org-scoped tables, org_id is added as the leading column on the index — Postgres queries always filter by org_id via RLS, so the leading column matches the access pattern. Partial indexes (WHERE clauses on the existing SQLite indexes, e.g. idx_entities_closed_at WHERE closed_at IS NOT NULL) are preserved verbatim. Indexes that exist purely for SQLite-specific query patterns (rare) are dropped if a org_id-leading equivalent already covers the access path.

§9

Test rig

D3 is the first ticket that needs Postgres at runtime. Picking the test rig now sets the pattern D2 (Postgres impls) and every later multi-mode ticket inherits.

Image: supabase/postgres:15.1.0.147

Pinned to the version the production compose stack uses (architecture §5). It bundles supabase_vault, pgsodium, pgcrypto, pgvector, pg_graphql, etc. — vanilla Postgres + manually-installed extensions is fragile because the Vault extension's pgsodium key bootstrap is finicky. Using the same image as production also collapses test-vs-prod parity questions.

Harness: testcontainers-go

One container per test package, started once at TestMain time and torn down in cleanup. Inter-test isolation via explicit TRUNCATE of org-scoped tables in Reset(db) — testcontainers-go has no built-in reset API, so we own it. The harness lives in internal/db/pgtest/ so any package needing Postgres can import it.

// internal/db/pgtest/harness.go (sketch)
type Harness struct {
  Container *postgres.PostgresContainer
  AdminDB   *sql.DB    // connects as 'postgres' (superuser) — for migrate + setup. RLS bypassed.
  AppDB     *sql.DB    // connects as 'authenticator' — for RLS-asserting tests via SET LOCAL ROLE tf_app.
}

func New(t *testing.T) *Harness {
  ctx := context.Background()
  pg, err := postgres.Run(ctx,
    "supabase/postgres:15.1.0.147",
    postgres.WithDatabase("tf_test"),
    postgres.WithUsername("postgres"),
    postgres.WithPassword("postgres"),
    // "database system is ready" fires before image-init scripts complete
    // (auth schema setup, role bootstrap, etc.). Wait for image-init's tail
    // marker AND probe for auth.users existence as a belt-and-suspenders check.
    testcontainers.WithWaitStrategy(wait.ForAll(
      wait.ForLog("PostgreSQL init process complete").WithStartupTimeout(2*time.Minute),
      wait.ForSQL("5432/tcp", "pgx", func(host string, port nat.Port) string {
        return fmt.Sprintf("postgres://postgres:postgres@%s:%s/tf_test?sslmode=disable", host, port.Port())
      }).WithQuery("SELECT 1 FROM auth.users LIMIT 1").WithStartupTimeout(2*time.Minute),
    )),
  )
  if err != nil { t.Fatal(err) }
  t.Cleanup(func() { pg.Terminate(ctx) })

  adminDSN, _ := pg.ConnectionString(ctx, "sslmode=disable")   // user=postgres
  adminDB, _ := sql.Open("pgx", adminDSN)

  // Run goose migrations as superuser (creates tf_app + grants).
  if err := db.Migrate(adminDB); err != nil { t.Fatal(err) }

  // Set authenticator's password (image leaves it unset) and open a second
  // connection as authenticator. Tests use AppDB for RLS-asserting work and
  // always do SET LOCAL ROLE tf_app inside their transactions.
  if _, err := adminDB.Exec(`ALTER ROLE authenticator WITH LOGIN PASSWORD 'auth_test_pw'`); err != nil { t.Fatal(err) }
  appDSN := rewriteUser(adminDSN, "authenticator", "auth_test_pw")
  appDB, _ := sql.Open("pgx", appDSN)

  return &Harness{Container: pg, AdminDB: adminDB, AppDB: appDB}
}

// Reset truncates all org-scoped tables. Tests call this between cases
// when they're sharing the package-scoped harness (typical pattern).
func (h *Harness) Reset(t *testing.T) {
  _, err := h.AdminDB.Exec(`TRUNCATE orgs, teams, users, ... RESTART IDENTITY CASCADE`)
  if err != nil { t.Fatal(err) }
}

The two-connection model is load-bearing: a test that connects through AdminDB when it should be using AppDB will silently bypass RLS and pass even when policies are wrong. The harness exposes both as separate fields so the failure mode is "test author picked the wrong connection" (visible in code review) rather than "test silently lied" (invisible until a real RLS bug ships).

Wait strategy is non-trivial. The supabase/postgres image runs init scripts (auth schema setup, GoTrue's migrations, role bootstrap) after Postgres logs "database system is ready." A naive wait on that log races image-init and produces flaky tests where auth.users doesn't exist yet. Two-stage wait above (log marker + SQL probe for auth.users) makes the failure mode explicit.

CI

GitHub Actions runner already has Docker available; testcontainers-go uses the host daemon. No service-container declaration needed — the test process boots its own Postgres. First-run pulls the image (~400MB); subsequent runs hit the layer cache.

Why not GHA service container: service containers can't easily install supabase_vault on a vanilla postgres image (requires pgsodium extension at compile time). Using the supabase/postgres image as a service container is doable, but testcontainers gives us better per-test isolation and lets local dev run the same path without docker-compose.
pgx, not lib/pq: the modern Postgres driver. Better connection pooling, native LISTEN/NOTIFY, COPY support. Add github.com/jackc/pgx/v5 + the stdlib sub-package so database/sql works.

Risks & open questions

Resolved decisions

All open questions answered before implementation. Recording the resolutions here so the rationale survives in the spec rather than being buried in chat history.

QuestionResolution
What's the role + connection pattern? Supabase-idiomatic authenticator (LOGIN, NOINHERIT) + tf_app (NOLOGIN, NOINHERIT, BYPASSRLS=false). App connects as authenticator, WithTx does SET LOCAL ROLE tf_app + set_config('request.jwt.claims', …). Migrations + test setup run as postgres superuser. See §4.
Auth schema dependency on GoTrue? None. Image probe (docker run supabase/postgres:15.1.0.147) confirmed the image pre-creates the full GoTrue auth schema (with auth.users already shaped to GoTrue's expectations) and pre-loads supabase_vault + pgsodium. Our migration depends on neither.
Migration file layout — one or split? One big file (202605100001_pg_baseline.sql) mirroring D1's single-baseline approach. Inter-section ordering (roles before grants before policies) makes split files fragile.
How is events_catalog seeded? Seed INSERTs baked into the migration itself. Avoids requiring app-layer write privileges on a global reference table that tf_app only needs to read. Same treatment for system_prompt_versions.
Settings shape — JSONB blob or columnar? Columnar. Three tables per scope (org_settings, team_settings, user_settings) plus jira_project_status_rules for per-project Jira workflow. Settings keys are a fixed set, so columnar wins on type safety, discoverability, and future-reorg ergonomics. See §8.
Curator scope? Per-user-per-project chat (creator-scoped). Project knowledge writes hit the org-shared project_knowledge table with last-write-wins on conflicts. v2 can elevate to shared chat sessions if usage shows it's wanted. Matches arch doc §10's default-private stance.
Jira status rules — org or per-project? Per-project. Jira workflows are project-keyed in reality (different projects can use different workflow schemes), so storing rules at org level is wrong for any org with mixed project workflows. New table jira_project_status_rules; UI rework for setting per-project rules lands with D9 / settings UI overhaul.
First-org bootstrap? Out of scope for D3. D7 (auth middleware) detects empty orgs on first user signup and creates one. Schema only needs to support it.
Test rig — testcontainers-go vs GHA service container? testcontainers-go. Local dev parity, per-test isolation, image-init compatibility (verified by probe).
Wait strategy? Two-stage: wait.ForLog("PostgreSQL init process complete") + SQL probe for auth.users existence. The default "database system is ready" log fires before image-init (auth schema setup, role bootstrap) is done, so a naive wait races and produces flaky tests.
