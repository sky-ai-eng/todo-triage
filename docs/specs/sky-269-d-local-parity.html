D-LocalParity — synthetic tenancy rows in SQLite

Make local mode = multi mode at N=1 true at the schema layer, not just the routing layer. Local SQLite gains orgs / teams / users / memberships / org_memberships tables — exactly one row each, deterministic UUIDs — so every FK from a resource table resolves the same way both backends do. assertLocalOrg shrinks from "convention enforced at every store method" to "structural via WHERE org_id = ?."

§0

Phasing

Four phases, one PR. The schema work in A+B is the load-bearing part; C is a mechanical store sweep; D is cleanup.

A
Tenancy tables + synthetic rows
Migration creates orgs/teams/users/memberships/org_memberships + inserts the one row of each. Deterministic UUIDs.
B
Resource-table columns
Add org_id / team_id / creator_user_id to every existing resource table. Backfill to the synthetic IDs.
C
Store sweep
SQLite store impls query the new columns; assertLocalOrg shrinks to a defensive check at the runtime layer.
D
Drop special-cases
"NULL in local" comments on agents/team_agents go away. FKs get added back where they were dropped.
§1

Why now

Every downstream multi-mode ticket has been writing the same kind of "in SQLite this FK doesn't have a target" workaround. The cost keeps adding up.

TicketWhat it had to dodge
SKY-260 (shipped)agents.org_id has no FK locally; agents.github_pat_user_id permanently NULL because there's no users row to point at.
SKY-259 (next)Unified event_handlers wants FKs to users/teams/agents. Without users/teams locally, the SQLite migration has to drop the FKs.
SKY-261 (after)tasks.claimed_by_user_id and runs.creator_user_id FK targets don't exist locally.

The team-as-unit reframe locked "local mode = multi mode at N=1" as a load-bearing invariant. Today it's true at the routing-code level but not at the schema level. This PR closes that gap.

§4

Synthetic identity

Five deterministic UUIDs, derived once. The migration inserts each row with a hardcoded value; the corresponding runmode constant returns the same string. Re-runs are byte-identical.

runmode.LocalDefaultOrgID "00000000-0000-0000-0000-000000000001" → orgs.id
runmode.LocalDefaultTeamID "00000000-0000-0000-0000-000000000010" → teams.id
runmode.LocalDefaultUserID "00000000-0000-0000-0000-000000000100" → users.id
+ org_memberships (user, org, 'owner')
+ memberships (user, team, 'admin')
Why nil-shape UUIDs (00000000-...000N) instead of uuid.NewSHA1? Visibility. A nil-shape UUID is instantly recognizable in logs as "the synthetic local row" rather than as a random tenant. The risk — collision with a real multi-mode UUID — is zero (gen_random_uuid() never produces these). Same shape choice as Postgres conventions for sentinel UUIDs.
Drift hazard: the Go constants and the SQL migration's INSERT literals must stay byte-identical. Mitigation: a one-line unit test in internal/db/bootstrap_test.go that asserts SELECT id FROM orgs after bootstrap equals runmode.LocalDefaultOrgID, repeated for the other four sentinels. If a future edit drifts one side, the test fails loudly rather than producing silently-broken FKs at runtime.
Existing constants: runmode.LocalDefaultOrg = "default" (the string) is renamed but the export name stays — it just returns the UUID string instead. Same for db.LocalDefaultTeamID. Old callers don't compile-break; their value just changes shape. The arch doc + memory entry are updated.
§5

Store-impl transition

Most SQLite store methods replace assertLocalOrg(orgID) + bare SELECT with WHERE org_id = ?. Behavior unchanged because there's still only one org row.

Before — assertion-as-convention
func (s *taskRuleStore) List(ctx context.Context, orgID string) ([]domain.TaskRule, error) {
  if err := assertLocalOrg(orgID); err != nil {
    return nil, err
  }
  rows, err := s.q.QueryContext(ctx, `
    SELECT ` + taskRuleColumns + `
    FROM task_rules
    ORDER BY sort_order ASC
  `)
  // ...
}
After — structural via column
func (s *taskRuleStore) List(ctx context.Context, orgID string) ([]domain.TaskRule, error) {
  rows, err := s.q.QueryContext(ctx, `
    SELECT ` + taskRuleColumns + `
    FROM task_rules
    WHERE org_id = ?
    ORDER BY sort_order ASC
  `, orgID)
  // ...
}

Conformance suites pass unchanged — they were schema-blind by design. The Postgres impls are untouched (they already had WHERE org_id = ?).

What stays of assertLocalOrg

One central defense-in-depth check at the runtime layer rather than per-method: a small wrapper at the top of main.go's startup path that verifies runmode.LocalDefaultOrgID is present in orgs + matches the constant. If it's missing or has drifted, the binary refuses to boot. Inside store methods, the WHERE org_id = ? is the gate.

Bootstrap function shape

Existing db.BootstrapLocalAgent in internal/db/bootstrap.go stays (inserts the agents + team_agents rows). New sibling db.BootstrapLocalTenancy inserts the five tenancy rows, runs first. Both idempotent via INSERT OR IGNORE; both invoked from main.go after migrations.

§6

Existing installs (v1.10.1 → after-this)

Forward-only, idempotent, invisible. Three pieces line up:

  • Migration inserts the synthetic rows. Same migration that creates the tables INSERTs the five sentinel rows, gated on NOT EXISTS. Re-runs on already-populated installs are no-ops.
  • Migration backfills every resource table. ALTER TABLE prompts ADD COLUMN org_id TEXT NOT NULL DEFAULT '00000000-0000-0000-0000-000000000001' etc. for every resource. SQLite ALTER TABLE permits NOT NULL with a DEFAULT, which is what makes the backfill atomic with the column add.
  • Existing rows get the synthetic IDs by default. No data movement, no orphans. After backfill the DEFAULT is unnecessary but harmless to leave on (the application always sets the column explicitly via the SQLite store impls post-PR).

The startup bootstrap is then a no-op on migrated installs — the rows are already there. Net result for a v1.10.1 user upgrading past this PR: five new rows, three new columns per resource, zero behavioral change.

§7

Out of scope

  • RLS in SQLite. No RLS engine; the structural WHERE org_id = ? + the startup-time tenancy assertion are the local-mode defense in depth.
  • Auth in local mode. Still implicit single user. No sessions table, no JWTs, no GoTrue. users.id is just a row, not an authenticated identity.
  • Multi-user local mode. Not a feature. If a local user wants multi-user, that's a hosted-deployment thing.
  • Local-to-hosted export. Architecture doc says local users don't migrate to multi. Parallel schema makes a future export feasible if we ever build it, but this PR doesn't.
  • Migration of pre-runner installs. Refused with ErrPreRunnerInstall as today — SKY-245's gate stays in place.

Implementation steps

  1. New SQLite migration 202605120003_local_tenancy.sql Create the five tenancy tables. INSERT five sentinel rows. ALTER every resource table to add org_id / team_id / creator_user_id columns with NOT NULL DEFAULT (synthetic UUIDs). Add CHECK constraints where Postgres has them.
  2. runmode constants Change LocalDefaultOrg = "default" → LocalDefaultOrgID = "00000000-...000001" (rename or alias). Add LocalDefaultTeamID, LocalDefaultUserID. Old db.LocalDefaultTeamID can stay as a back-compat alias if any callers persist past the sweep.
  3. db.BootstrapLocalTenancy Idempotent INSERT-or-IGNORE on the five rows. Called from main.go after db.Migrate + before BootstrapLocalAgent.
  4. SQLite store sweep Every method that currently calls assertLocalOrg(orgID) swaps for a WHERE org_id = ? clause. assertLocalOrg itself either deletes or shrinks to a defense-in-depth top-of-bootstrap check.
  5. Wire agents/team_agents FKs SQLite agents/team_agents tables get real FK to orgs/teams. The "always NULL in local" comments on agents.github_pat_user_id change to "FK to users(id), the synthetic local user" — and the local-install bootstrap can now populate it.
  6. Tests + conformance Conformance suites should pass unchanged — they're schema-blind. Add TestBootstrapLocalTenancy_FreshInstall + _Idempotent + _PreservesUserChanges mirroring the existing agent bootstrap tests. Plus TestBootstrapLocalTenancy_ConstantsMatchRows as the anti-drift assertion (constant string == row's id for all five sentinels).
  7. Build + test + lint + ship Per the standard memory'd flow (frontend first, then go build, then go test, then lint).
§9

Risks

RiskMitigation
Renaming LocalDefaultOrg = "default" to a UUID string breaks any in-the-wild local DB whose existing rows already reference the literal "default". No existing rows do — the constant is only used in code paths that pass it through assertLocalOrg, never as a foreign-key value stored anywhere. Audit confirms zero stored references before this PR ships. Keep the old constant name + value as a back-compat alias for one release if paranoid.
SQLite ALTER TABLE ADD COLUMN NOT NULL DEFAULT '00000000-...' applied to a million-row table is slow. Largest local install ~10k rows in practice. Migration completes in <1s. Existing migration runner doesn't impose a deadline.
FK constraints on existing rows fail because (e.g.) a swipe_events row references a creator_user_id that doesn't exist. The DEFAULT clauses point at synthetic IDs that the same migration INSERTs first. No row can fail the FK at backfill time. New rows are FK-enforced going forward.
D-Predicate (SKY-264) was going to live in settings.row.github_username — does this preempt that decision? Yes, deliberately. users.github_username works on both backends after this PR, so the settings.row dance becomes unnecessary. Update SKY-264 to reflect.
