D-Predicate — replace author_is_self with author_in

"Self" has no canonical referent on a team-shared rule. Hard-cut the self-reference predicates (author_is_self, reviewer_is_self) to handle allowlists (author_in: ["aidan"], reviewer_in: [...]). Local mode captures the user's GitHub login from the keychain at bootstrap; multi mode will capture it from Supabase OAuth (SKY-251). Case-insensitive match at compare time.

Phasing

Four phases, one PR. Schema + identity capture first, then registry + matcher, then the JSON migration, then FE.

A
Schema + bootstrap

B

C

D

§1

Why now

SKY-261 made rules and triggers team-scoped. A predicate that says "author_is_self: true" was already ambiguous on a team — whose self? — and now there's no defensible answer because the row may have been authored by user A but is in force for user B.

ContextWhat "author_is_self" means todayProblem
Local mode (N=1)creds.GitHubUsername (keychain)Works. There's only ever one user.
Multi mode, rule owned by user Auser A's github_usernameIf user B has the rule applied to their team's tasks, "self" still means A — surprising.
Multi mode, system triggerundefinedSystem triggers have no creator. The predicate either silently never fires or panics.
Multi mode, shared team ruleundefined"Self" relative to the team is N-valued, not 1-valued.
Reframe: the predicate should name who it cares about, not say "me." A team rule that wants "any PR authored by anyone on Platform" needs the allowlist. A solo rule "any PR I opened" is just an allowlist of size 1. Same primitive, both cases.

Predicate shape change

Side-by-side, before and after, for the canonical case: PR review requested, filtered to PRs the user opened.

Before — self-reference
// event_handlers.scope_predicate_json
{
  "author_is_self": true,
  "is_draft": false
}

// matched against
{
  "author": "AidanAllchin",
  "author_is_self": true,
  // ↑ computed at emission
  //   from creds.GitHubUsername
}
After — allowlist
// event_handlers.scope_predicate_json
{
  "author_in": ["AidanAllchin"],
  "is_draft": false
}

// matched against
{
  "author": "AidanAllchin"
  // ↑ no derived field;
  //   the predicate decides
  //   who counts as "self"
}
Two simplifications fall out:
  • Events shrink: drop the AuthorIsSelf bool field from every event metadata struct. The emitter no longer needs to know who the active user is.
  • System triggers stop being a special case. A trigger shipped by us in seed.go with author_in: [] no longer depends on an undefined "self" concept; under the current matcher semantics, an empty allowlist means no author filter (matches all), not match-none.

Identity capture

The matcher needs one piece of data: each user's GitHub login. Already captured in local mode; deferred for multi.

Keychain no longer stores the username. Username isn't a secret — it's on every PR. Capturing it in the keychain was a historical accident from the PAT setup flow. With this PR, the keychain holds only the PAT (the actual credential); the DB users row holds the identity. Same shape as multi mode → no schema or storage divergence between modes.

Three capture call sites — all write the users row directly

  1. Setup wizard (internal/auth/setup_*.go): after auth.ValidateGitHub returns ghUser.Login, write Login to users.github_username instead of into the keychain Credentials struct.
  2. Settings handler (internal/server/settings.go:247): same — after ValidateGitHub succeeds, write to the users row. Don't write to creds.GitHubUsername (that field is being removed).
  3. Repos handler / onGitHubChanged callback: re-resolves Login via ValidateGitHub if the PAT changed; writes to users row.

All three call a shared helper SetLocalUserGitHubIdentity(db, login string) error that does the UPDATE users against the synthetic row. Idempotent on identical input.

One-shot migration for existing installs

Existing local installs have the username in the keychain (TRIAGE_FACTORY_GITHUB_USERNAME). At boot, after the schema migration runs, a one-shot backfill function fires:

func migrateLocalUserGitHubIdentity(database *sql.DB) error {
    if runmode.Current() != runmode.ModeLocal { return nil }
    // If DB already has a value, the migration has run before. No-op.
    var existing sql.NullString
    scanErr := database.QueryRow(
        `SELECT github_username FROM users WHERE id = ?`,
        runmode.LocalDefaultUserID,
    ).Scan(&existing)
    switch {
    case scanErr == nil:
        // Row and column exist; continue below.
    case errors.Is(scanErr, sql.ErrNoRows):
        return nil // synthetic local user row not present yet; nothing to migrate
    case strings.Contains(scanErr.Error(), "no such column"):
        return nil // schema ordering drift; keep legacy value and let a later run migrate it
    default:
        return scanErr
    }
    if existing.Valid && existing.String != "" { return nil }

    // Read legacy keychain location.
    legacyLogin, err := auth.LoadLegacyGitHubUsername() // reads TRIAGE_FACTORY_GITHUB_USERNAME
    if err != nil || legacyLogin == "" { return nil }

    // Write to DB, then clear the legacy keychain entry only if the synthetic row exists.
    result, err := database.Exec(
        `UPDATE users SET github_username = ? WHERE id = ?`,
        legacyLogin, runmode.LocalDefaultUserID,
    )
    if err != nil { return err }
    rowsAffected, err := result.RowsAffected()
    if err != nil { return err }
    if rowsAffected == 0 { return nil }

    return auth.DeleteLegacyGitHubUsername()
}

After this runs once per install, the keychain no longer stores the username. New PATs saved post-upgrade go through the new capture path (setup wizard / settings handler write directly to DB).

State transitions handled

  • Fresh install — setup wizard writes Login to DB at first PAT save. Migration backfill is a no-op (legacy keychain entry doesn't exist).
  • Upgrade from pre-SKY-264 — schema migration adds the column NULL; backfill function reads legacy keychain entry, writes to DB, deletes keychain entry. One-shot.
  • User cleared keychain via OS keychain UI — DB row still has username (we don't blank it). Predicates keep working. User re-auths to re-populate the PAT; ValidateGitHub re-resolves Login → settings handler rewrites DB. No restart needed.
  • User swaps to a different GitHub account via Settings — ValidateGitHub returns the new Login; settings handler writes the new value to DB. No restart needed.
  • User has no PAT configured yet — DB column is NULL. FE Variant-A toggle disables (see §8).
No fallback when NULL. If users.github_username is NULL and the user tries to render a "match my PRs" toggle, the FE shows a one-line "GitHub identity not yet captured — configure GitHub on Settings page" hint and disables the toggle. We don't pre-fill a plausible-but-wrong value like the OS username.

Multi mode — deferred to SKY-271

The right capture path in multi mode depends on auth context that doesn't exist yet (SKY-251 hasn't landed) and varies by deployment topology — hosted-SaaS against github.com/GHEC vs self-deploy against the customer's GHE. Candidate flows include SCIM-directory intersect when the customer has GHE SCIM enabled against a shared IdP, GitHub App user-to-server OAuth as a "Connect GitHub" action, an org-members picker with email/name fuzzy match, and PAT paste as a fallback. SKY-271 tracks the design once SKY-251's session/IdP shape is settled.

This spec ships the column + the local-mode capture. Multi-mode writes are guarded behind runmode.Current() == "local" at the bootstrap call site so we don't accidentally write to a real user's row in multi mode. NULL is a supported steady state in multi mode — predicates that need the user's own identity disable cleanly until SKY-271 ships.

No UNIQUE constraint. Two team members could theoretically have the same GitHub login bound to different user rows (rare bug condition during onboarding). We don't want a UNIQUE to refuse the second OAuth callback. Authoritative dedup happens at the auth layer, not the schema.

Boot wiring

Two functions, both called from main.go server startup after config.Init, before the HTTP server starts listening:

  1. migrateLocalUserGitHubIdentity(db) (see §4) — one-shot keychain → DB backfill + keychain deletion. No-op on installs where the DB already has a value.
  2. Poller wiring — poller startup reads username from users.github_username via the users store (not from creds.GitHubUsername, which is gone). If NULL: poller short-circuits the same way it does today when the PAT is missing.

Steady-state writes happen at the capture call sites (setup wizard, settings handler, repos handler), not at boot.

§6

Data migration

Rewrite every existing event_handlers.scope_predicate_json row that contains the deprecated keys. Go-side, inside the goose migration's -- +goose StatementBegin block, so we get JSON manipulation on both dialects without SQLite-version constraints.

Rewrite rule

Before (predicate JSON)AfterNotes
{"author_is_self":true} {"author_in":["<creator.github_username>"]} creator is event_handlers.creator_user_id; lookup users.github_username; if NULL, drop the field entirely (rule no longer self-scoped)
{"author_is_self":false} {} "not me" was rarely the intent; closer reading is "no filter on author". Drop the field.
{"reviewer_is_self":true} {"reviewer_in":["<creator.github_username>"]} same as author_is_self path
{"reviewer_is_self":false} {} drop

Migration shape

-- +goose Up
-- +goose StatementBegin

-- Loop in Go (called from migrations.go RunMigrations after
-- goose Up): SELECT id, creator_user_id, scope_predicate_json
-- FROM event_handlers; for each row, decode JSON, rewrite,
-- UPDATE if changed. Single transaction.

-- +goose StatementEnd
Drift edge case: if a row was created by a user whose github_username is NULL at migration time (user existed pre-this-PR; backfill happened on previous boot but for a different user; or username never captured) — drop the author_is_self field. The rule becomes "match any author" rather than "match nobody." That's a permissive failure mode, surfaced via a migration warning log: [migrate] handler 0123: dropped author_is_self predicate, creator's github_username was NULL. User can fix by editing the rule.
Local mode is fine. The bootstrap write happens at every boot, so by the time the migration runs in production, the synthetic user row has its github_username populated. We sequence: schema migration → bootstrap write → data migration. (Or: bootstrap-then-migrate, since the data migration is one-shot.)
§7

Matcher contract

One helper, lowercase at compare time, no allocation per call.

Helper

// stringInSliceFold returns true iff the slice contains s
// under ASCII case-insensitive comparison. nil/empty slice
// returns false (the predicate field was not set; matcher
// reads "no filter" not "match nothing" — caller decides).
func stringInSliceFold(slice []string, s string) bool {
    if len(slice) == 0 { return true }   // no filter
    for _, v := range slice {
        if strings.EqualFold(v, s) { return true }
    }
    return false
}
Empty slice = no filter, mirroring how the nil-pointer predicates behaved before (a missing author_is_self field was "don't filter on author"). An empty author_in: [] from the FE means "I haven't added anyone yet" — also no filter. If the user wants "match nobody," they delete the rule.

Per-event Matches function (example)

Before
func (p Predicate) Matches(m Metadata) bool {
    return boolEq(p.AuthorIsSelf, m.AuthorIsSelf) &&
        strEq(p.Author, m.Author) &&
        strEq(p.Repo, m.Repo) &&
        boolEq(p.IsDraft, m.IsDraft) &&
        hasLabel(p.HasLabel, m.Labels)
}
After
func (p Predicate) Matches(m Metadata) bool {
    return stringInSliceFold(p.AuthorIn, m.Author) &&
        strEq(p.Author, m.Author) &&
        strEq(p.Repo, m.Repo) &&
        boolEq(p.IsDraft, m.IsDraft) &&
        hasLabel(p.HasLabel, m.Labels)
}

What the event metadata loses

The AuthorIsSelf bool field disappears from every event metadata struct. The emitter (tracker / diff) no longer needs to know the active user's login at emission time — that knowledge moves to the predicate, where it belongs. Net code: events package shrinks.

§8

Frontend UX

One predicate field, two form variants. Local / one-person teams see a toggle; multi-person teams see a handle-list.

Variant A — N=1 (local or solo team)

Rule editor · predicate · github:pr:review_requested
Match my PRs only on
Match draft PRs off
  "my PRs" = author is AidanAllchin (captured from GitHub)

Serialization: toggle ON → author_in: ["AidanAllchin"]. Toggle OFF → field omitted. The user never sees author_in; it's a boolean to them.

Variant B — N≥2 (multi-person team)

Rule editor · predicate · github:pr:review_requested
Match PRs authored by Aidan Allchin · AidanAllchin × Jess Park · jess-platform × dependabot[bot] ×
  ▾ search team or type a handle…
Kevin Bow · kbow
Sarah Lin · slin-eng
Mike Chen · no GitHub identity
+ Add "dependabot[bot]" as external handle
Match draft PRs off

Searchable multi-select. Each selected user/handle renders as a chip with display name + GitHub login; clicking × removes. The dropdown lists team members (rows from /api/team/members); members whose github_username is NULL appear disabled with a "no GitHub identity" hint. Typing a string that doesn't match any team member surfaces an "Add as external handle" option (for bots like dependabot[bot] or contractors not on the team). Selecting a team member writes their github_username to author_in (not the display name). Wire shape: author_in: ["AidanAllchin", "jess-platform", "dependabot[bot]"].

Degraded state — github_username not captured

Rule editor · predicate · github:pr:review_requested
Match my PRs only disabled
  GitHub identity not yet captured. Re-auth on the Settings page to enable this filter.

Toggle disabled, hint links to /settings#github. In multi mode (Variant B), the user can still pick other team members from the dropdown even if their own identity is not captured — the disabled state only blocks the "match my own PRs" shortcut.

Stale chips after team membership changes. A user is removed from the team — their chip stays in any rule that referenced them (it's a literal string in author_in, not an FK). The chip renders with a muted style and a "no longer on team" tooltip. The rule keeps firing for their author events. To remove, the user clicks × explicitly. This matches the principle that author_in is a literal allowlist, not a live reference.
No save-time GitHub API validation. Team-member rows came from the OAuth-backed users table, so they're already valid GitHub handles. External handles are user-typed strings we accept as-is — the matcher does case-insensitive compare; typos won't match anything and surface when the rule doesn't fire.
§9

API surface

Two new endpoints. One tells the FE which variant to render; the other backs the searchable dropdown in Variant B.

GET /api/config

{
  "deployment_mode": "local",   // or "multi"
  "team_size": 1,             // active user's team headcount
  "current_user": {
    "id": "00000000-0000-0000-0000-000000000001",
    "github_username": "AidanAllchin" // or null
  }
}

FE rules:

  • team_size === 1 → render Variant A (toggle).
  • team_size > 1 → render Variant B (searchable multi-select).
  • current_user.github_username === null → render Variant A as disabled; Variant B still works (user can pick other team members or type external handles).

Local mode always returns deployment_mode: "local", team_size: 1, and the synthetic user row. Multi mode computes team_size from memberships for the active user's team (any team they belong to; if they're on multiple, take the max — UI conservatism).

GET /api/team/members

{
  "members": [
    { "user_id": "…", "display_name": "Aidan Allchin",
      "github_username": "AidanAllchin",    // or null
      "is_current_user": true },
    { "user_id": "…", "display_name": "Jess Park",
      "github_username": "jess-platform",
      "is_current_user": false },
    { "user_id": "…", "display_name": "Mike Chen",
      "github_username": null,                // hasn't completed OAuth backfill
      "is_current_user": false }
  ]
}

Returns the membership roster scoped to the active user's team. Used by Variant B's dropdown. Local-mode response is a single-entry array containing the synthetic user.

Caching: short FE cache (60s) is fine — a new team member joining mid-session doesn't need to immediately appear in the rule editor. The websocket can invalidate the cache on memberships changes if that becomes annoying, but v1 ships without that wiring.

Why two endpoints, not one. /api/config is a one-shot read at FE boot (deployment shape doesn't change). Membership rosters are mutable during a session (joins/leaves). Conflating them would force re-fetching the entire config blob to refresh the dropdown, or worse, force a websocket invalidation on something that's mostly static.
§10

Out of scope

  • Jira *_is_self predicates. Same conceptual fix, different identity store. Sibling ticket once Atlassian OAuth lands.
  • Validation against GitHub's API at save time. Handle typos surface at match time.
  • Team-membership-aware "anyone on my team" shortcut. A future ergonomic improvement (author_in_team: true) — not needed v1 because the user can manually list the team.
  • Live websocket invalidation of /api/team/members cache. v1 ships with a 60s FE-side cache. New team member appears in the dropdown on next cache miss; this is fine. Websocket plumbing for memberships changes can wait.
  • Migration of seed triggers in seed.go. They ship disabled and use author_is_self as a hint to the user. We rewrite the seed to use author_in: [] with a comment explaining the user needs to fill it in. (System triggers are reference examples, not auto-firing.)
  • Reviewer team memberships. review_requested on a team-level reviewer is a separate scenario already handled by tracker enrichment; out of scope here.

§12

Risks

RiskMitigation
Migration silently drops user rules. If a user has a rule with author_is_self: true and their github_username is NULL at migration time, the field is dropped → rule becomes "match any author." WARN-level log per dropped predicate identifies the handler ID. Document in release notes. Add a post-migration toast in the FE on first boot ("N rules were degraded during upgrade — review here.")
Tracker still emits old field name on the wire. Pollers in flight when the binary upgrades could write old-shape JSON. Not actually a risk: events are in-process structs, not wire JSON. The DB never stores events.metadata_json with author_is_self — it stores it with author. The matcher already reads author; the deprecated bool was a convenience field.
Case mismatch surprises users. User types "aidanallchin", GitHub emits "AidanAllchin". Lowercase-at-compare via strings.EqualFold. Both sides handled. FE preserves casing for display round-trip.
Empty allowlist semantics drift. Is "author_in": [] "no filter" or "match nobody"? Pin "no filter." Same semantics as nil-pointer predicates today. Documented in the helper. UI emits undefined rather than [] when the user has typed nothing, but matcher handles both identically.
Stale FE caches predicate UI from before the change. User loads the app on an old SPA bundle, sees the toggle, submits old shape. Embedded SPA — every Go-binary upgrade refreshes the bundle. Acceptable; not handling separate FE/BE versioning here.
Multi-mode rollout coupling. Multi installs need github_username populated by OAuth (SKY-251), which hasn't shipped. This PR ships the column and the local-mode bootstrap. Multi-mode write site is left as a TODO in the auth handler stub (SKY-251 owns it). No production multi installs exist yet, so no degraded UX in practice.
Keychain → DB migration runs before schema column exists. Migration ordering bug could leave a partial state. Strict ordering in main.go: db.Migrate (schema migration applies first) → migrateLocalUserGitHubIdentity → poller start. Migration function is idempotent and a no-op if the column read returns "no such column" (defensive — shouldn't trigger since schema ran first, but the function guards anyway).
Poller starts before keychain → DB migration. Poller would read NULL from DB, short-circuit, then the migration writes the value but the poller doesn't pick it up until restart. Same ordering rule: migration runs before poller start. Also, the existing onGitHubChanged callback already restarts the poller on credential change — even if ordering breaks, a subsequent settings save fixes it without manual restart.
