02How we keep tenants apart

Three layers of kernel.

A delegated agent runs arbitrary shell. That means the isolation needs to be real. Hosted TF puts every agent run inside three concentric containment boundaries. Click a ring to see what each one enforces.

Fly Machine · Firecracker
TF container
gVisor sandbox

Fly Machine layer 1 · microVM

Each TF instance lives in its own Firecracker microVM. The Machine boundary is enforced by a hypervisor, not a namespace — this is the primary tenant-isolation layer.

  • Dedicated kernel per Machine, full caps available
  • Seccomp/AppArmor unconfined by default — we control the profile
  • No noisy neighbours: one Machine cannot see another's memory or disk
  • This is why Fly, not Railway: Railway's PaaS profiles block runsc

TF container layer 2 · the app

Inside the Machine, the Go server runs in a normal container. It holds your sessions, talks to Postgres + GoTrue + Vault, and is the only thing your browser ever speaks to.

  • Web tier has no /data/ volume — it can't even see worktrees
  • All credentials decrypted in-memory, never written
  • Cookie session → opaque sid → encrypted JWT in Postgres
  • The blast radius if this layer is breached is the Machine, not the host

gVisor sandbox layer 3 · per delegation

Every agent run gets its own runsc sandbox. The agent sees an alpine rootfs, its own /work/ worktree, and a network namespace with egress allowed only to Anthropic and the run's GitHub host.

  • Bash inside the agent cannot reach /data/orgs/<other-org>
  • /proc/1/environ shows only the per-run env — no TF_SESSION_KEY
  • Egress allowlist resolved at spawn time, pinned to IPs
  • Single mechanism on both self-host and Fly — identical runtime model
Linux is required · hosted runs on Fly Machines · Mac developers use local mode
03Why local mode survives

Local mode is just N=1.

There aren't two data models. SQLite (local) and Postgres (hosted) hold the same shape — one org, one team, one user, one agent on a local install; many of each on hosted. Every query is the same. Every visibility check is the same. The seam is at the storage driver, not at the schema.

04Identity ladder

From a dev login to enterprise SSO.

Authentication grows with the customer. v1 ships with the path every developer already has — this week. v2 is the rung corporate IT departments need — two weeks behind. v3 is what arrives, eventually, when an org's headcount makes manual user management absurd. Click a rung.

v1
GitHub OAuth
next 3 days
Every developer already has a GitHub account. Sign in with one click. Backend runs the OAuth dance, mints an opaque session cookie, holds the JWT server-side.
  • GoTrue handles signin, TF backend is the auth proxy
  • No @supabase/* in the SPA — backend-only
  • Per-org GitHub App for the bot's own identity
  • Covers self-host + early hosted use
v2
Entra SAML SSO
~2 weeks
Corporate IT requires SSO via their identity provider. Entra signs the user in; TF picks up the claims and maps to an org membership. Same session model behind the cookie.
  • Drives the corp customer's broader rollout
  • SAML provider config sits on the org admin surface
  • JIT user provisioning on first signin
  • Token lifecycle still managed by GoTrue
v3
SCIM provisioning
when scale demands it
Auto-provision users from Entra groups at scale. When an org reaches the headcount where manual admin gets absurd, IT pushes users + memberships into TF automatically.
  • SCIM 2.0 provisioning endpoints
  • Group → team mapping at the org level
  • Deprovisioning closes sessions in real time
  • Lands only after v2 auth surface stabilises

The bot has its own identity — a workload account, not a flavour of human.
